# Elfenstein — Design Document (living)

Last updated: 2026-04-09 (**ADR-0252**–**0258**: modal hit alignment, **`aria-disabled`**, **`GamePopup`** header/footer hit pass-through, modal chrome **`pointerup`** + deduped **`click`**, **`cancelDrag`** + capture retarget (only after a drag **`pointerdown`** session), **`stopPropagation`** after **`endPointerUp`** on nested **trade** panel / **paperdoll** equipment so parent surfaces do not run **`endPointerUp`** twice; interactive **trade** mounts in **`HudLayout`** **`.panel.game`**)

## 1) High concept
**Elfenstein** is a web-based, grid-based dungeon crawler with a first-person **2.5D** view. Visuals combine **3D dungeon geometry** with **2D sprites** for characters/items, prioritizing a crunchy dithered/print look and a highly contextual, low-clutter UX.

## 2) Product goals
- **Modular + extensible**: content and stats are compartmentalized and easy to expand (items, NPCs, species, classes, floor types, etc.).
- **Multiplayer-ready later**: generation and state should be compatible with a future model where the **host player** runs the authoritative database and **1–3 clients** connect.
- **Contextual UX over menus**: few actions are available at any moment; actions branch into contextual interactions/modals.

## 3) Target platform & constraints
- **Platform**: Web (desktop first).
- **Layout**: HUD and compositor are authored at **1920×1080** CSS px inside **`FixedStageViewport`** (`stageDesign.ts`). The **`.clip`** is exactly the scaled **1920×1080** content box (**`STAGE_CSS_*`** in **`stageDesign.ts`**). A **3px** white frame uses **`outline`** on **`.clip`** with negative **`outline-offset`** so it sits just inside that rectangle without changing layout size. The stage **uniformly scales** by **`s = min(1, viewportW/1920, viewportH/1080)`** (using **`visualViewport`** when present): on viewports **larger than 1080p** the box stays **1920×1080 CSS px** centered with black margins; smaller viewports **shrink** so everything remains visible. **All stage-internal layers (presenter canvas, interactive HUD hit layer, capture HUD) are stage-relative (positioned against the 1920×1080 stage), not browser-viewport fixed**, so letterboxing/margins can’t offset the 3D viewport vs HUD. **`FramePresenter`** / WebGL canvas and **HUD `html2canvas` capture use the same 1920×1080 CSS size**—not the browser viewport. **`getBoundingClientRect()` includes that outer scale**, so the compositor’s **`gameRectPx`** and the **3D render target** use **layout** sizes (divide by **`s`** where needed, and **`clientWidth` / `clientHeight`** for the game viewport) so the scene + UI map correctly inside the stage. In-game CSS uses **`--stage-w` / `--stage-h`** instead of **`100vw` / `100vh`** for max widths/heights where needed.
- **Camera**: first-person, grid movement; no looking up/down; pits can exist but player can’t fall in. **Game yaw** (from `playerDir` / `view.camYaw`) uses forward \((\sin y,\,-\cos y)\) on the XZ plane; **Three.js** `camera.rotation.y` is set to **`-y`** so the rendered view matches that forward (Three’s Y rotation would otherwise flip the X component of view direction).
- **Renderer**: Three.js/WebGL for dungeon geometry.
- **Debug (F2)**: sliders for render/audio tuning, including **camera** (eye height, field of view, optional pitch for development) and **lighting** (lantern/beam/torch intensity + distance, base emissive lift, **shadow** toggles for point/beam, shadow map size and filter, max **nearest** POI torch lights). Includes quick procgen tools: **Regen**, **Descend (debug)**, **set `floorIndex`** (apply and apply+regen), and **floor property toggles** (`Infested` / `Cursed` / `Destroyed` / `Overgrown`) which take effect on Regen/Descend (**not** written to `debug-settings.json`). Includes **Spawn (1 cell ahead)** for **NPC**, **POI**, and **item** (pick any seeded `ItemDefId`; item requires the cell ahead to be **floor**). **Hub hotspots**: **x / y / w / h** sliders (0–1, normalized to the **game viewport** cell) for each click region on the **village** and **tavern** 2D hub scenes; values persist in **`hubHotspots`** inside `debug-settings.json` and local storage like render/audio. **Hub innkeeper sprite scale** (F2, NPC/POI render sliders): scales only the **tavern bartender** 2D sprite visually (**CSS** `transform` from bottom center); the **innkeeper hotspot** rect is unchanged unless hotspot sliders are edited—persists with other **render** keys in `debug-settings.json` / local storage. **`debugUi`** in the same JSON / local snapshot persists **Proc overlay** mode, **debug BGM track** override, **room telegraph** (mode + strength for the compositor tint), and the **NPC dialog** / **death** modal **preview** checkboxes. **UI**: those checkboxes show the **NPC dialog** modal (uses the **first NPC** on the current floor; no encounter state) and the **death** modal (current run stats; does **not** set `ui.death` or block gameplay). Includes a small **Pose** readout (playerDir and camera yaw values) for diagnosing orientation issues and a **Cell** readout showing the **player cell** with tile + procgen room tags (district/roomFunction/etc). **Portrait**: portrait shake envelope (**hold/decay**) + amplitude, mouth flicker (**Hz** + **amount**), and min/max gap (ms) between Igor **idle** flashes and min/max **flash** duration (ms). **Audio** includes **master music** volume, spatial emitter mix, and **munch** SFX (volume, noise **LP sweep** endpoints, **HP** corner and **HP/LP Q**, duration, thump Hz, tremolo). Values load from `web/public/debug-settings.json` first, then **localStorage** overrides on the same keys. Slider edits auto-save **locally**; persisting into the repo requires an explicit **Save to project** action (**Vite dev** only; the toast reports failure if the write does not succeed). **Clear local overrides** (dev) removes the local snapshot and reloads so the committed JSON applies without stale storage. There are **no always-on on-screen debug overlays** during play; debug UI is accessed via **F2** (and renderer-only debugging uses explicit query params when needed). Pitch is a debug aid only; core UX remains yaw-on-grid.

## 4) Core player experience (pillars)
- **Touch what you see**: the hand cursor is the primary verb (click, drag, drop).
- **Party as a single organism**: shared inventory with a **fixed** slot grid (see §7.2).
- **Discovery through interaction**: crafting by experimentation; NPC language as a memorization puzzle.
- **Atmosphere via light + sound**: darkness is default; proximity audio supports exploration.

## 5) Core loop
Explore → find POIs/NPCs/items → manage inventory/craft → resolve encounters → recover at POIs → push deeper (keys/doors) → repeat.

### 5.1 Title, hub, and entering the dungeon
- **`ui.screen`** is **`title`** (menu only), **`hub`** (2D bespoke scenes in the **game** HUD cell), or **`game`** (first-person dungeon).
- **New run** from the title screen starts a fresh run in **`hub`** with **`hubScene: village`**: the center cell shows a **2D image** (`Content/village.png` → `/content/village.png`) and **click regions** (currently drawn as **empty rectangles with a red outline**) for **Tavern** (switch to tavern hub) and **Cave** (set **`screen: 'game'`** and play the 3D dungeon). Dungeon floor state is still generated at new-run time; the player only **sees** the 3D view after **Cave**.
- **Tavern** hub uses **`Content/tavern_background.png`** with **`Content/tavern_foreground.png`** over the scene (non-interactive) and **`bartender_*.png`** (`base` / `idle` / `blink`) drawn inside the **innkeeper** hotspot rect (same normalized rect as the click target) for a simple timed swap + blink; optional F2 **hub innkeeper sprite scale** enlarges or shrinks the sprite **without** resizing that click rect. **Innkeeper** opens **`TradeModal`**: a **`ui.tradeSession`** with **`kind: 'hub_innkeeper'`** drives **their stock** (from content defaults; remaining quantities persist for the run in **`run.hubInnkeeperTradeStock`**), **wants** (which **`ItemDefId`s** they buy), **your offer** slot, **you request** slot, **Trade** (1 unit for 1 unit), and **Close**; **their stock** (and the **offer** / **request** slots) reuse the same **CSS grid + square slot + emoji `font` + corner qty** pattern as the HUD **`InventoryPanel`** (**same column count** as **`party.inventory.cols`** for the merchant grid). **Interactive hits** mount **inside** **`HudLayout`**’s **`.panel.game`** (same **`interactiveHud`** lineage as **`InventoryPanel`**): **`TradeModal`** renders **`tradeGameOverlay`** (**`position: absolute; inset: 0`**, **`z-index`** above hub/3D content) and a **`tradePanelInGameCell`** **`popup.panel`** centered in **stage** space—no **`document.body`** portal. **Dismiss**: click on the **overlay** background (**`target === currentTarget`**) closes the session (**staged offer** restows). **Capture** trade still rasterizes from **`captureFullHudOverlay`** (**`fullHudCaptureLayer`**) for dither parity (**ADR-0254**). Inventory slots whose items match **wants** are **highlighted** (strong border + glow) while the session is open; **clicking** such a slot (without starting a drag) stages that item into **your offer**, same as dragging it onto the offer slot. **Their stock** rows can be **clicked** to set **you request**, same as dragging onto the ask slot. **Exit** returns to the village hub.
- **Reload checkpoint** restores dungeon play and does **not** return through the hub.
- While **`screen === 'hub'`**, grid movement and other dungeon actions are **disabled**; the on-screen **navigation pad** is **inactive**.

**On party wipe** (all party HP ≤ 0): gameplay input is blocked and a **death screen** is shown with a short **run summary** and actions:
- **Reload checkpoint** (if a checkpoint exists)
- **New run**

## 6) Controls & interaction model
### 6.1 Cursor (hand) states
- **OS cursor**: hidden globally (`cursor: none` on `document.body`); interactive HUD widgets also set **`cursor: none`** so controls (e.g. navigation buttons) never revert to the system pointer—only the **`CursorLayer`** hand sprite is visible. **`CursorLayer`** is a **sibling** of **`FixedStageViewport`** (not inside the scaled **`.stage`**): **`position: fixed`** with **`clientX`/`clientY`** must use the **viewport** containing block; a transformed ancestor would offset the hand from the real pointer. **`CursorLayer`** uses **`z-index: 9999`** above most UI; **all descendants use `pointer-events: none`** so the hand/ghost/affordance never **capture** clicks meant for DOM below. The **NPC dialog** portaled hit layer uses **`z-index: 10100`** (**above** the cursor layer) so **Close** / drag targets stay reliable; it stays **visually empty** (**`opacity: 0`**) so the hand remains visible through it.
- **Default**: `Hand_Point`
- **Holding/dragging**: `Hand_Hold` (held item sprite follows pointer)
- **Hover active**: `Hand_Active` when pointing at an interactable
- **Craft-ready hover**: while dragging an inventory item over another inventory item that can be crafted with it, the hand cursor flickers between `Hand_Hold` and `Hand_Active` and the affordance pill shows **⚗ Craft**; an icon badge shows `?` for unknown recipes or the result icon for discovered recipes.
- **Click micro-feedback**: on **pointer down**, the hand cursor can perform a **tiny shake** (purely cosmetic; debug-tunable in F2 under **Cursor**). This is a *micro* cue meant to add tactility without implying a successful action (success/failure feedback still comes from SFX/log/shake events).
- **Deadzone behavior**: hovering non-interactive UI areas should clear any previous “active” hover so the cursor returns to `Hand_Point`; pointer move/up are tracked globally across the HUD so cursor state can’t get stuck when moving over panels.
- **3D viewport hover override**: the WebGL viewport may inject a **virtual hover target** (e.g. `floorDrop`) on pointer-move to compensate for `elementFromPoint` seeing the compositor canvas rather than a DOM hit target. This override is **ephemeral (one move event)** so normal DOM-derived hover targets (inventory slots, portraits, etc.) take over immediately when the pointer leaves the viewport.

### 6.2 Interaction rules (mouse-first)
- **Click**: attempt to use/interact with target (object/NPC/POI/UI element).
- **3D viewport ray pick**: when several pickables lie on the same camera ray, **floor items take priority** over POIs (and over NPCs/doors on that ray) so loot is not blocked by large POI billboards (e.g. a chest in front of a dropped item).
- **Drag (items)**: dragging starts when you **press and move** beyond a small threshold (to avoid accidental drags on click). **Press + hold** is also supported as a fallback to begin dragging items from **inventory** and from **world floor items** in the 3D view. **Movement promotion** in **`CursorProvider`** uses the same **synchronous refs** as **`beginPointerDown`** (**`pendingPayload`**, **`dragStartPos`**, **`dragPromotedRef`**) so the first moves after **`pointerdown`** are not ignored while React state is still catching up (otherwise the UI falls back to the **~140ms** hold timer and feels laggy). **`pointerup`** / **drop** resolution uses the same **“drag promoted”** flag so **`pointerdown`’s `preventDefault`** does not strand **tap-to-pick** UX (e.g. trade stock → **you request**, wanted inventory → **offer**) waiting on a stale **`dragging.started`** read. **Trade** / **inventory** item buttons call **`stopPropagation` on `pointermove`** after **`onPointerMove`** so parent grids/panels do not run **`elementsFromPoint` + `setState` multiple times per frame**.
- **Drop**:
  - Onto **inventory**: store item
  - Onto **empty 3D view**: **cursor-aimed drop** (raycast to the floor under the pointer), clamped by a max Manhattan range (tunable). If the aimed cell is invalid/blocked, snap to the nearest valid floor cell within range; if none exist, fall back to dropping near the player
  - Onto **another item**: attempt crafting / recipe discovery (see §7.3)
  - Onto **NPC**: use item on NPC (may be rejected/consumed/apply status/attack)
  - Onto **portrait eye**: inspect interaction
  - Onto **portrait mouth**: feed interaction
  - Onto **character equipment strip** slots (**`head`**, **`handLeft`**, **`handRight`**): equip via **`equipmentSlot`** drops (same rules as §7.4 / paperdoll; **`equipItem`** in the reducer)
  - While **`ui.tradeSession`** is open (**§5.1**): onto **trade offer** slot — stage an inventory item whose **`defId`** is in the merchant’s **`wants`** (item leaves the grid until traded or the modal closes), or **click** a **highlighted** inventory slot to stage it (no drag); onto **trade ask** slot — drag from **their stock** row (or **click** the row) to pick what you request (**1×**); drag the staged offer back onto **inventory** to return it
- **Timing**: drop resolution uses the pointer-up timestamp (same `performance.now()` clock as `time/tick`) so short-lived UI cues (e.g. portrait mouth “chomp” burst) can’t expire immediately due to a stale tick time.
- **Preview affordance**: contextual icon appears near relevant elements indicating what releasing will do.

### 6.3 UX feedback
Interactions should resolve with:
- short relevant animation (e.g. eyes/mouth)
- sound
- **Activity log** (**`ui.activityLog`**, rendered as **`ActivityLog`** in a **bottom-right corner stack** in **`HudLayout`** with **`CombatIndicator`** during encounters—the log sits **above** encounter chrome so the two do not overlap): each line of feedback (POI outcomes, crafting, combat hits, door messages, inventory pickup/full, inspect/feed text, debug **`ui/toast`** dispatches, etc.) is **appended** (newest last); the list is **capped** in memory (**`ACTIVITY_LOG_MAX_ENTRIES`**), each line **expires** after **10 seconds** of game time (**`ACTIVITY_LOG_ENTRY_TTL_MS`**, pruned on **`time/tick`**; pruning is **paused** while the **death** modal is up so the recap stays stable), and the widget **shows at most the four newest** entries. It uses **no** filled panel background (text sits over the scene, using **`--buttonTitle*`** color and shadow for contrast). Lines are **right-aligned** in that corner stack. Typography reuses **`--buttonTitleFontFamily`** / spacing / color / shadow from **`index.css`** at **`font-size: calc(var(--buttonTitleFontSize) - 5px)`**, with **regular (400)** weight (unlike **`--buttonTitleFontWeight`** on modal buttons). There is **no** separate centered transient **`ui.toast`** overlay; the **`ui/toast`** action remains as a convenience and **only** pushes a log line.
- subtle camera shake for key interactions (driven by `ui.shake` and tunable in Debug/F2). **Shake length / hold (ms)** and **shake decay / fade (ms)** control the envelope for the **3D camera**, **HUD overlay shake**, and **portrait frame shake** (shared `shakeEnvelopeFactor` in `web/src/game/shakeEnvelope.ts`). With length 0, decay uses the legacy ramp `min(1, remaining/decay)`; with length above 0, full strength holds for the scaled hold segment, then linearly fades over the scaled decay segment within each event’s `startedAtMs`→`untilMs` window.
- **Portrait frame shake** on portrait interaction resolution (inspect and feed), driven by `ui.portraitShake` on the matching character slot

### 6.4 Grid movement (first-person)
- **Model**: discrete cells; facing is one of four compass directions (**`playerDir`** is canonical 0=N,1=E,2=S,3=W); the 3D camera **yaw** matches facing when idle. **Strafe** moves one cell **left or right relative to facing** without rotating (**A** / **D**). **W** / **S** move **forward** / **backward** along facing.
- **Turning**: left/right turns always animate the **shortest** 90° rotation. Internally, the animated camera yaw may be temporarily **unwrapped** across the \(0 \leftrightarrow 2\pi\) boundary so the interpolation never takes a 270° path; when the turn completes the camera yaw snaps back to the canonical \(dir \cdot \pi/2\) in \([0,2\pi)\).
- **Keyboard** (see `GameApp.tsx`): **W** / **S** (and **↑** / **↓**) forward/back; **A** / **D** strafe left/right; **Q** / **E** turn left/right. **F2** toggles the debug panel. **During an encounter** (§7.7): **R** attempts **flee**; **F** **defends** (same actions as the **CombatIndicator** buttons on the game viewport).
- **HUD**: the bottom grid row (minimap + inventory + navigation) has a shared row height of **285 px** (**−5%** vs the prior **300 px** row, after the earlier **−25%** step from **400 px**); it stays the **third** grid row so widgets remain **bottom-aligned** with **`1fr` / `1fr`** rows above taking the freed space. The **minimap is north-up** and shows a **local “pixel-circle”** around the player: it **only draws** dungeon tiles that satisfy **dx² + dy² ≤ 9** (**radius 3**, Euclidean on the square grid) **and** lie **inside the current floor rectangle**—**no** placeholder squares for out-of-disk or **out-of-bounds** cells (those spots are simply empty; **HUD/game chrome** shows through). Tile positions use a **7×7** logical grid with **absolute** placement so on-map cells stay **north-up aligned**. The widget is **clipped to a circle** with **no** filled circular backdrop. Each cell is drawn at **~22.5×22.5 CSS px** (**+25%** vs the prior **18 px** cell) with a **~2.5 px** gap (**`MINIMAP_CELL_PX`** / **`MINIMAP_GAP_PX`** in **`MinimapPanel.tsx`**, **`--minimap-cell`** on the map); cells are **square** (**`border-radius: 0`**) for a chunky look. The player tile includes a **facing arrow** (4-way) so orientation is readable at a glance; tile fills use **opaque** colours with **moderate** wall/floor separation and softened feature accents so the map stays legible through HUD capture and dither without looking harsh. **Left and right** portrait **rails** (rows **1–2**) use **`minmax(0, 1fr) 120px minmax(0, 1.12fr) 120px minmax(0, 1fr)`** so outer **map**/**navigation** alignment and **1fr** portrait column widths match the layout before empty statue strips were removed. The **game** `<section>` **`grid-area`** spans the **three** center tracks (**120px + 1.12fr + 120px**), so the **3D viewport** fills that band (**no** separate statue panels; **`StatuePanel`** removed). **CHAR1/CHAR2** (left) and **CHAR3/CHAR4** (right) `<section>`s use **75%** rail width with **`justify-self: start`** / **`end`** so they line up with the **map** / **navigation** blocks below. Each rail uses **`charRail`** (**`HudLayout.module.css`**): **`flex`** row, **`gap: 0`**. **Left** columns: **`[Portrait][Strip]`**; **right**: **`[Strip][Portrait]`**. Portrait lives in **`charRailPortraitGrow`** (**`flex: 1 1 0`**, **`min-width: 0`**). The **second** flex child always gets **`charRailPushEnd`** (**`margin-left: auto`**), so **all** spare width is **between** portrait and strip—**mirrored** layout, **identical** spacing math on both sides (no per-side **`translate`** or spacer divs). Row **3** is a **`bottomRow`** wrapper: nested **`grid-template-columns: 0.75fr 120px 1.62fr 120px 0.75fr`** (same **3.12fr** sum as **`1 + 1.12 + 1`**) with **inventory** spanning the **120 + 1.62fr + 120** band. **Map** and **navigation** `<section>`s use **100%** of their **0.75fr** tracks (**0.75fr** total), matching portrait strips’ **75%** of the **1fr** rails above (**0.75fr**); they have **no** MAP/NAVIGATION title headers; the **inventory** panel likewise has **no** section title, and **inventory** grid item icons (emoji) render at **~44 CSS px**. **`HudLayout`** **`display: flex`** + **`align-items`/`justify-content: center`** centers each widget in its cell. At a reference desktop width this yields roughly the former **~518 px** outer / **~580 px** center-band proportion for rails vs merged game span. The **navigation** panel is a **3×2** image pad (sources under `Content/ui/navigation/`, mirrored to `web/public/content/ui/navigation/` for the dev server): each **cell is drawn at 50% of the bezel PNG native size** (**88.5×88.5 px** from 177×177 sources), **square** corners (no radius on the cells); on press it shows **ui_navigationbutton_pushed** for **0.5 seconds**, then returns to default. Direction glyphs are separate overlays (**arrow up/down/left/right**, **turn left/right** icons). Layout row 1: turn left, forward, turn right; row 2: strafe left, back, strafe right. The pad’s intrinsic width is **~271.5 px** (three cells + gaps); it stays **centered** inside the **navigation** panel. Tooltips still echo keyboard shortcuts (§6.4). **Note**: the on-screen HUD is composited from an offscreen **capture** DOM (`HudLayout` with `captureForPostprocess`); the **interactive** HUD is pointer-transparent (`opacity: 0`). Navigation “pushed” bezel state therefore lives in `DitheredFrameRoot` and is passed into **both** trees so `html2canvas` sees **`ui_navigationbutton_pushed.png`** on the cell you pressed.
- **Viewport**: clicking a **door** attempts a **forward** step into that cell (open normal door or try key on locked door).
- **While a move/turn animation is playing**, new step, strafe, and turn input is ignored so the player resolves one grid action at a time.
- **Audio**: a successful step or strafe plays a short **step** SFX; walking into a solid tile uses a distinct **bump** SFX (with **activity log** line + shake).
- **POI cells**: the player **cannot enter** a floor cell that holds a **POI** (props occupy their tile). Attempting to step or strafe onto that cell uses the same **bump** SFX/shake as a wall, with a distinct **activity-log** line. **Exit**, **Well**, and other POIs remain usable via **viewport click** or **dragging items onto them** from **adjacent** cells (or wherever ray pick already allows). **Regen**, **descend**, and **initial floor** place **`playerPos`** on **`gen.entrance`** when that cell has no POI; if the **Well** (or any POI) sits on the entrance, spawn is the first orthogonal **floor** neighbor without a POI (**N→E→S→W** tie-break). **Checkpoint reload** nudges **`playerPos`** the same way when an older snapshot still has the party on a POI tile.

- **Portrait press (frame)**: a **primary-button press** (`pointerdown`) on the **portrait frame** triggers a short **`ui.portraitIdlePulse`** window so the **idle sprite** shows for one burst (same **min/max ms** tuning as ambient idle flashes: `portraitIdleFlashMinMs` / `portraitIdleFlashMaxMs` in F2). **`HudLayout`** handles this at the HUD root capture on `[data-portrait-character-id]` so it remains responsive even when a release would be delayed/lost; if the press becomes a drag or exceeds movement slop, the pulse is cancelled (`ui/portraitIdleCancel`).
- **Portrait scaling**: the portrait frame scales up to fill as much of its HUD slot as possible while **preserving the portrait asset aspect ratio**; portrait art is rendered using **no-crop fit** (scaled as large as possible while fully visible within the frame). Layered portrait sprites are offset **30px upward** from the geometric center of the frame for composition; mouth “chomp” animations preserve that offset.
- **Portrait equipment mirror (read-only)**: **Head** and **hand** equipped icons from **`ContentDB`** are drawn on the portrait in the **same** top / bottom bands as the legacy on-portrait equip layout (**`PortraitPanel.module.css`**, **`z-index`** below eyes/mouth), with **`pointer-events: none`** so **HUD** equip drag/drop stays on **`CharacterEquipStrip`** only. These layers render in **both** the offscreen **HUD capture** tree and the invisible interactive HUD so equipped icons appear in the **dithered presenter** bitmap (unlike mouth flicker / idle overlays, which are compositor-time-only and are suppressed in capture). **`getCharacterEquipmentHudModel`** in **`equipment.ts`** is shared with the strip so visibility rules (**two-hand** → single bottom-right icon, etc.) stay aligned.
- **Portrait stats presentation**: character vitals + status use a **compact bottom overlay inside the portrait frame** (gradient dark at the bottom, fading upward). The overlay is painted **in front of** portrait sprites (**higher `z-index`**) so bars stay readable; it uses **`pointer-events: none`** so drag/hit targets (**eyes**, **mouth**) still work underneath. **Equipped items**: interactive **icons** on the adjacent **`CharacterEquipStrip`**; the portrait also shows **matching non-interactive** icons in the same bands (see **Portrait equipment mirror** below). Vitals sit in a **2×2 grid**: **row 1** = **Health** | **Stamina**, **row 2** = **Hunger** | **Thirst** (left-to-right). Each cell is a small panel (**black** **`#000`**, **`2px solid`** border using **`var(--hud-inventory-slot-border)`** (same token as inventory slots), **`border-radius: 0`**, inner **padding**) containing **only** a **horizontal bar** (**no** emoji): bar **track** **black**, **~9px** tall, **square** corners (**`border-radius: 0`** on track and fill), **fill** colors: **HP** **`#ff2400`**, **STA** **`var(--hud-stamina-vital-fill)`**, **HUN** **`#547d39`**, **THR** **`#3d75dd`** (see **`VITAL_BAR_FILL`** in **`PortraitPanel.tsx`**). Grid and overlay use tight **`3px`** gaps; status line **`~3px`** below the grid. Overlay padding **`6px clamp(10px, 8%, 70px) 20px`** (keeps **~70px** side inset on wide portraits; shrinks on narrow rails next to **`CharacterEquipStrip`** so bar cells stay usable). The overlay is **not** shifted in **X** relative to the portrait; it uses **`translateY(30px)`** so the **2×2 attributes grid** and **status** line sit slightly below the portrait bottom (**`PortraitPanel.module.css`** **`.statsOverlay`**). The **whole** **`PortraitPanel`** column (portrait + overlay + read-only equip mirror) uses **`translateY(-20px)`** on **`.root`**, plus **`translateX`** from **`HudLayout`**: **`PORTRAIT_AND_VITALS_NUDGE_TOWARD_GAME_PX`** (**35**) → **+35** on **CHAR1/CHAR2**, **−35** on **CHAR3/CHAR4** (mirror toward the game column). **CHAR1–CHAR4** HUD **`section`**s use **`overflow: visible`** (overriding default **`panel`** clipping) so nudged stats and equip strips are not cut off; portrait **sprites** stay clipped to the rounded frame via an inner **`.portraitArtClip`** wrapper (**`overflow: hidden`**, **`border-radius: inherit`**). **Bar fill** is **proportional** to current vitals: **HP** and **Stamina** use the same run-level caps as healing and feed (**`hpMax` / `staminaMax`** in **`runProgression.ts`**); **Hunger** and **Thirst** use a cap of **100** (same as feed clamp). Current values are clamped to their caps for display. No numeric readout. A **status** line below (**`Status: …`**) stays **single-line truncated** when long.
- **Portrait blinking**: the eyes layer is **occasionally hidden briefly** to simulate blinking.
- **Portrait idle overlay**: while the **idle overlay** sprite is visible (ambient flash or `ui.portraitIdlePulse`), the base **eyes sprites are hidden** so they can’t show through under the overlay (inspect hover still overrides).
- **Portrait inspect hover**: while **dragging** an item and hovering the **eyes** target area, the portrait swaps to an **inspect eyes** sprite (if available). Inspect hover **overrides blink hiding**.
- **Portrait mouth visibility**: mouth layer is **hidden by default**; it becomes visible during **feeding interactions** (dragging over mouth target and briefly after a feed attempt).
- **Portrait mouth feedback (feed)**: after releasing an item on the mouth target, successful feeding plays a short “chomp” (mouth flicker + tiny wiggle), a short **portrait frame shake**, and a brief **munch** SFX. If the item is **refused**, do **not** play the mouth cue; instead show a brief **idle overlay pulse** and play a **reject** SFX (still using `ui.portraitShake` for the slot). **No** `ui.shake` (3D view or empty HUD overlay) for portrait inspect/feed—only `ui.portraitShake` on the relevant slot.
- **Portrait inspect**: resolving an eye drop plays a short **portrait frame shake** (gentler than feed), likewise without 3D shake. The **inspected character** (whose eyes you dropped onto) makes a **deterministic Perception check**: **d20 + Perception** vs a **DC** derived from the item’s tags (food/material **10**, container **11**, weapon/tool/hat **12**, quest **14**, default **12**; when multiple apply, the **highest** DC applies). Results are logged to **`ui.activityLog`**: always **who** inspects **which item name** and the **numeric check** (`d20 + Perception = total vs DC`). **Fail** (`total < DC`): a short “nothing more” line. **Success** (`total ≥ DC`): one line with a **high-level read** of the item plus **character-relevant** hints (e.g. remedy fit if they have a matching ailment, or species-tied feed effects when applicable). **Great success** (`total ≥ DC + 5`): **one or two** extra lines with **full mechanical detail** (feed deltas, possible status chances, weapon stats, equip slots, POI transforms) when the item defines them.
- **Portrait reaction rendering**: the HUD base is captured to a texture for postprocess, but **mouth flicker** and **idle pulses** are rendered as **compositor-time overlays** in the WebGL presenter (using the live HUD layout rects). This keeps reactions **snappy** and **full-FPS** without waiting for async HUD recapture. During a portrait press / `portraitIdlePulse`, HUD capture temporarily runs in a **burst** cadence and includes the **pressed-portrait key** so the eyes-hidden layer is captured immediately (avoids a one-capture lag where eyes remain visible under the idle overlay).
- **Portrait shake tuning**: portrait frame shake has its own envelope and amplitude tuning via `RenderTuning`:
  - `portraitShakeLengthMs` / `portraitShakeDecayMs` (envelope)
  - `portraitShakeMagnitudeScale` (amplitude multiplier applied to `ui.portraitShake.magnitude`)
  - `portraitShakeHz` (oscillation speed for the portrait shake transform)
  - Portrait shake duration is at least the authored base duration, but increases to match the tuned envelope window so “hold/decay” can produce multi-oscillation shakes.
- **Character equipment strip** (`CharacterEquipStrip` beside each portrait): vertical chrome PNG **`ui_hud_charequip_background.png`** (**198×589** px, `Content/ui/hud/`) with three hit regions (**head** → **handLeft** → **handRight**, top to bottom) aligned to the art frames. Each region shows a **2px** frame via a **`::after`** border: default **`var(--hud-inventory-slot-border)`** (inventory slot color). While the cursor has an **active drag** (`CursorProvider` **`dragging.started`**), any slot where the dragged item **fits** (**`itemFitsCharacterEquipmentSlot`** in **`equipment.ts`**: **`hat`** / **`oneHand`** / **`twoHand`** plus optional **`equipSlots`**) uses **`var(--hud-stamina-vital-fill)`** (stamina bar color); on release or cancel, borders return to the default token. Strip **width** and **equipped icon** sizing use **`CharacterEquipStrip.module.css`** **`--equip-strip-scale`** (**1.15** vs a **78px**-wide baseline) so the whole widget scales together. Each region is an **`equipmentSlot`** drop target; equipped items render as draggable icons (void-drop **stow** uses the same **`fromPortrait`** equipment-drag path as before). **Two-handed** items show in the **bottom (right-hand) frame** only; drag/unequip still uses the **`handLeft`** slot id when both hands reference the same item. **Placement** follows **`charRail`** + **`charRailPushEnd`** (see **HUD** bullet above) so **left** and **right** rails are **symmetric**. Additional nudges (all **`HudLayout`** constants, single place to tune): **`EQUIP_STRIP_NUDGE_TOWARD_GAME_PX`** (**55**) → **`equipTranslateXPx`**: **+55** on **CHAR1/CHAR2** (left equipment toward the **right** / game), **−55** on **CHAR3/CHAR4** (**mirrored** toward the game). **`EQUIP_STRIP_NUDGE_UP_PX`** (**20**) → **`equipNudgeUpPx`** on **all** four strips (**`translateY(-n)`**, same value everywhere).
- Portrait **drop targets** (inside the portrait frame only):
  - **Eyes area** → inspect drop target
  - **Mouth area** → feed drop target
- **Handedness (content tags on `ItemDef`)**: **`oneHand`** items go to one hand—**left first**, then **right**; if both are occupied, the new item **replaces the left hand** (previous item is stowed to the first free inventory slot when possible). **`twoHand`** items occupy **both** hands at once; the same `itemId` is stored in **`handLeft`** and **`handRight`** so unequip/replace clears both without duplicating the item in inventory.

### 7.7 Combat (encounter mode)
- **Trigger**: attempting to step into a **hostile** NPC starts an **encounter** (the player does not move into the NPC cell). **Non-hostile** NPCs do not block movement or start encounters. **Roster**: the encounter includes that NPC plus every other **hostile** NPC in the **same procgen room** (first `floor.gen.rooms` rect containing the trigger cell). If no room contains that cell (corridor / missing gen), hostiles within **Chebyshev distance ≤ 1** of the **player** are included instead. **Swarms** are excluded from the roster while the party holds a **Swarm Queen** (same neutrality rule as outside combat).
- **Mode**: while an encounter is active, **movement is blocked**; the encounter proceeds in **turn order**.
- **NPC combat data**: per-**`NpcKind`** **Speed**, **damage**, **armor**, **resists**, **on-hit statuses**, and **`hpMax`** live in **`web/src/game/content/npcCombat.ts`** (`NPC_COMBAT_BY_KIND`). Runtime NPCs carry **`hp`** and **`hpMax`** (spawn and **CombatIndicator** use the same caps; **`hydrateFloorNpcs`** back-fills **`hpMax`** for older saves).
- **Turn order (initiative)**: each living **party member** and each encounter **NPC** gets a turn; initiative is driven primarily by **Speed** with a small **deterministic tie-break** so ordering is stable per encounter seed.
- **Player actions**: the party acts through the existing **cursor-first drag/drop** model:
  - dragging a **weapon** onto the target NPC performs an **attack** (consumes that character’s turn when the dragged weapon originates from that character’s equipment; inventory-origin weapons act as the current turn’s character).
  - **clicking** an **encounter enemy** in the 3D view on a **PC’s turn** performs an **attack** using the same weapon resolution as a convenience default: **equipped** hands (**left**, then **right**; two-hand held once), else the **first weapon** found in the **inventory grid** (if none, feedback only; drag a weapon explicitly if needed).
  - dragging a non-weapon item onto an NPC performs a **give/use** attempt and also consumes the turn.
  - **Fireshield** (see §7.3): on a **PC’s turn** in an encounter, drag **Fireshield** onto that PC’s portrait **hands** (not onto an NPC). Spends **stamina**, **consumes** the item, applies **Fire resist** vs **NPC hits** for **`shieldTurns`** of that PC’s **turns** (countdown when their initiative recurs). **Does not** equip through this path during combat.
- **Defend**: on a PC’s turn, **Defend** (HUD button or **F**) spends **stamina** and grants **temporary armor and resist** against **NPC hits** until that PC’s **next** turn begins (the buff clears when their initiative slot comes around again).
- **Flee**: **R** or the HUD button; stamina cost and a **Speed**-influenced check; failure provokes a **free enemy hit**, then **initiative advances** (the attempt consumed the turn). **Too exhausted to flee** rejects without advancing.
- **Combat HUD**: **CombatIndicator** on the game viewport lists **roster enemies** with a **compact HP bar** each (**current / hpMax**), plus current turn, a short **“Next:”** line for the following queue entry (when any), and **Flee** / **Defend** (mirrors **R** / **F**). No separate encounter modal.
- **NPC actions**: on NPC turns, the NPC automatically attacks a party member using **soft targeting**: prefer **lower current HP**, **deprioritize PCs who are Defending** (virtual HP padding in the pick), with a tiny **deterministic jitter** so ties stay stable per encounter seed.
- **NPC hit vs party**: **d20 + NPC Speed** vs **10 + target PC Speed** decides hit/miss; **crit** on natural 20. Damage uses that NPC kind’s **base damage** and **damage type**, then **PC armor** (Blunt/Pierce/Cut only), **resistances**, **Defend** bonuses, and **Fireshield** extra **Fire** resist only (`npcTakeTurn`).
- **PC damage resolution**: in **encounter mode**, a **d20 + Perception + Agility** check vs **10 + NPC Speed** decides hit/miss; **crit** on natural 20. **Weapon stamina**: if the weapon defines **`staminaCost`**, the cost is paid on **every swing** in an encounter, **hit or miss**. **Weapon damage** starts from **`ItemDef.weapon.baseDamage`**, optional **`damageStat`** (**Strength** or **Agility**) adds **`floor(stat × 0.25)`** before run **damage %** bonus, then **NPC kind** applies **armor** (Blunt/Pierce/Cut only) and **per-type resist** (capped like PC resists). **Open-world** weapon strikes (not in an encounter) skip the to-hit roll but still apply stat bump and mitigation.
- **Activity log (combat swings):** **Encounter** PC attacks and **NPC** attacks each append **one line** that spells out the **to-hit** math in the same style as inspect checks: PC lines use **`d20+Per+Agi … = total vs 10+Spd …`**, then **miss** or **hit** (note **nat 20** when the d20 is 20) and damage; NPC lines use **`d20+Spd … vs 10+Spd …`**. Open-world weapon hits keep the shorter outcome line (no encounter roll).
- **Quest line in combat:** Hostile NPCs with **`quest.wants`** sometimes echo their quest ask in the activity log **on their turn**, **before** the swing line. Chance is **20%** per NPC turn, **deterministic** from **`floor.seed`**, **`combat.encounterId`**, NPC id, and **`combat.turnIndex`** (same keyed hash style as other combat rolls). The text is **English** **`{name}: "…bring me {item name}."`**, matching the dialog modal’s substrate before **`toGibberish`** (implemented via **`npcQuestEnglishLine`** in **`web/src/game/npc/npcQuestSpeech.ts`**).
- **Weapon procs**: `weapon.statusOnHit` can apply **timed debuffs** to **NPCs** on a successful hit (deterministic % roll).
- **World & crafting during combat**: **POI** use (viewport click or dragging an item onto a POI), **floor item pickup** (pickup action or drag onto a floor item), **starting** an inventory recipe craft, and **finishing** an in-flight craft (timer resolution) are **blocked** until the encounter ends (“Not while in combat.”). **Equip**, **unequip**, non-recipe inventory rearrangement, and **portrait** inspect / feed / most equip **stay available** during combat and do **not** consume initiative—**except** **Fireshield** dropped on the acting PC’s **hands**, which **consumes** the turn like an attack. **Portrait feed** (food, remedies, and similar **feed**-path items) during an encounter is **deliberate party triage**: it does **not** cost the acting PC’s turn, so players can stabilize vitals or statuses under pressure; balance leans on **item scarcity**, **stamina** costs on attacks, and **enemy pressure** rather than blocking feed mid-fight.
- **Item actions during combat**: **Hive** floor drops, **Swarm Basket** capture, **Captured Swarm** release, and **Swarm Queen** “calm swarm” updates are **blocked** during an encounter (“Not while in combat.”); use normal **weapon** / **give** / **flee** / **defend** flows instead.
- **End conditions**: encounter ends when all encounter NPCs are dead (**victory**, grants **XP** scaled by **number of enemies that were in the encounter roster**) or when all party HP ≤ 0 (**party wipe**, death screen).

### 7.8 Status effects
- Status effects apply to PCs and NPCs; positive/negative/neutral.
- Some decay over time; some persist until cured by an item. **`time/tick`** prunes expired entries for **both** party characters and **floor NPCs**.
- Baseline content set includes **Poisoned, Blessed, Sick**; additional ids (e.g. **Bleeding**) appear from items and combat procs.

## 8) Dungeon & world generation
### 8.1 Taxonomy (tags drive content)
- **Floor types**: Ruins, Dungeon, Cave
- **Floor properties**: Infested, Cursed, Destroyed, Overgrown, None (not shown)
- **Room function**: Passage, Habitat, Workshop, Communal, Storage
- **Room properties**: Burning, Flooded, Infected (cause resistance check; fail can cause damage)
- **Room status**: Overgrown, Destroyed, Collapsed
- **Doors/keys**: passages can be blocked by door types requiring appropriate keys; procgen records **`keyDefId`** per locked door in **`floor.gen.doors`** (gameplay opens a door only if the party holds a matching item, e.g. **IronKey** / **BrassKey**). In the 3D view, doors are rendered as **sprite billboards** (`door_closed.png` on the tile). When a door is opened, the tile becomes `floor` immediately and the renderer may show a **brief open flash** (`door_open.png`) as a short-lived FX cue.
- **Difficulty** (`0` = easy, `1` = normal default, `2` = hard): **`GameState.floor.difficulty`** and **`FloorGenInput.difficulty`**. Tunes how many **validation rerolls** run (**8** / **6** / **5**), minimum entrance→exit shortest-path length before placing locks (**10** / **6** / **5** for any lock; two-lock branch only when path length ≥ **14** at normal, ≥ **11** at hard, and **never** at easy), and scales the **`layoutScore`** lock+loop bonus when choosing among valid rerolls (easier prefers layouts with less lock+loop weight; harder the opposite). Echoed as **`floor.gen.meta.difficulty`**.

### 8.2 Starting algorithm (prototype baseline)
**Floor type** (`Dungeon` | `Cave` | `Ruins`) selects a **geometry realizer**, then shared post-passes and population:

1. **Layout (per `floorType`)**
   - **Dungeon**: BSP partition (min leaf ~6×6; depth ~6), room carve (~45–70% of leaf), sibling **L-corridor** stitch, then a guarded **door-frame** pass that introduces a small number of 1-tile **throats** in straight corridors so rooms read as connected by doorways (locks prefer these frames).
   - **Cave**: seeded worm carve + occasional widen, then a single **GenRoom** from the floor bounding box of carved cells (fallback box if empty).
   - **Ruins**: ~5×5 macro-cell stamps with random doorways between cells; stamped chambers are deterministically **clustered into macro rooms** (bounded count) for tagging/districts/population.
2. Connectivity repair (deterministic), then CA **carve-only** smoothing (one pass).
3. Exit selection (exit = **farthest** reachable floor cell from the entrance by BFS).
4. **Derived connector rooms**: a small number of **junction/connector** rooms are derived from corridor junction clusters (bounded) and treated as **Passage** anchors for tags/spawn bias.
5. **Districts**: seeded Voronoi on room centers → **`district`** tags (`NorthWing`, `SouthWing`, …) for spawn bias.
6. **Room tagging**: quota-aware function assignment (e.g. try to keep at least one **Storage** among tiny rooms) plus rolls from **`floorProperties`** (Infested/Cursed/Destroyed/Overgrown). Derived connector rooms keep their fixed `Passage` function.
6. **Tag constraints** (post-quota): **Storage** prefers a **dead-end** room center (swap with a tiny **Passage** room when needed); on **Cursed** floors, **Flooded** expands to **edge-adjacent** rooms with a fixed probability so hazards form a small cluster.
7. Population pass:
   - POIs (**Well** / **Bed** / **Chest** / **Barrel** / **Crate** / optional **Shrine** / optional **CrackedWall** / **Exit**) from entrance–exit heuristics + storage-room bias for chest/barrel/crate (Exit spawns at/near `gen.exit`). Each blocking PoI candidate is accepted only if **progression stays valid**: from the party spawn rule at **`entrance`** (nudge off a PoI on that tile), BFS over **`isWalkable`** tiles treats every **placed PoI cell** and the **exit** cell as non-traversable and must still reach **at least one orthogonal neighbor of `gen.exit`** (so the Exit POI remains usable from beside the stairs).
   - NPCs and floor items from **`spawnTables.ts`** (room function, **`roomStatus`**, district, **`floorType`**, **`floorProperties`**, and whether the room center lies on a shortest entrance→exit path before locks) so balancing can stay data-shaped without touching placement loops
8. **Lock/key pass** on the entrance→exit **shortest path**: place **ordered** separating `lockedDoor` tiles when possible. Long paths may get **two** gates (**A** + **IronKey**, **B** + **BrassKey**); shorter paths use **one** (**A** + **IronKey**); otherwise skip locks. Keys carry **`forLockId`** in **`floor.gen.floorItems`** for validation/debug.
9. **`missionGraph`**: nodes (entrance, exit, POIs, locks, keys); **path** edges chain POIs by BFS distance from entrance, then the lock sequence, then exit; **`shortcut`** edge (entrance→exit) is present when multiple shortest-length routes exist; **`hasAlternateEntranceExitRoute`** mirrors that flag for tools.
10. **Theme**: **`floor.gen.theme`** (`id` + floor/wall/ceiling **color** multipliers) from a dedicated RNG stream; **`WorldRenderer`** tints **Lambert** dungeon materials (textures unchanged).

**First load**: `makeInitialState` runs the same `generateDungeon` pipeline as `floor/regen` (**31×31**), with **`floor.seed`** (random on cold start), **`floor.floorIndex`** (0 on first floor), **`floor.floorType`** (default **Dungeon**), **`floor.floorProperties`** (default empty), and **`floor.difficulty`** (default **1**). The canonical bundle is **`floor.gen`** (`genVersion` **5**); the party spawns at **`gen.entrance`**.

**Meta-progression**: floors are advanced by interacting with an **Exit POI** placed at/near `gen.exit`. Descending increments **`floorIndex`**, cycles **`floorType`** (Dungeon→Cave→Ruins), and generates a **new random `floor.seed`** for the next floor.
Default floor dimensions are **31×31**; `floor/regen` keeps the current **`floor.w` / `floor.h`** and reapplies **`floorType` / `floorProperties` / `floorIndex` / `difficulty`** from state unless you add UI to change them.

### 8.3 Determinism requirement
Generation should be **seeded** and phase-stable (changes to one phase shouldn’t ripple unpredictably into others) to support future host-synced multiplayer.

Validation is part of generation: the generator runs **bounded deterministic rerolls** (attempt count depends on **`difficulty`**; see §8.1) when **ordered** lock/key constraints fail, when **safety POIs** violate reachability caps (**Well** within **3** BFS steps of **`entrance`**, **Bed** within **48**), when **PoI layout** would block **spawn → exit-adjacent** reachability (same rule as placement; applies even when there are **no** procgen locks), or when the floor has **any** procgen **locks** but the entrance→exit shortest-path **lattice** is not wider than a single spine (no meaningful alternate route of the same shortest length). Among attempts that pass validation, the run picks the one with the highest **soft `layoutScore`** (reachable floor mass, dead-end penalty, path length bonus, **junction** count for 3+ neighbor floor cells, bonus when that lattice is wide, extra bonus when locks are present and the lattice is wide — **lock+loop** term scaled by **`difficulty`**); if none validate, it returns the **last** attempt (still full geometry) like before. Output always has a non-empty **`rooms`** list except catastrophic fallback (single carved room).

**Meta** records **`inputSeed`**, **`attemptSeed`**, **`attempt`**, **`layoutScore`**, **`difficulty`**, and **phase streams** (`layout`, `tags`, `population`, `locks`, `districts`, `score`, **`theme`**, **`mission`** for future / reserved **`planMissionBeforeGeometry`**).

### 8.4 Reference pipeline (expanded)
See `Dungeon_generation_plan_summary.md` for an expanded phased pipeline (mission graph → lock correctness → geometry → districts → tagging solve → population → validation → debug overlays). The **geometry-first** pipeline is still what ships; **`web/src/procgen/missionFirst.ts`** defines **`PlannedMission`** / **`planMissionBeforeGeometry`** (returns `null` today), the **embedding contract**, and an in-file **Track B** specification (abstract plan vs realized **`missionGraph`**, per-**`floorType`** embed strategy, future **`genVersion`** when embedding ships).

**Debuggability**: the F2 Debug panel includes a **Procgen** readout (**difficulty**, **floorType**, **floorIndex**, **genVersion**, **`streams.mission`**, **attempt**, **layoutScore**, **theme** id, **missionGraph** size and alternate-route hint), **Cycle type** (Dungeon→Cave→Ruins for the next regen), **Cycle difficulty** (easy→normal→hard for the next regen), **Proc overlay** (cycles **off** / **districts** / **roomTags** / **mission** tints on the 3D floor from **`floor.gen`**), and **Dump `floor.gen`** to download the canonical JSON.

## 9) Points of interest (POIs)
Static interactables. POIs **do not** auto-trigger on **walking past** them, but their grid cell **blocks player occupancy**: you **cannot stand on the same tile** as a POI (see §6.4). Interact via **click** or by **dragging items onto them** (typically from beside the object in the 3D view).

- **Placement invariant**: there is **at most one POI per grid cell**. If procgen produces multiple POIs for the same cell, the game resolves it deterministically (canonical IDs win, then kind priority) so runs remain stable and interactions remain unambiguous. **Progression invariant**: no PoI may sit on a tile that would make it **impossible** to walk from the **entrance spawn** (see §6.4) to **beside the Exit** while treating all PoI tiles and the **exit stair** tile as blocked; procgen skips or relocates candidates (optional **Shrine** / **CrackedWall** may be omitted if no valid tile exists).
- **3D view**: POIs render as **sprite billboards** (same texture pipeline as NPC billboards: nearest filtering, transparent PNG). **Well (filled)** uses **`npc_well.png`** plus extra **non-pickable** billboards: **`npc_well_glow.png`** (slightly larger halo) and a small **sparkle** layer cycling **`npc_well_sparkle_1..3.png`** (~280 ms per frame). **Well (drained)** uses **`npc_well_drained.png`** only (no glow/sparkle). **Chest**, **Barrel**, and **Crate** use closed/open PNG pairs (`chest_*`, `barrel_*`, `crate_*`). **Exit** uses **`stairs_down.png`** at **2×** the default POI billboard scale (other POIs use a uniform base height). **Shrine** uses **`shrine_gnome.png`** until the first interaction, then **`shrine_gnome_off.png`**. **Bed** and **CrackedWall** use **`/content/poi_placeholder.png`** (a copy of **`Placeholders/Placeholder_NPC.png`**) until dedicated POI art exists. Billboards use a **bottom-centered pivot** (`Sprite.center`) so the texture’s lower edge anchors to the floor; vertical placement is **`poiFootLift`** (F2: **POI above ground**, independent of **`npcFootLift`**). Per-art **`poiGroundY_Well`** / **`poiGroundY_Chest`** (and **`npcGroundY_Wurglepup`** for other kinds) still shift where the “ground contact” sits within the PNG. Brightness uses **`render.poiSpriteBoost`** (F2; default tuned brighter than 1.0).

Initial POIs:
- **Well**: **checkpoint save point** (clear notification); used to fill Waterbag; a successful **Waterbag (Empty)** use on this well sets the POI to **drained** (visual swap + VFX off); **checkpoint save still works when drained**. Checkpoints are reloadable from the **death screen** and the **title** screen.
- **Chest**: opens (sprite change) and drops random item
- **Barrel**: opens (sprite change) and drops random item
- **Crate**: opens (sprite change) and drops random item
- **Bed**: restores full HP and Stamina
- **Exit**: descends to the next floor (meta-progression)

## 10) Audio
Three audio layers run simultaneously via the Web Audio API:

- **Background music** (`MusicPlayer` / `MusicLayer`): loads an audio file, loops it continuously, and applies a tunable master volume (`masterMusic`). Starts on first user interaction (browser autoplay policy). Currently plays `Assets/sounds/theme.mp3` (served from `public/sounds/`).
- **Spatial ambient** (`SpatialAudio` / `SpatialAudioLayer`): NPCs and POIs emit continuous synthesized tones audible from a configurable number of cells away. Attenuation uses both **lowpass** and **volume** with distance (tunable in F2).
- **Procedural SFX** (`SfxEngine` / `FeedbackLayer`): discrete one-shot sounds generated from scratch using oscillators, noise, envelopes, and filters. Triggered via `ui.sfxQueue`. Kinds: `ui`, `hit`, **`swing`** (weapon swing / to-hit miss, distinct from UI **`reject`**), `reject`, `pickup`, `munch`, `step`, `bump`.

Volume controls: `masterMusic` (music layer) and `masterSfx` (SFX + spatial) are both in `AudioTuning` and adjustable in F2.

## 11) Graphics & rendering spec
- Three.js WebGL with `MeshLambertMaterial` on dungeon geometry (diffuse + emissive; no PBR metalness/roughness).
- **Dungeon albedo**: floor / wall / ceiling voxels use tiled PNGs from `Content/` (stable **`/content/…`** URLs; Vite serves the repo **`Content/`** tree in dev and copies it into **`dist/content/`** on build). **`FloorType` `Dungeon`** uses **`dungeon_floor.png`**, **`dungon_wall.png`**, and **`dungon_ceiling.png`**. **`Cave`** uses **`cave_floor.png`**, **`cave_wall.png`**, and **`cave_ceiling.png`**. **`Ruins`** uses **`ruins_floor.png`**, **`ruins_wall.png`**, and **`ruins_ceiling.png`**. Textures use **repeat wrapping** (~one repeat per 1×1 world-unit face).\n+- **Overgrown rooms (selective env swap)**: procgen rooms with `roomStatus: Overgrown` render their **floor + ceiling** using **`overgrown_floor.png`** and **`overgrown_ceiling.png`** (served at `/content/overgrown_*.png`). **Walls** bordering an overgrown walkable tile use **`overgrown_wall.png`** (4-neighbor adjacency) so room boundaries read clearly without changing the core geometry pipeline.
- Anti-aliasing **off**; pixel ratio capped at **1.5×** (sharp edges, dither-friendly).
- Lighting:
  - warm **PointLight** on camera (“lantern”)
  - optional forward **SpotLight** “beam” aligned to camera direction (tunable; often off via intensity scale). When effective beam intensity is zero, the spot is **not visible** and does **not** cast shadows (no wasted shadow pass).
  - POI **torch** PointLights: up to **`torchPoiLightMax`** (F2), choosing the **nearest** POIs to the player by Manhattan grid distance (0 disables torches)
  - torch flicker via per-torch sine intensity
  - **Shadow mapping** is enabled only while at least one lantern light has shadows on: **point** shadows (expensive cube map) and **beam** shadows are separate F2 toggles (`shadowLanternPoint`, `shadowLanternBeam`). **Shadow map size** and **filter** (basic / PCF / PCF soft) are tunable in F2. Dungeon geometry casts/receives where applicable.
  - low emissive lift on dungeon materials; the lantern is the primary visibility driver
  - `FogExp2` is **optional** for depth falloff; **disabled by default** (can be enabled in F2 debug); the renderer reuses one fog instance instead of allocating every frame
- Post-process: ordered dithering as shader pass (EffectComposer), controls:
  - Strength
  - Colour preserve
  - Pixel size
  - Levels
  - Post-dither levels (classic lift/gain/gamma; affects HUD + 3D equally)
  - Matrix (Bayer 2×2/4×4/8×8)
  - Palette (Dungeon warm, Cold crypt, Monochrome, Sepia print, No palette snap)
  - **Warm palette mix** (F2): when palette is **Dungeon warm** (0), blends between **quantised dither only** (like no snap) and **full warm five-colour snap**; other palette indices ignore this control
- **Room-property telegraph (compositor)**: when the player is standing in a room tagged with a **room property** (`Burning` / `Flooded` / `Infected`), the compositor applies a **luma-preserving color tint override** (blend toward a hazard RGB at uniform strength over the full 3D viewport—**no vignette**) so the hazard read clearly **overrides** the floor theme cast while keeping local brightness structure. Hazard telegraph **intensity pulses slowly** (~**3.8 s** sine) by scaling the **graded tint only** (fixed hazard RGB ratios), not the blend weight—so the cast stays **one hue** and does not swing toward unrelated scene colors. **`Burning`** reads as **hot red**; **`ruins_umber`** floor theme lighting uses **brown umber** intent (not fire red) so the two do not collide. Room lookup matches **`roomForCell`** (containment only; no nearest-room fallback). Tuning/preview can be forced via the debug telegraph override.
- **Room-hazard floor decals (3D)**: `WorldRenderer` draws **non-interactive** sprite billboards on **`floor`** tiles inside procgen rooms whose **`tags.roomProperties`** matches the hazard: **`hazard_fire.png`** / **`hazard_water.png`** / **`hazard_poison.png`** (URLs under **`/content/`**). Placement is **sparse and deterministic** (~**40%** of eligible cells, keyed by **`floor.seed`** + room id + cell), and cells occupied by a **POI**, **NPC**, or **floor item** are skipped. Decals sit low on the floor (ground-stain height), use the same **theme tint** as other billboards, and rebuild with **`floorGeomRevision`**.
- **Frame presentation pipeline**:
  - The 3D world is rendered offscreen into a **render target** sized to match the on-screen **game viewport rect** (the HUD “game” panel), not necessarily the full window.
  - The HUD exists as HTML/CSS twice:
    - an **interactive HUD** (`opacity: 0` over the final canvas but above the WebGL presenter, handles pointer input)
    - a **capture HUD** (offscreen, non-interactive) that is rasterized into a canvas texture. The capture HUD does not mount **`GameViewport`** (the 3D view is not part of **`html2canvas`**); the **game** grid cell uses an **invisible same-sized shell** with the same **`gameViewportRef`** attachment pattern for compositor/game-rect parity—**except** when **`ui.screen === 'hub'`**, where **`HubViewport`** renders the **2D hub image + hotspot outlines** into both **interactive** and **capture** trees so **`html2canvas`** picks up **opaque** pixels in that cell and the compositor **masks** the 3D scene there (same mechanism as other HUD-over-scene blending). The **dithered NPC dialog** mounts inside that **game** cell (**`npcCaptureLayer`**: **stretch**ed overlay; inner **`captureRoot`** fills the cell and pins **top panel + bottom speech strip**) rather than as a sibling of the whole **`HudLayout`**. The **death** modal (**real** or **debug preview**) uses the same **game** cell overlay path (**priority over NPC** in capture when both would apply): dim + panel are **centered** in the game widget. **Title** and **paperdoll** mount as **full-bleed capture** overlays on the entire HUD grid (**`captureFullHudOverlay`** → **`fullHudCaptureLayer`**). **Interactive** death mirrors the viewport with **`gameViewportRef`** + **`position: fixed`** bounds (like the NPC portaled hit layer).
  - **Modal interaction**: an **offscreen capture** variant feeds the **dithered** HUD texture. **Title**, **death**, **paperdoll**, and **NPC dialog** use **`createPortal`** to **`document.body`** at high **`z-index`** (**above** **`CursorLayer`**) with **`GamePopup`’s** invisible **`modalPortalHitRoot`** or **`NpcDialog`**-style layers so hits win over **`FixedStageViewport`** **`transform: scale`**. **`TradeModal` (interactive)** is different: it mounts **inside** **`HudLayout`**’s **game** cell (**same depth as inventory**, **ADR-0258**) with **`position: absolute`** overlays in **stage** coordinates; **capture** trade still comes from **`captureFullHudOverlay`**. **`DitheredFrameRoot`’s `stageModalLayer`** mounts **`TitleScreen`**, **`DeathModal`**, **`PaperdollModal`**, and **`NpcDialogModal`** (not interactive **trade**) while the corresponding **`ui.*`** flags apply; the layer uses **`pointer-events: none`** because real modal hits are on the portaled roots.
  - Because the capture HUD is rasterized by **`html2canvas`**, prefer conservative CSS for critical visuals (avoid relying on `object-fit` for aspect-correct sprite presentation; use intrinsic sizing patterns instead).
  - **HUD shell art**: a single transparent **`ui_hud_background.png`** (`Content/ui/hud/`, mirrored to `web/public/content/ui/hud/` for some workflows) is drawn full-bleed behind the HUD grid in `HudLayout` (CSS `::before`, `background-size: contain`, centered). **Per-character equipment** uses a separate strip asset **`ui_hud_charequip_background.png`** in the portrait rails (also under `Content/ui/hud/`); it is part of the same DOM HUD capture and is dithered with the rest of the UI. Existing widgets stay in the current CSS grid; opaque “glass” panel styling is removed so content sits on the artwork (fine-tune slot positions vs the art next).
  - Presenter sizing is derived from the **viewport CSS size** (prefer `visualViewport.width/height`, else `documentElement.clientWidth/clientHeight`) rather than measuring the presenter canvas element, to avoid resize feedback loops caused by renderers writing inline canvas CSS sizes.
  - A presenter compositor shader places the scene render target into the frame **only inside** the game viewport rect and overlays the captured HUD everywhere else (and over the scene where UI alpha exists).
  - The final composite then runs through the ordered-dither post-process so the **same dithering/pixelation** applies to both 3D and HUD.

### 13.1 Asset serving & caching (web)
- Runtime-served assets (portraits, NPC sprites, UI art, cursor sprites) live under repo-level `Content/` and are referenced via **stable** `/content/...` URLs.
- The game relies on **browser HTTP caching** for these static URLs; avoid cache-busting query params for runtime art.
- Imperative image loads (e.g., measuring portrait aspect ratio or preloading sprite layers) go through a small shared **in-app image cache** (`web/src/ui/assets/imageCache.ts`) to dedupe concurrent loads and reduce repeated decodes/revalidation requests.

### 13.2 Procgen content coverage (data discipline)
- **Seeded item defs** live in `web/src/game/content/items.ts` (`DEFAULT_ITEMS`). **Not every item is meant to appear from dungeon generation**: some are **craft-only**, **Well transforms**, or **quest-chain** items. **Headwear** (`WoolCap`, `HerbCirclet`, `SporeCap`) is both **craftable** and can appear from **floor spawns** and **POI container** loot when wired in `spawnTables` / `poiLootTables`.
- **Procgen** places floor items via `web/src/procgen/spawnTables.ts` (`pickFloorItemDefFromTable`); **locks** add keys via `web/src/procgen/locks.ts`. **Chest / barrel / crate** opening uses deterministic tables in `web/src/game/content/poiLootTables.ts` (rolled in `web/src/game/state/poi.ts`). **Neutral NPC quests** reference a small want/hated item set from `web/src/procgen/population.ts`.
- **Policy**: when adding a new `ItemDefId`, either wire it into one of those pipelines or add it to **`ITEM_DEF_IDS_INTENTIONALLY_NON_PROCGEN`** in `web/src/tools/procgenContentNonSpawnAllowlist.ts` so the audit stays honest.
- **Audit CLI** (from `web/`): `npm run audit:procgen-content` — compares seeded items to the union of procgen + POI loot + quest references; exits non-zero if a seeded item is neither used nor allowlisted, or if listed ids are missing from `DEFAULT_ITEMS`. **Status effects** are not placed by procgen; they are applied in gameplay (combat, feed, hazards). Items may reference statuses via `feed.statusChances` (e.g. **Blessed** / **Sick** on mushrooms).
