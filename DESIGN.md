## 1) High concept
**Elfenstein** is a web-based, grid-based dungeon crawler with a first-person **2.5D** view. Visuals combine **3D dungeon geometry** with **2D sprites** for characters/items, prioritizing a crunchy dithered/print look and a highly contextual, low-clutter UX.

## 2) Product goals
- **Modular + extensible**: content and stats are compartmentalized and easy to expand (items, NPCs, species, classes, floor types, etc.).
- **Multiplayer-ready later**: generation and state should be compatible with a future model where the **host player** runs the authoritative database and **1–3 clients** connect.
- **Contextual UX over menus**: few actions are available at any moment; actions branch into contextual interactions/modals.

## 3) Target platform & constraints
- **Platform**: Web (desktop first).
- **Automated testing**: **`web/`** uses **Vitest** for game logic and reducers; **Playwright** (Chromium) covers browser smoke E2E against **`vite build` + `vite preview`** (`npm run test:e2e`), including title/settings, **Start** → **title** dialog closes while **Bobr** plays over **hub** → full **8.6s** intro → village hub, **Cave** → **`screen: 'game'`** (nav pad **Step forward** enabled), tavern **Leave tavern** round-trip, and innkeeper **Trade** open/**Close** (`web/e2e/smoke.spec.ts`, **`web/e2e/hub.spec.ts`**, shared **`web/e2e/helpers.ts`**). Interactive village/tavern hotspots expose **`data-testid`** **`hub-hotspot-tavern`**, **`hub-hotspot-cave`**, **`hub-hotspot-innkeeper-trade`** on **`HubViewport`** **`HotspotBox`** buttons (**ADR-0320**). Browsers install to **`web/node_modules/.cache/ms-playwright`** via **`npm run playwright:install`** (uses **`--with-deps`** when **`CI`** is set). CI: **`.github/workflows/web.yml`** (**ADR-0319**).
- **Layout**: HUD and compositor are authored at **1920×1080** CSS px inside **`FixedStageViewport`** (`stageDesign.ts`). The **`.clip`** is exactly the scaled **1920×1080** content box (**`STAGE_CSS_*`** in **`stageDesign.ts`**). A **3px** white frame uses **`outline`** on **`.clip`** with negative **`outline-offset`** so it sits just inside that rectangle without changing layout size. The stage **uniformly scales** by **`s = min(1, viewportW/1920, viewportH/1080)`** (using **`visualViewport`** when present): on viewports **larger than 1080p** the box stays **1920×1080 CSS px** centered with black margins; smaller viewports **shrink** so everything remains visible. **All stage-internal layers (presenter canvas, interactive HUD hit layer, capture HUD) are stage-relative (positioned against the 1920×1080 stage), not browser-viewport fixed**, so letterboxing/margins can’t offset the 3D viewport vs HUD. **`FramePresenter`** / WebGL canvas and **HUD `html2canvas` capture use the same 1920×1080 CSS size**—not the browser viewport. **`getBoundingClientRect()` includes that outer scale**, so the compositor’s **`gameRectPx`** and the **3D render target** use **layout** sizes (divide by **`s`** where needed, and **`clientWidth` / `clientHeight`** for the game viewport) so the scene + UI map correctly inside the stage. In-game CSS uses **`--stage-w` / `--stage-h`** instead of **`100vw` / `100vh`** for max widths/heights where needed.
- **Camera**: first-person, grid movement; no looking up/down; pits can exist but player can’t fall in. **Game yaw** (from `playerDir` / `view.camYaw`) uses forward \((\sin y,\,-\cos y)\) on the XZ plane; **Three.js** `camera.rotation.y` is set to **`-y`** so the rendered view matches that forward (Three’s Y rotation would otherwise flip the X component of view direction).
- **Renderer**: Three.js/WebGL for dungeon geometry. The **compositor** (`FramePresenter` + **`CompositeShader`**) blends a **captured HUD** texture (`html2canvas`) with the **3D** render target. **`DitheredFrameRoot`** drives **one** **`requestAnimationFrame`** loop that calls **`renderOnce`** per frame (stable pacing; HUD rasterization stays **async** via idle scheduling). On **`screen: 'title'`**, until the first capture exists, a **boot placeholder** **`CanvasTexture`** (**`/content/village.png`**, **cover**-fit to the stage) is used as **`uiTex`** so the **3D** view does not flash through the game cell (**ADR-0302**); replaceable later with dedicated title art. When **`screen`** becomes **`hub`** but the last successful HUD capture was still for **`title`** (tracked via **`lastCapturedHudScreenRef`**, not only **`lastHudKeyRef`** string parsing), the compositor uses that same **village boot** texture instead of **`uiTexRef`** (including after a **scale** dispose leaves **`uiTex` null**) until **hub** capture completes (**ADR-0326**, **ADR-0327**).
- **GPU quality**: **`GameState.render.gpuTier`** is **`low`**, **`balanced`**, **`high`**, or **`custom`**. Presets set **`pixelRatioCap`** (1.0 / 1.25 / 1.5, clamped to the device DPR), **lantern shadow** toggles and map size/filter, **max POI torch lights**, and lighter **dither** on Low. **`pixelRatioCap`** applies to **`FramePresenter.syncSize`**, **`WorldRenderer.syncViewportRect`**, and **`html2canvas`** scale so backing-store resolution stays matched. **`custom`** is set when any **tier-owned** field is changed via F2 **`render/set`** sliders. **Settings** (§6.4) exposes the same three named tiers plus a read-only **Custom** row when applicable (**ADR-0318**).
- **Debug (F2)**: sliders for render/audio tuning, including **GPU quality tier** (**Low** / **Balanced** / **High**) plus **pixel ratio cap**, **camera** (eye height, field of view, optional pitch for development) and **lighting** (per-mode **camera** intensity/distance: bare, held torch, equipped lantern, headlamp, glowbug; **POI torch** intensity/distance; lantern beam + offsets + flicker; base emissive lift; **shadow** toggles for point/beam, shadow map size and filter, max **nearest** POI torch lights). Includes quick procgen tools: **Regen**, **Descend (debug)**, **set `floorIndex`** (apply and apply+regen), and **floor property toggles** (`Infested` / `Cursed` / `Destroyed` / `Overgrown`) which take effect on Regen/Descend (**not** written to `debug-settings.json`). Includes **Spawn (1 cell ahead)** for **NPC**, **POI**, and **item** (pick any seeded `ItemDefId`; item requires the cell ahead to be **floor**). **Hub hotspots**: **x / y / w / h** sliders (0–1, normalized to the **game viewport** cell) for **village** regions and **tavern** **`innkeeper`** (bartender sprite frame) + **`innkeeperTrade`** (open trade; shipped default **~34%** width × **~58%** height of the viewport, **`x` ~0.31**; **`y`** does **not** set vertical position—the trade **`HotspotBox`** uses a fixed **`350px`** top inset in **`HubViewport`**; **`.hotspotTrade`** uses **no** visible border); **Leave tavern** is the labeled button in **`HubViewport`**, not a hotspot. Values persist in **`hubHotspots`** inside `debug-settings.json` and local storage like render/audio. **Hub innkeeper sprite scale** (F2, NPC/POI render sliders): scales only the **tavern bartender** 2D sprite visually (**CSS** `transform` from bottom center); the **innkeeper hotspot** rect is unchanged unless hotspot sliders are edited—persists with other **render** keys in `debug-settings.json` / local storage. **`debugUi`** in the same JSON / local snapshot persists **Proc overlay** mode, **debug BGM track** override, **room telegraph** (mode + strength for the compositor tint), and the **NPC dialog** / **death** modal **preview** checkboxes. **UI**: those checkboxes show the **NPC dialog** modal (uses the **first NPC** on the current floor; no encounter state) and the **death** modal (current run stats; does **not** set `ui.death` or block gameplay). Includes a small **Pose** readout (playerDir and camera yaw values) for diagnosing orientation issues and a **Cell** readout showing the **player cell** with tile + procgen room tags (district/roomFunction/etc). **Portrait**: portrait shake envelope (**hold/decay**) + amplitude, mouth flicker (**Hz** + **amount**), and min/max gap (ms) between Igor **idle** flashes and min/max **flash** duration (ms). **Audio** includes **master music** volume, spatial emitter mix, and **munch** SFX (volume, noise **LP sweep** endpoints, **HP** corner and **HP/LP Q**, duration, thump Hz, tremolo). Values load from `web/public/debug-settings.json` first, then **localStorage** overrides on the same keys. Slider edits debounce-save to **localStorage** (~450 ms) and, under **`vite dev`**, to **`web/public/debug-settings.json`** (~2 s) using **`POST /__debug_settings/save`** (same path as **Save to project**). Persisted **`render`** / **`audio`** objects include **every** key from the current **`DEFAULT_RENDER`** / **`DEFAULT_AUDIO`** (so new fields such as per-mode camera lights are not dropped). **Save to project** (F2, **`vite dev` only**) still triggers an **immediate** write and a success/failure toast. **Clear local overrides** (F2, **all builds**) removes **`elfenstein.debugSettings`** and reloads so the shipped **`debug-settings.json`** applies without stale **localStorage**. There are **no always-on on-screen debug overlays** during play; debug UI is accessed via **F2** (and renderer-only debugging uses explicit query params when needed). Pitch is a debug aid only; core UX remains yaw-on-grid.

## 4) Core player experience (pillars)
- **Touch what you see**: the hand cursor is the primary verb (click, drag, drop).
- **Party as a single organism**: shared inventory with a **fixed** slot grid (see §7.2).
- **Discovery through interaction**: crafting by experimentation; NPC language as a memorization puzzle.
- **Atmosphere via light + sound**: darkness is default; proximity audio supports exploration.

## 5) Core loop
Explore → find POIs/NPCs/items → manage inventory/craft → resolve encounters → recover at POIs → push deeper (keys/doors) → repeat.

## 6) Controls & interaction model
### 6.1 Cursor (hand) states
- **OS cursor**: hidden globally (`cursor: none` on `document.body`); interactive HUD widgets also set **`cursor: none`** so controls (e.g. navigation buttons) never revert to the system pointer—only the **`CursorLayer`** hand sprite is visible. **`CursorLayer`** is a **sibling** of **`FixedStageViewport`** (not inside the scaled **`.stage`**): **`position: fixed`** with **`clientX`/`clientY`** must use the **viewport** containing block; a transformed ancestor would offset the hand from the real pointer. **`CursorLayer`** uses **`z-index: 9999`** above most UI; **all descendants use `pointer-events: none`** so the hand/ghost/affordance never **capture** clicks meant for DOM below. The **NPC dialog** portaled hit layer uses **`z-index: 10100`** (**above** the cursor layer) so **Close** / drag targets stay reliable; it stays **visually empty** (**`opacity: 0`**) so the hand remains visible through it.
- **Default**: `Hand_Point`
- **Combat attack (3D NPC hover)**: **`hand_attack_01.png` / `hand_attack_02.png`**, **~280 ms** flip; encounter active, acting PC’s turn, pointer over a hostile in-roster NPC, weapon resolves (`resolveWeaponItemIdForPcTurn`); dragging a **weapon** still shows **⚔ Attack**.
- **Hub tavern (open trade hit)**: **`hand_trade_01.png` / `hand_trade_02.png`**, same cadence. **`screen: hub`**, **`hubScene: tavern`**, **`innkeeperTrade`** has **`data-drop-kind="hubInnkeeperTrade"`**; **`HubViewport`** publishes the hotspot’s screen rect to **`hubTavernTradeHoverRectRef`**, and **`CursorProvider`’s `applyPointerMove`** sets **`virtualHover`** to **`hubInnkeeperTrade`** when the pointer lies inside that rect (**before** `elementsFromPoint`), so **`hoverTarget`** matches under the **opacity-0** interactive HUD. Drop on that region rejects (**`ui/sfx` `reject`**).
- **Holding/dragging**: `Hand_Hold` (held item sprite follows pointer)
- **Hover active**: `Hand_Active` when pointing at an interactable
- **Craft-ready hover**: while dragging an inventory item over another inventory item that can be crafted with it, the hand cursor flickers between `Hand_Hold` and `Hand_Active` and the affordance pill shows **⚗ Craft**; an icon badge shows `?` for unknown recipes or the result icon for discovered recipes.
- **Click micro-feedback**: on **pointer down**, the hand cursor can perform a **tiny shake** (purely cosmetic; debug-tunable in F2 under **Cursor**). This is a *micro* cue meant to add tactility without implying a successful action (success/failure feedback still comes from SFX/log/shake events).
- **Global pointer sync**: **`CursorProvider`** listens for **`pointermove`** on **`window`** (**passive**) and runs the same hit-test / drag-promotion path as the legacy React **`onPointerMove`** API, so the hand tracks the pointer over **any** DOM (including **`stageModalLayer`** modals that are **not** under **`HudLayout`**, e.g. **Settings**). Per-widget **`onPointerMove`** forwarding was removed to avoid double updates and to keep ordering with **`setVirtualHover`** predictable (**React** handlers on the HUD tree run in the same native event **before** the **`window`** bubble listener consumes the one-frame virtual hover ref).
- **Deadzone behavior**: hovering non-interactive UI areas should clear any previous “active” hover so the cursor returns to `Hand_Point`; combined with the global listener, cursor state does not get stuck when crossing panels or modals.
- **Title / hub vs dungeon**: **3D pick / `floorDrop` virtual hover** runs only when **`ui.screen === 'game'`** (not on **title** or **hub**), so the center cell does not pin **`hoverTarget`** to **`floorDrop`** and block **`Hand_Active`** on real UI.
- **Menu chrome without `data-drop-kind`**: **Title** and **Settings** primary actions set **`data-cursor-hand-active`** so **`CursorLayer`** can show **`Hand_Active`** over those controls (via **`elementFromPoint`** + **`closest`**), matching other interactive affordances.
- **3D viewport hover override**: during **game**, the WebGL viewport may inject a **virtual hover target** (e.g. `floorDrop`) on pointer-move to compensate for `elementFromPoint` seeing the compositor canvas rather than a DOM hit target. This override is **ephemeral (one move event)** so normal DOM-derived hover targets (inventory slots, portraits, etc.) take over immediately when the pointer leaves the viewport.

### 6.2 Interaction rules (mouse-first)
- **Click**: attempt to use/interact with target (object/NPC/POI/UI element).
- **3D viewport ray pick**: when several pickables lie on the same camera ray, **floor items take priority** over POIs (and over NPCs/doors on that ray) so loot is not blocked by large POI billboards (e.g. a chest in front of a dropped item).
- **Drag (items)**: dragging starts when you **press and move** beyond a small threshold (to avoid accidental drags on click). **Press + hold** is also supported as a fallback to begin dragging items from **inventory** and from **world floor items** in the 3D view. **Movement promotion** in **`CursorProvider`** uses the same **synchronous refs** as **`beginPointerDown`** (**`pendingPayload`**, **`dragStartPos`**, **`dragPromotedRef`**) so the first moves after **`pointerdown`** are not ignored while React state is still catching up (otherwise the UI falls back to the **~140ms** hold timer and feels laggy). **`pointerup`** / **drop** resolution uses the same **“drag promoted”** flag so **`pointerdown`’s `preventDefault`** does not strand **tap-to-pick** UX (e.g. wanted inventory → **trade offer**) waiting on a stale **`dragging.started`** read. **Trade** / **inventory** item buttons still use **`stopPropagation` on `pointerup`** where needed so parent handlers do not double-resolve drops.
- **Drop**:
  - Onto **inventory**: store item
  - Onto **empty 3D view**: **cursor-aimed drop** (raycast to the floor under the pointer), clamped by a max Manhattan range (tunable). If the aimed cell is invalid/blocked, snap to the nearest valid floor cell within range; if none exist, fall back to dropping near the player. **POI / NPC / floor-item** targets use the same **`WorldRenderer.pickTarget`** raycast at pointer-up via **`resolveGameViewportDragDropTarget`** (not DOM `data-drop-kind`), including drags that start on **inventory** or **equip** slots—those use **`pointerCapture`**, so **`HudLayout`’s** root **`pointerup`** does not run (**ADR-0302**, **ADR-0303**).
  - Onto **another item**: attempt crafting / recipe discovery (see §7.3)
  - Onto **NPC**: use item on NPC (may be rejected/consumed/apply status/attack)
  - Onto **portrait eye**: inspect interaction
  - Onto **portrait mouth**: feed interaction
  - Onto **character equipment strip** slots (**`head`**, **`handLeft`**, **`handRight`**): equip via **`equipmentSlot`** drops (same rules as §7.4 / paperdoll; **`equipItem`** in the reducer)
  - While **`ui.tradeSession`** is open (**§5.1**): onto **trade offer** slot — stage an inventory item whose **`defId`** is in the merchant’s **`wants`** (item leaves the grid until traded or the modal closes), or **click** a **highlighted** inventory slot to stage it (no drag); **their stock** — **click** only to select what you receive (**no** drag from stock); drag the staged offer back onto **inventory** to return it
- **Timing**: drop resolution uses the pointer-up timestamp (same `performance.now()` clock as `time/tick`) so short-lived UI cues (e.g. portrait mouth “chomp” burst) can’t expire immediately due to a stale tick time.
- **Preview affordance**: contextual icon appears near relevant elements indicating what releasing will do.

### 6.3 UX feedback
Interactions should resolve with:
- short relevant animation (e.g. eyes/mouth)
- sound
- **Activity log** (**`ui.activityLog`**, rendered as **`ActivityLog`** in a **bottom-right corner stack** in **`HudLayout`** with **`CombatIndicator`** during encounters—the log sits **above** encounter chrome so the two do not overlap): each line of feedback (POI outcomes, crafting, combat hits, door messages, inventory pickup/full, inspect/feed text, debug **`ui/toast`** dispatches, etc.) is **appended** (newest last); the list is **capped** in memory (**`ACTIVITY_LOG_MAX_ENTRIES`**), each line **expires** after **10 seconds** of game time (**`ACTIVITY_LOG_ENTRY_TTL_MS`**, pruned on **`time/tick`**; pruning is **paused** while the **death** modal is up so the recap stays stable), and the widget **shows at most the four newest** entries. It uses **no** filled panel background (text sits over the scene, using **`--buttonTitle*`** color and shadow for contrast). **`ActivityLog`** applies **`pointer-events: none`** to **all** descendants so when the stack is **raised above** tavern trade (**`gameCornerStackAboveTrade`**, §5.1) long lines cannot **intercept** clicks meant for **Trade** / modal chrome. Lines are **right-aligned** in that corner stack. Typography reuses **`--buttonTitleFontFamily`** / spacing / color / shadow from **`index.css`** at **`font-size: calc(var(--buttonTitleFontSize) - 5px)`**, with **regular (400)** weight (unlike **`--buttonTitleFontWeight`** on modal buttons). There is **no** separate centered transient **`ui.toast`** overlay; the **`ui/toast`** action remains as a convenience and **only** pushes a log line.
- subtle camera shake for key interactions (driven by `ui.shake` and tunable in Debug/F2). **Shake length / hold (ms)** and **shake decay / fade (ms)** control the envelope for the **3D camera**, **HUD overlay shake**, and **portrait frame shake** (shared `shakeEnvelopeFactor` in `web/src/game/shakeEnvelope.ts`). With length 0, decay uses the legacy ramp `min(1, remaining/decay)`; with length above 0, full strength holds for the scaled hold segment, then linearly fades over the scaled decay segment within each event’s `startedAtMs`→`untilMs` window.
- **Portrait frame shake** on portrait interaction resolution (inspect and feed) and when an **NPC hit** deals damage to a PC in **`npcTakeTurn`**, driven by `ui.portraitShake` on the **struck** character’s slot (existing encounter **`ui.shake`** for the 3D view is unchanged—no second global shake)

### 6.4 Grid movement (first-person)
- **Model**: discrete cells; facing is one of four compass directions (**`playerDir`** is canonical 0=N,1=E,2=S,3=W); the 3D camera **yaw** matches facing when idle. **Strafe** moves one cell **left or right relative to facing** without rotating (**A** / **D**). **W** / **S** move **forward** / **backward** along facing.
- **Turning**: left/right turns always animate the **shortest** 90° rotation. Internally, the animated camera yaw may be temporarily **unwrapped** across the \(0 \leftrightarrow 2\pi\) boundary so the interpolation never takes a 270° path; when the turn completes the camera yaw snaps back to the canonical \(dir \cdot \pi/2\) in \([0,2\pi)\).
- **Keyboard** (see `GameApp.tsx`): **W** / **S** (and **↑** / **↓**) forward/back; **A** / **D** strafe left/right; **Q** / **E** turn left/right. **F2** toggles the debug panel. **Escape**: if **settings** are open → close them; else if **NPC dialog** (or F2 NPC preview) → **close dialog**; else if **paperdoll** → close; else if **trade** → **close trade**; else **toggle settings** (opens pause menu from title, hub, or game). **During an encounter** (§7.7): **R** attempts **flee**; **F** **defends** (same actions as the **CombatIndicator** buttons on the game viewport).
- **HUD**: the bottom grid row uses a shared row height of **285 px**; **`screen === 'game'`** shows **minimap + inventory + navigation**, while **`title`** / **`hub`** show **inventory** only in the **same** center band (**120px + 1.62fr + 120px**)—outer **map** / **navigation** tracks stay **empty** so the inventory panel does **not** resize (**−5%** vs the prior **300 px** row, after the earlier **−25%** step from **400 px**); it stays the **third** grid row so widgets remain **bottom-aligned** with **`1fr` / `1fr`** rows above taking the freed space. The **minimap is north-up** and shows a **local “pixel-circle”** around the player: it **only draws** dungeon tiles that satisfy **dx² + dy² ≤ 9** (**radius 3**, Euclidean on the square grid) **and** lie **inside the current floor rectangle**—**no** placeholder squares for out-of-disk or **out-of-bounds** cells (those spots are simply empty; **HUD/game chrome** shows through). Tile positions use a **7×7** logical grid with **absolute** placement so on-map cells stay **north-up aligned**. The widget is **clipped to a circle** with **no** filled circular backdrop. Each cell is drawn at **~22.5×22.5 CSS px** (**+25%** vs the prior **18 px** cell) with a **~2.5 px** gap (**`MINIMAP_CELL_PX`** / **`MINIMAP_GAP_PX`** in **`MinimapPanel.tsx`**, **`--minimap-cell`** on the map); cells are **square** (**`border-radius: 0`**) for a chunky look. The player tile includes a **facing arrow** (4-way) so orientation is readable at a glance; tile fills use **opaque** colours with **moderate** wall/floor separation and softened feature accents so the map stays legible through HUD capture and dither without looking harsh. **Left and right** portrait **rails** (rows **1–2**) use **`minmax(0, 1fr) 120px minmax(0, 1.12fr) 120px minmax(0, 1fr)`** so outer **map**/**navigation** alignment and **1fr** portrait column widths match the layout before empty statue strips were removed. The **game** `<section>` **`grid-area`** spans the **three** center tracks (**120px + 1.12fr + 120px**), so the **3D viewport** fills that band (**no** separate statue panels; **`StatuePanel`** removed). **CHAR1/CHAR2** (left) and **CHAR3/CHAR4** (right) `<section>`s use **75%** rail width with **`justify-self: start`** / **`end`** so they line up with the **map** / **navigation** blocks below. Each rail uses **`charRail`** (**`HudLayout.module.css`**): **`flex`** row, **`gap: 0`**. **Left** columns: **`[Portrait][Strip]`**; **right**: **`[Strip][Portrait]`**. Portrait lives in **`charRailPortraitGrow`** (**`flex: 1 1 0`**, **`min-width: 0`**). The **second** flex child always gets **`charRailPushEnd`** (**`margin-left: auto`**), so **all** spare width is **between** portrait and strip—**mirrored** layout, **identical** spacing math on both sides (no per-side **`translate`** or spacer divs). Row **3** is a **`bottomRow`** wrapper: nested **`grid-template-columns: 0.75fr 120px 1.62fr 120px 0.75fr`** (same **3.12fr** sum as **`1 + 1.12 + 1`**) with **inventory** spanning the **120 + 1.62fr + 120** band. **Map** and **navigation** `<section>`s use **100%** of their **0.75fr** tracks (**0.75fr** total), matching portrait strips’ **75%** of the **1fr** rails above (**0.75fr**); they have **no** MAP/NAVIGATION title headers; the **inventory** panel likewise has **no** section title, and **inventory** grid item icons (emoji) render at **~44 CSS px**. **`HudLayout`** **`display: flex`** + **`align-items`/`justify-content: center`** centers each widget in its cell. At a reference desktop width this yields roughly the former **~518 px** outer / **~580 px** center-band proportion for rails vs merged game span. The **navigation** panel is a **3×2** image pad (sources under `Content/ui/navigation/`, mirrored to `web/public/content/ui/navigation/` for the dev server): each **cell is drawn at 50% of the bezel PNG native size** (**88.5×88.5 px** from 177×177 sources), **square** corners (no radius on the cells); on press it shows **ui_navigationbutton_pushed** for **0.5 seconds**, then returns to default. Direction glyphs are separate overlays (**arrow up/down/left/right**, **turn left/right** icons). Layout row 1: turn left, forward, turn right; row 2: strafe left, back, strafe right. The pad’s intrinsic width is **~271.5 px** (three cells + gaps); it stays **centered** inside the **navigation** panel. Tooltips still echo keyboard shortcuts (§6.4). **Note**: the on-screen HUD is composited from an offscreen **capture** DOM (`HudLayout` with `captureForPostprocess`); the **interactive** HUD is pointer-transparent (`opacity: 0`). Navigation “pushed” bezel state therefore lives in `DitheredFrameRoot` and is passed into **both** trees so `html2canvas` sees **`ui_navigationbutton_pushed.png`** on the cell you pressed.
- **Viewport**: clicking a **door** attempts a **forward** step into that cell (open normal door or try key on locked door).
- **While a move/turn animation is playing**, new step, strafe, and turn input is ignored so the player resolves one grid action at a time.
- **Audio**: a successful step or strafe plays a short **step** SFX; walking into a solid tile uses a distinct **bump** SFX (with **activity log** line + shake).
- **POI cells**: the player **cannot enter** a floor cell that holds a **POI** (props occupy their tile). Attempting to step or strafe onto that cell uses the same **bump** SFX/shake as a wall, with a distinct **activity-log** line. **Exit**, **Well**, and other POIs remain usable via **viewport click** or **dragging items onto them** from **adjacent** cells (or wherever ray pick already allows). **Regen**, **descend**, and **initial floor** set **`playerPos`** with **`pickPlayerSpawnCell`** (**`web/src/game/state/playerFloorCell.ts`**): **strict** BFS from **`gen.entrance`** over **`floor`/`door`** with **POI** cells **non-traversable** (same as **`player/step`**), **N→E→S→W** per layer; if that finds no plain **`floor`** stand tile, **relaxed** BFS may cross **POI** cells **only as search hops** (spawn is still never on a POI); then an orthogonal **nudge** that must yield plain **`floor`**, then row-major scan—**never** a **door** tile when any free **`floor`** exists (avoids door billboards and bogus **“solid stone”** bumps). **Hub → dungeon** applies **`snapViewToGrid`** and increments **`floorGeomRevision`** once so **WorldRenderer** rebuilds after the hub’s different viewport rect. **Checkpoint reload** still uses **`nearestFloorCellWithoutPoi`** to nudge **`playerPos`** off a POI tile when an older snapshot has the party on a POI cell, then re-snaps the camera.

### 7.6 NPC quests & languages
- NPC may ask for an item in text using a **gibberish language** that is **consistent** (player can learn meanings).

- **Speech strip (presentation)** is generated by **`toGibberish`** in **`web/src/game/npc/gibberish.ts`** (deterministic from **floor seed** + **`npc.id`** salt). The modal **hint** and **activity log** (including optional **English** quest shouts in combat per §7.7) carry readable guidance.
- **Deep Gnome**: several space-separated tokens, each a **safe** variant of the word **“gnome”** (plain Latin letters plus **diacritics on vowels** and optional **ñ** only — no unrelated syllables).
- **Mojibake**: **2–4** procedural “words” built from mixed **ASCII + multibyte** source text, then rendered as **UTF-8 bytes read as single-byte Latin** (classic mojibake look); **longer** segments than Zalgo.
- **Zalgo**: **2–5** procedural **short** **a–z** words with **combining diacritical marks** stacked on letters (distinct generator and RNG lane from Mojibake so the two styles do not share fake vocabulary).
- Giving the correct item can change status:
  - Hostile → Neutral
  - Neutral → Friendly
- Each NPC type has **Hated** items; giving these makes them hostile.

### 7.7 Combat (encounter mode)
- **Trigger**: attempting to step into a **hostile** NPC starts an **encounter** (the player does not move into the NPC cell). **Non-hostile** NPCs do not block movement or start encounters. **Roster**: the encounter includes that NPC plus every other **hostile** NPC in the **same procgen room** (first `floor.gen.rooms` rect containing the trigger cell). If no room contains that cell (corridor / missing gen), hostiles within **Chebyshev distance ≤ 1** of the **player** are included instead. **Swarms** are excluded from the roster while the party holds a **Swarm Queen** (same neutrality rule as outside combat).
- **Mode**: while an encounter is active, **movement is blocked**; the encounter proceeds in **turn order**.
- **NPC combat data**: per-**`NpcKind`** **Speed**, **damage**, **armor**, **resists**, **on-hit statuses**, and **`hpMax`** live in **`web/src/game/content/npcCombat.ts`** (`NPC_COMBAT_BY_KIND`). Runtime NPCs carry **`hp`** and **`hpMax`** (spawn and **CombatIndicator** use the same caps; **`hydrateFloorNpcs`** back-fills **`hpMax`** for older saves).
- **Turn order (initiative)**: each living **party member** and each encounter **NPC** gets a turn; initiative is driven primarily by **Speed** with a small **deterministic tie-break** so ordering is stable per encounter seed.
- **Player actions**: the party acts through the existing **cursor-first drag/drop** model:
  - dragging a **weapon** onto the target NPC performs an **attack** (consumes that character’s turn when the dragged weapon originates from that character’s equipment; inventory-origin weapons act as the current turn’s character).
  - **clicking** an **encounter enemy** in the 3D view on a **PC’s turn** performs an **attack** using the same weapon resolution as a convenience default: **equipped** hands (**left**, then **right**; two-hand held once), else the **first weapon** found in the **inventory grid** (if none, feedback only; drag a weapon explicitly if needed).
  - dragging a non-weapon item onto an NPC performs a **give/use** attempt and also consumes the turn.
  - **Fireshield** (see §7.3): on a **PC’s turn** in an encounter, drag **Fireshield** onto that PC’s portrait **hands** (not onto an NPC). Spends **stamina**, **consumes** the item, applies **Fire resist** vs **NPC hits** for **`shieldTurns`** of that PC’s **turns** (countdown when their initiative recurs). **Does not** equip through this path during combat.
- **Defend**: on a PC’s turn, **Defend** (HUD button or **F**) spends **stamina** and grants **temporary armor and resist** against **NPC hits** until that PC’s **next** turn begins (the buff clears when their initiative slot comes around again).
- **Flee**: **R** or the HUD button; stamina cost and a **Speed**-influenced check; failure provokes a **free enemy hit**, then **initiative advances** (the attempt consumed the turn). **Too exhausted to flee** rejects without advancing.
- **Combat HUD**: **CombatIndicator** on the game viewport lists **roster enemies** with a **compact HP bar** each (**current / hpMax**), plus current turn, a short **“Next:”** line for the following queue entry (when any), and **Flee** / **Defend** (mirrors **R** / **F**). No separate encounter modal.
- **NPC actions**: on NPC turns, the NPC automatically attacks a party member using **soft targeting**: prefer **lower current HP**, **deprioritize PCs who are Defending** (virtual HP padding in the pick), with a tiny **deterministic jitter** so ties stay stable per encounter seed.
- **NPC hit vs party**: **d20 + NPC Speed** vs **10 + target PC Speed** decides hit/miss; **crit** on natural 20. Damage uses that NPC kind’s **base damage** and **damage type**, then **PC armor** (Blunt/Pierce/Cut only), **resistances**, **Defend** bonuses, and **Fireshield** extra **Fire** resist only (`npcTakeTurn`).
- **PC damage resolution**: in **encounter mode**, a **d20 + Perception + Agility** check vs **10 + NPC Speed** decides hit/miss; **crit** on natural 20. **Weapon stamina**: if the weapon defines **`staminaCost`**, the cost is paid on **every swing** in an encounter, **hit or miss**. **Weapon damage** starts from **`ItemDef.weapon.baseDamage`**, optional **`damageStat`** (**Strength**, **Agility**, or **Intelligence**) adds **`floor(stat × 0.25)`** before run **damage %** bonus, then **NPC kind** applies **armor** (Blunt/Pierce/Cut only) and **per-type resist** (capped like PC resists). **Open-world** weapon strikes (not in an encounter) skip the to-hit roll but still apply stat bump and mitigation.
- **Activity log (combat swings):** **Encounter** PC attacks and **NPC** attacks each append **one line** that spells out the **to-hit** math in the same style as inspect checks: PC lines use **`d20+Per+Agi … = total vs 10+Spd …`**, then **miss** or **hit** (note **nat 20** when the d20 is 20) and damage; NPC lines use **`d20+Spd … vs 10+Spd …`**. Open-world weapon hits keep the shorter outcome line (no encounter roll).
- **Quest line in combat:** Hostile NPCs with **`quest.wants`** sometimes echo their quest ask in the activity log **on their turn**, **before** the swing line. Chance is **20%** per NPC turn, **deterministic** from **`floor.seed`**, **`combat.encounterId`**, NPC id, and **`combat.turnIndex`** (same keyed hash style as other combat rolls). The text is **English** **`{name}: "…bring me {item name}."`**, matching the dialog modal’s substrate before **`toGibberish`** (implemented via **`npcQuestEnglishLine`** in **`web/src/game/npc/npcQuestSpeech.ts`**).
- **Weapon procs**: `weapon.statusOnHit` can apply **timed debuffs** to **NPCs** on a successful hit (deterministic % roll).
- **World & crafting during combat**: **POI** use (viewport click or dragging an item onto a POI), **floor item pickup** (pickup action or drag onto a floor item), **starting** an inventory recipe craft, and **finishing** an in-flight craft (timer resolution) are **blocked** until the encounter ends (“Not while in combat.”). **Equip**, **unequip**, non-recipe inventory rearrangement, and **portrait** inspect / feed / most equip **stay available** during combat and do **not** consume initiative—**except** **Fireshield** dropped on the acting PC’s **hands**, which **consumes** the turn like an attack. **Portrait feed** (food, remedies, and similar **feed**-path items) during an encounter is **deliberate party triage**: it does **not** cost the acting PC’s turn, so players can stabilize vitals or statuses under pressure; balance leans on **item scarcity**, **stamina** costs on attacks, and **enemy pressure** rather than blocking feed mid-fight.
- **Item actions during combat**: **Hive** floor drops, **Swarm Basket** capture, **Captured Swarm** release, and **Swarm Queen** “calm swarm” updates are **blocked** during an encounter (“Not while in combat.”); use normal **weapon** / **give** / **flee** / **defend** flows instead.
- **End conditions**: encounter ends when all encounter NPCs are dead (**victory**, grants **XP** scaled by **number of enemies that were in the encounter roster**) or when all party HP ≤ 0 (**party wipe**, death screen).

### 7.8 Status effects
- Status effects apply to PCs and NPCs; positive/negative/neutral.
- Some decay over time; some persist until cured by an item. **`time/tick`** prunes expired entries for **both** party characters and **floor NPCs**.
- Baseline content set includes **Poisoned, Blessed, Sick**; additional ids (e.g. **Bleeding**, **Burning**, **Frightened**) appear from items, combat procs, and **room hazards**.
- **Expansion**: **NanoTagged** (encounter: **−1** to PC attack to-hit total), **Spored** and **Parasitized** (encounter: each **−1** to PC “AC” vs NPC hits, i.e. `10 + Speed` defense term). **Salt** / **Moss** / **Antitoxin vial** and some feeds clear or mitigate these (see `interactions.ts`).

## 8) Dungeon & world generation
### 8.1 Taxonomy (tags drive content)
- **Floor types**: `Dungeon`, `Cave`, `Ruins`, plus **reskin types** `Jungle`, `LivingBio`, `Bunker`, `Catacombs`, `Golem`, `Palace`. Each reskin maps to a **layout profile** that reuses one of the three base realizers (**Cave**, **Ruins**, or **Dungeon** geometry): Jungle/LivingBio → Cave; Bunker/Golem → Dungeon; Catacombs/Palace → Ruins. Bespoke generators per fantasy floor are deferred; differentiation is **theme** (`floor.gen.theme`), **env texture triples** (each **`FloorType`** has its own **`/content/*.png`** filenames in `dungeonEnvTextures.ts`; reskin triples are **binary copies** of the matching base art until bespoke environment PNGs ship), **music bias**, and **spawn tables**.
- **Run progression (segments & camps)**: Dungeon floors are grouped into **segments** of **`render.campEveryFloors`** floors (default **10**, clamped **1–99** in tuning). Each segment uses one **floor type** from the fixed rotation: Cave → Ruins → Dungeon → Jungle → Catacombs → Palace → Bunker → LivingBio → Golem (then repeats). **`floor.floorIndex`** still counts individual dungeon floors (0-based). **Procgen difficulty** (`0` / `1` / `2`) follows **segment index**: segments **0–1** → easy, **2–4** → normal, **5+** → hard. After completing the last floor of a segment (**e.g.** exiting floor **9** when **N = 10**), the game opens a **camp hub** (`ui.hubKind === 'camp'`): same village + tavern flow as the **starting hub**, but **camp tavern/bartender** art under **`/content/camp_tavern_*.png`** / **`camp_bartender_*.png`**, **`CAMP_INNKEEPER_TRADE`**, and a **themed camp village** backdrop: **`/content/camp_<skin>.png`** where **skin** is **`layoutProfile`** applied to the **`FloorType`** of the **segment the player just finished**—**`floorTypeForFloorIndex(max(0, floor.floorIndex - 1), clamped campEveryFloors)`** (same schedule as **`runFloorSchedule.ts`**; while at camp, **`floor.floorIndex`** is already the **next** dungeon floor, so **index - 1** is the last floor of the completed segment): **`Cave`** profile (**`Cave`**, **`Jungle`**, **`LivingBio`**) → **`camp_cave`**; **`Dungeon`** profile (**`Dungeon`**, **`Bunker`**, **`Golem`**) → **`camp_dungeon`**; **`Ruins`** profile (**`Ruins`**, **`Catacombs`**, **`Palace`**) → **`camp_village`**. On the **interactive** village view, pointer hover over **Tavern** or **Cave** (dungeon exit) shows **`/content/camp_<skin>_tavern_hover.png`** or **`_dungeon_hover.png`** respectively; the **starting hub** uses **`village_tavern_hover.png`** / **`village_dungeon_hover.png`** over **`village.png`**. **Capture** hub rendering omits hover overlays. The **next** floor is **pre-generated** before the camp screen; play resumes when the player leaves the camp via the cave entrance. The **starting village** after **New run** is not a camp (`hubKind` unset).
- **Floor properties**: Infested, Cursed, Destroyed, Overgrown, None (not shown)
- **Room function**: Passage, Habitat, Workshop, Communal, Storage
- **Room properties**: Burning, Flooded, Infected, SporeMist, NanoHaze, Unstable, Haunted, RoyalMiasma (on-enter effects and HUD telegraph; some skew spawns)
- **Room status**: Overgrown, Destroyed, Collapsed
- **Doors/keys**: passages can be blocked by door types requiring appropriate keys; procgen records **`keyDefId`** per locked door in **`floor.gen.doors`** (gameplay opens a door only if the party holds a matching item, e.g. **IronKey** / **BrassKey**). Grid tiles include plain **`door`** / **`lockedDoor`** plus variant **`doorOctopus`** / **`lockedDoorOctopus`** (same walk/open rules; procgen rolls **`lockedDoorOctopus`** on a subset of locks, biased upward on **Infested** floors). In the 3D view, wooden doors use **`door_closed.png`**; octopus doors use **`door_octopus_closed.png`**. When a door opens, the tile becomes **`floor`** immediately and the renderer shows a short-lived FX: **`door_open.png`** (**420 ms**) for wooden doors, or three frames **`door_octopus_opening_01..03.png`** cycled at **~280 ms** per frame (**900 ms** total) for octopus doors.
- **Difficulty** (`0` = easy, `1` = normal default, `2` = hard): **`GameState.floor.difficulty`** and **`FloorGenInput.difficulty`**. Tunes how many **validation rerolls** run (**8** / **6** / **5**), minimum entrance→exit shortest-path length before placing locks (**8** / **4** / **4** cells for any lock; two-lock branch when path length ≥ **11** at normal and ≥ **9** at hard — easy keeps two-lock **disabled**), and scales the **`layoutScore`** lock+loop bonus when choosing among valid rerolls (easier prefers layouts with less lock+loop weight; harder the opposite). Echoed as **`floor.gen.meta.difficulty`**.

### 8.2 Starting algorithm (prototype baseline)
**Floor type** (full **`FloorType`** union; see §8.1) selects a **layout profile** and thus a **geometry realizer**, then shared post-passes and population:

1. **Layout (per `layoutProfile(floorType)` → `Dungeon` | `Cave` | `Ruins`)**
   - **Dungeon profile**: BSP partition (min leaf ~6×6; depth ~6), room carve (~45–70% of leaf), sibling **L-corridor** stitch, then a guarded **door-frame** pass that introduces a small number of 1-tile **throats** in straight corridors so rooms read as connected by doorways (locks prefer these frames).
   - **Cave profile**: seeded worm carve + occasional widen, then a single **GenRoom** from the floor bounding box of carved cells (fallback box if empty).
   - **Ruins profile**: ~5×5 macro-cell stamps with random doorways between cells; stamped chambers are deterministically **clustered into macro rooms** (bounded count) for tagging/districts/population.
2. Connectivity repair (deterministic), then CA **carve-only** smoothing (one pass).
3. Exit selection (exit = **farthest** reachable floor cell from the entrance by BFS).
4. **Derived connector rooms**: a small number of **junction/connector** rooms are derived from corridor junction clusters (bounded) and treated as **Passage** anchors for tags/spawn bias. Max derived rooms is higher on the **Dungeon** layout profile than on Cave/Ruins.
5. **Districts**: seeded Voronoi on room centers → **`district`** tags (`NorthWing`, `SouthWing`, …) for spawn bias.
6. **Room tagging**: quota-aware function assignment (e.g. try to keep at least one **Storage** among tiny rooms) plus rolls from **`floorProperties`** (Infested/Cursed/Destroyed/Overgrown). Derived connector rooms keep their fixed `Passage` function.
6. **Tag constraints** (post-quota): **Storage** prefers a **dead-end** room center (swap with a tiny **Passage** room when needed); on **Cursed** floors, **Flooded** expands to **edge-adjacent** rooms with a fixed probability so hazards form a small cluster.
7. Population pass:
   - POIs (**Well** / **Bed** / **Chest** / **Barrel** / **Crate** / optional **Shrine** / optional **CrackedWall** / **Exit**) from entrance–exit heuristics + storage-room bias for chest/barrel/crate (Exit spawns at/near `gen.exit`). Each blocking PoI candidate is accepted only if **progression stays valid**: from the party spawn cell implied by **`pickPlayerSpawnCell`** at **`entrance`** (see §6.4), BFS over **`isWalkable`** tiles treats every **placed PoI cell** and the **exit** cell as non-traversable and must still reach **at least one orthogonal neighbor of `gen.exit`** (so the Exit POI remains usable from beside the stairs).
   - NPCs and floor items from **`spawnTables.ts`** (room function, **`roomStatus`**, district, **`floorType`**, **`floorProperties`**, and whether the room center lies on a shortest entrance→exit path before locks) so balancing can stay data-shaped without touching placement loops
8. **Lock/key pass** on the entrance→exit **shortest path**: the path BFS **shuffles orthogonal neighbor order** with the **locks** RNG stream so ties among equal-length routes vary by seed (more chances to hit a **separating** cell). Place **ordered** separating **`lockedDoor`** or **`lockedDoorOctopus`** when possible (same gameplay; art variant is seeded). Long paths may get **two** gates (**A** + **IronKey**, **B** + **BrassKey**); shorter paths use **one** (**A** + **IronKey**); otherwise skip locks. Keys carry **`forLockId`** in **`floor.gen.floorItems`** for validation/debug.
9. **`missionGraph`**: nodes (entrance, exit, POIs, locks, keys); **path** edges chain POIs by BFS distance from entrance, then the lock sequence, then exit; **`shortcut`** edge (entrance→exit) is present when multiple shortest-length routes exist; **`hasAlternateEntranceExitRoute`** mirrors that flag for tools.
10. **Theme**: **`floor.gen.theme`** (`id` + floor/wall/ceiling **color** multipliers) from a dedicated RNG stream; **`WorldRenderer`** tints **Lambert** dungeon materials (textures unchanged).

**First load**: `makeInitialState` runs the same `generateDungeon` pipeline as `floor/regen` (**31×31**), with **`floor.seed`** (random on cold start), **`floor.floorIndex`** (0 on first floor), **`floor.floorType`** (default **Dungeon**), **`floor.floorProperties`** (default empty), and **`floor.difficulty`** (default **1**). The canonical bundle is **`floor.gen`** (`genVersion` **5**); **`playerPos`** is **`pickPlayerSpawnCell(..., gen.entrance, gen.pois)`** (§6.4), not necessarily the raw **`gen.entrance`** tile when that tile is a door or blocked by a POI.

**Meta-progression**: floors are advanced by interacting with an **Exit POI** placed at/near `gen.exit`. Descending increments **`floorIndex`**, cycles **`floorType`** through the full set (base three then reskins, in code order), and generates a **new random `floor.seed`** for the next floor.
Default floor dimensions are **31×31**; `floor/regen` keeps the current **`floor.w` / `floor.h`** and reapplies **`floorType` / `floorProperties` / `floorIndex` / `difficulty`** from state unless you add UI to change them.

### 8.3 Determinism requirement
Generation should be **seeded** and phase-stable (changes to one phase shouldn’t ripple unpredictably into others) to support future host-synced multiplayer.

Validation is part of generation: the generator runs **bounded deterministic rerolls** (attempt count depends on **`difficulty`**; see §8.1) when **ordered** lock/key constraints fail, when **safety POIs** violate reachability caps (**Well** within **3** BFS steps of **`entrance`**, **Bed** within **48**), when **PoI layout** would block **spawn → exit-adjacent** reachability (same rule as placement; applies even when there are **no** procgen locks), or when the floor has **any** procgen **locks** but the lock-aware entrance→exit shortest-path **lattice** is too narrow (**`latticeCells` must exceed `shortestLen + 1`**). When the floor has procgen **locks**, those checks (lock probes, key-order reachability, lattice width, and Well/Bed distances) use BFS where **`lockedDoor` / `lockedDoorOctopus`** are **not** traversable; **`layoutScore`** and earlier layout metrics still use the generic walkability that treats all door tiles as open for graph purposes. Among attempts that pass validation, the run picks the one with the highest **soft `layoutScore`** (reachable floor mass, dead-end penalty, path length bonus, **junction** count for 3+ neighbor floor cells, bonus when that lattice is wide, extra bonus when locks are present and the lattice is wide — **lock+loop** term scaled by **`difficulty`**); if none validate, it returns the **last** attempt (still full geometry) like before. Output always has a non-empty **`rooms`** list except catastrophic fallback (single carved room).

**Meta** records **`inputSeed`**, **`attemptSeed`**, **`attempt`**, **`layoutScore`**, **`difficulty`**, and **phase streams** (`layout`, `tags`, `population`, `locks`, `districts`, `score`, **`theme`**, **`mission`** for future / reserved **`planMissionBeforeGeometry`**).

### 8.4 Reference pipeline (expanded)
See `Dungeon_generation_plan_summary.md` for an expanded phased pipeline (mission graph → lock correctness → geometry → districts → tagging solve → population → validation → debug overlays). The **geometry-first** pipeline is still what ships; **`web/src/procgen/missionFirst.ts`** defines **`PlannedMission`** / **`planMissionBeforeGeometry`** (returns `null` today), the **embedding contract**, and an in-file **Track B** specification (abstract plan vs realized **`missionGraph`**, per-**`floorType`** embed strategy, future **`genVersion`** when embedding ships).

**Debuggability**: the F2 Debug panel includes a **Procgen** readout (**difficulty**, **floorType**, **floorIndex**, **genVersion**, **`streams.mission`**, **attempt**, **layoutScore**, **theme** id, **missionGraph** size and alternate-route hint), **Cycle type** (advances through all **`FloorType`** values for the next regen), **Cycle difficulty** (easy→normal→hard for the next regen), **Proc overlay** (cycles **off** / **districts** / **roomTags** / **mission** tints on the 3D floor from **`floor.gen`**), and **Dump `floor.gen`** to download the canonical JSON.

## 9) Points of interest (POIs)
Static interactables. POIs **do not** auto-trigger on **walking past** them, but their grid cell **blocks player occupancy**: you **cannot stand on the same tile** as a POI (see §6.4). Interact via **click** or by **dragging items onto them** (typically from beside the object in the 3D view).

- **Placement invariant**: there is **at most one POI per grid cell**. If procgen produces multiple POIs for the same cell, the game resolves it deterministically (canonical IDs win, then kind priority) so runs remain stable and interactions remain unambiguous. **Progression invariant**: no PoI may sit on a tile that would make it **impossible** to walk from the **entrance spawn** (see §6.4) to **beside the Exit** while treating all PoI tiles and the **exit stair** tile as blocked; procgen skips or relocates candidates (optional **Shrine** / **CrackedWall** may be omitted if no valid tile exists).
  - **3D view**: POIs render as **sprite billboards** (same texture pipeline as NPC billboards: nearest filtering, transparent PNG). **Well (filled)** uses **`npc_well.png`** plus extra **non-pickable** billboards: **`npc_well_glow.png`** (slightly larger halo) and a small **sparkle** layer cycling **`npc_well_sparkle_1..3.png`** (~280 ms per frame). **Well (drained)** uses **`npc_well_drained.png`** only (no glow/sparkle). **Chest**, **Barrel**, and **Crate** use closed/open PNG pairs (`chest_*`, `barrel_*`, `crate_*`). **Exit** uses **`stairs_down.png`** at **2×** the default POI billboard scale (other POIs use a uniform base height). **Shrine** uses **`shrine_gnome.png`** until the first interaction, then **`shrine_gnome_off.png`**. **Bed** and **CrackedWall** use **`/content/poi_bed.png`** and **`/content/poi_cracked_wall.png`** (each a copy of **`poi_placeholder.png`**) until dedicated POI art exists. Billboards use a **bottom-centered pivot** (`Sprite.center`) so the texture’s lower edge anchors to the floor; vertical placement is **`poiFootLift`** (F2: **POI above ground**, independent of **`npcFootLift`**). Per-kind **`render.poiGroundY_*`** (F2 for **every** **`PoiKind`**) shifts where the “ground contact” sits within the PNG. Brightness uses **`render.poiSpriteBoost`** (F2; default tuned brighter than 1.0).

Initial POIs:
- **Well**: **checkpoint save point** (clear notification); used to fill Waterbag; a successful **Waterbag (Empty)** use on this well sets the POI to **drained** (visual swap + VFX off); **checkpoint save still works when drained**. Checkpoints are reloadable from the **death screen** and the **title** screen.
- **Chest**: opens (sprite change) and drops random item
- **Barrel**: opens (sprite change) and drops random item; the **open** barrel still **blocks** its cell until a **further** use **clears** it (POI removed, no rubble sprite).
- **Crate**: opens (sprite change) and drops random item; the **open** crate still **blocks** its cell until a **further** use **clears** it (POI removed, no rubble sprite).
- **Bed**: **one use per floor**: restores HP and Stamina (same heal amounts as before); further clicks log **Already used.** (POI stays on the tile; `opened` tracks use)
- **Shrine**: **use** (click or item drop without a matching **`useOnPoi.Shrine`** hook) removes **Cursed** from the party and marks the shrine used (sprite swap). Dragging an item whose def includes **`useOnPoi.Shrine`** with **`consumeOffering`** / **`blessPartyMs`** runs the **offering** path instead: consumes that item, applies **Blessed** per content, logs a toast; does not replace the standard “purge **Cursed**” use (player can still use the shrine normally afterward if not yet opened).
- **Exit**: descends to the next floor (meta-progression)

## 10) Audio
Three audio layers run simultaneously via the Web Audio API:

- **Background music** (`MusicPlayer` / `MusicLayer`): loads an audio file, loops it continuously, and applies a tunable master volume (`masterMusic`). Starts on first user interaction (browser autoplay policy). Currently plays `Assets/sounds/theme.mp3` (served from `public/sounds/`).
- **Spatial ambient** (`SpatialAudio` / `SpatialAudioLayer`): NPCs and POIs emit continuous synthesized tones audible from a configurable number of cells away. Attenuation uses both **lowpass** and **volume** with distance (tunable in F2).
- **Procedural SFX** (`SfxEngine` / `FeedbackLayer`): discrete one-shot sounds generated from scratch using oscillators, noise, envelopes, and filters. Triggered via `ui.sfxQueue`. Kinds: `ui`, `hit`, **`swing`** (weapon swing / to-hit miss, distinct from UI **`reject`**), `reject`, `pickup`, `munch`, `step`, `bump`.

Volume controls: `masterMusic` (music layer) and `masterSfx` (SFX + spatial) are both in `AudioTuning` and adjustable in F2.

## 11) Graphics & rendering spec
- Three.js WebGL with `MeshLambertMaterial` on dungeon geometry (diffuse + emissive; no PBR metalness/roughness).
- **Dungeon albedo**: floor / wall / ceiling voxels use tiled PNGs from `Content/` (stable **`/content/…`** URLs; Vite serves the repo **`Content/`** tree in dev and copies it into **`dist/content/`** on build). Each **`FloorType`** has a dedicated triple in **`dungeonEnvTextures.ts`**: **`Dungeon`** — **`dungeon_floor.png`**, **`dungon_wall.png`**, **`dungon_ceiling.png`**; **`Cave`** — **`cave_*`**; **`Ruins`** — **`ruins_*`**; **`Jungle`** — **`jungle_*`** (copy of **`cave_*`** for now); **`LivingBio`** — **`livingbio_*`** (copy of **`cave_*`**); **`Bunker`** / **`Golem`** — **`bunker_*`** / **`golem_*`** (copies of the **`Dungeon`** triple); **`Catacombs`** / **`Palace`** — **`catacombs_*`** / **`palace_*`** (copies of **`ruins_*`**).
- **Overgrown rooms (selective env swap)**: procgen rooms with `roomStatus: Overgrown` render their **floor + ceiling** using **`overgrown_floor.png`** and **`overgrown_ceiling.png`** (served at `/content/overgrown_*.png`). **Walls** bordering an overgrown walkable tile use **`overgrown_wall.png`** (4-neighbor adjacency) so room boundaries read clearly without changing the core geometry pipeline.
- Anti-aliasing **off**; pixel ratio capped at **1.5×** (sharp edges, dither-friendly).
- Lighting:
  - warm **PointLight** on camera (“lantern”): **intensity** and **distance** use a **per-mode** F2 pair from **`render`** — **`bareLight*`**, **`heldTorch*`**, **`equippedLantern*`**, **`headlamp*`**, **`glowbug*`** — selected by the **strongest** party-wide equipped **`playerLight`** (see §7.3). **POI** torches do **not** use those keys.
  - optional forward **SpotLight** “beam” aligned to camera direction (tunable; often off via intensity scale). When effective beam intensity is zero, the spot is **not visible** and does **not** cast shadows (no wasted shadow pass).
  - POI **torch** PointLights: **`torchIntensity` / `torchDistance`** only; up to **`torchPoiLightMax`** (F2), choosing the **nearest** POIs to the player by Manhattan grid distance (0 disables torches). Shipped **`debug-settings.json`** uses a short **`torchDistance`** (**2** world units) as the authored baseline.
  - torch flicker via per-torch sine intensity
  - **Shadow mapping** is enabled only while at least one lantern light has shadows on: **point** shadows (expensive cube map) and **beam** shadows are separate F2 toggles (`shadowLanternPoint`, `shadowLanternBeam`). **Shadow map size** and **filter** (basic / PCF / PCF soft) are tunable in F2; shipped **`debug-settings.json`** sets **`shadowFilter`** to **Basic** (**0**). Dungeon geometry casts/receives where applicable.
  - low emissive lift on dungeon materials; the lantern is the primary visibility driver
  - `FogExp2` is **optional** for depth falloff; **disabled by default** (can be enabled in F2 debug); the renderer reuses one fog instance instead of allocating every frame
- Post-process: ordered dithering as shader pass (EffectComposer), controls:
  - Strength
  - Colour preserve
  - Pixel size
  - Levels
  - Post-dither levels (classic lift/gain/gamma; affects HUD + 3D equally)
  - Matrix (Bayer 2×2/4×4/8×8)
  - Palette (Dungeon warm, Cold crypt, Monochrome, Sepia print, No palette snap)
  - **Warm palette mix** (F2): when palette is **Dungeon warm** (0), blends between **quantised dither only** (like no snap) and **full warm five-colour snap**; other palette indices ignore this control
- **Room-property telegraph (compositor)**: when the player is standing in a room tagged with a **room property** (`Burning` / `Flooded` / `Infected` / `SporeMist` / `NanoHaze` / `Unstable` / `Haunted` / `RoyalMiasma`), the compositor applies a **luma-preserving color tint override** (blend toward a hazard RGB at uniform strength over the full 3D viewport—**no vignette**) so the hazard read clearly **overrides** the floor theme cast while keeping local brightness structure. Hazard telegraph **intensity pulses slowly** (~**3.8 s** sine) by scaling the **graded tint only** (fixed hazard RGB ratios), not the blend weight—so the cast stays **one hue** and does not swing toward unrelated scene colors. **`Burning`** reads as **hot red**; **`ruins_umber`** floor theme lighting uses **brown umber** intent (not fire red) so the two do not collide. Room lookup matches **`roomForCell`** (containment only; no nearest-room fallback). Tuning/preview can be forced via the debug telegraph override.
- **Room-hazard floor decals (3D)**: `WorldRenderer` draws **non-interactive** sprite billboards on **`floor`** tiles inside procgen rooms whose **`tags.roomProperties`** matches the hazard. Each property has a **dedicated** decal path in **`ROOM_HAZARD_SPRITE_SRC`** (`hazardDefs.ts`): **`Burning`** / **`Flooded`** / **`Infected`** use **`hazard_fire.png`**, **`hazard_water.png`**, **`hazard_poison.png`**; **`SporeMist`**, **`NanoHaze`**, **`Unstable`**, **`Haunted`**, **`RoyalMiasma`** use **`hazard_spore_mist.png`**, **`hazard_nano_haze.png`**, **`hazard_unstable.png`**, **`hazard_haunted.png`**, **`hazard_royal_miasma.png`** (currently **binary copies** of the baseline three hazard PNGs until bespoke art ships). Placement is **sparse and deterministic** (~**40%** of eligible cells, keyed by **`floor.seed`** + room id + cell), and cells occupied by a **POI**, **NPC**, or **floor item** are skipped. Decals sit low on the floor (ground-stain height), use the same **theme tint** as other billboards, and rebuild with **`floorGeomRevision`**. **Gameplay** runs `applyRoomHazardOnEnter` **once per visit** to a given procgen room (tracked by `floor.roomHazardAppliedForRoomId`): on the **first step** into a tagged room **or** when the party is **placed** in that room (initial floor, **new run**, **regen**, **descend**, **Exit** POI). It applies statuses, chip damage, log lines, SFX, and shake per hazard kind—**not** on every tile walked inside the same room.
- **Frame presentation pipeline**:
  - The 3D world is rendered offscreen into a **render target** sized to match the on-screen **game viewport rect** (the HUD “game” panel), not necessarily the full window.
  - The HUD exists as HTML/CSS twice:
    - an **interactive HUD** (`opacity: 0` over the final canvas but above the WebGL presenter, handles pointer input)
    - a **capture HUD** (offscreen, non-interactive) that is rasterized into a canvas texture. The capture HUD does not mount **`GameViewport`** (the 3D view is not part of **`html2canvas`**); the **game** grid cell uses an **invisible same-sized shell** with the same **`gameViewportRef`** attachment pattern for compositor/game-rect parity—**except** when **`ui.screen === 'hub'`**, where **`HubViewport`** renders the **2D hub image + hotspot outlines** into both **interactive** and **capture** trees so **`html2canvas`** picks up **opaque** pixels in that cell and the compositor **masks** the 3D scene there (same mechanism as other HUD-over-scene blending). The **dithered NPC dialog** mounts inside that **game** cell (**`npcCaptureLayer`**: **stretch**ed overlay; inner **`captureRoot`** fills the cell and pins **top panel + bottom speech strip**) rather than as a sibling of the whole **`HudLayout`**. The **death** modal (**real** or **debug preview**) uses the same **game** cell overlay path (**priority over NPC** in capture when both would apply): dim + panel are **centered** in the game widget. **Title** and **paperdoll** mount as **full-bleed capture** overlays on the entire HUD grid (**`captureFullHudOverlay`** → **`fullHudCaptureLayer`**). **Interactive** **title**, **death**, and **paperdoll** hit targets mount **inside** the **`interactiveHud`** subtree in **`HudLayout`** (**`fullHudInteractiveLayer`** for title/paperdoll; **`gameCellModalHitLayer`** for death—same padded box as capture).
  - **Modal interaction**: an **offscreen capture** variant feeds the **dithered** HUD texture. **Interactive** **title**, **death**, and **paperdoll** mount **inside** **`HudLayout`** with **stage-local** **`position: absolute`** layers (same **`transform: scale`** context as inventory), matching **trade** (**ADR-0263**). **`NpcDialogModal` (interactive)** still **`createPortal`s** to **`document.body`** at **`z-index: 10100`** with **`NpcDialog`**-style invisible hit layers for **fixed** viewport anchoring. **`GamePopup`’s** **`modalPortalHitRoot`** is unused by those three modals. **`TradeModal` (interactive)** mounts in **`HudLayout`**’s **game** cell; **capture** trade still comes from **`captureFullHudOverlay`**. **`DitheredFrameRoot`’s `stageModalLayer`** mounts **`NpcDialogModal`** only; **`pointer-events: none`** because the dialog hit tree is portaled to **`document.body`**. **`CursorProvider`** (after each primary-button **`pointerup`**) runs a **microtask** that **`elementFromPoint`s** for **`[data-modal-chrome-hit]`** **`button`s**: if the **release coordinates** sit on that chrome but the **native `pointerup` target** was **not** inside that button (e.g. **`setPointerCapture`** on a drag source), it **synthesizes `click()`** so **Trade** / **Close** and other marked modal buttons stay reliable (**ADR-0280**).
  - Because the capture HUD is rasterized by **`html2canvas`**, prefer conservative CSS for critical visuals (avoid relying on `object-fit` for aspect-correct sprite presentation; use intrinsic sizing patterns instead).
  - **HUD shell art**: a single transparent **`ui_hud_background.png`** (`Content/ui/hud/`, mirrored to `web/public/content/ui/hud/` for some workflows) is drawn full-bleed behind the HUD grid in `HudLayout` (CSS `::before`, `background-size: contain`, centered). **Per-character equipment** uses a separate strip asset **`ui_hud_charequip_background.png`** in the portrait rails (also under `Content/ui/hud/`); it is part of the same DOM HUD capture and is dithered with the rest of the UI. Existing widgets stay in the current CSS grid; opaque “glass” panel styling is removed so content sits on the artwork (fine-tune slot positions vs the art next).
  - Presenter sizing is derived from the **viewport CSS size** (prefer `visualViewport.width/height`, else `documentElement.clientWidth/clientHeight`) rather than measuring the presenter canvas element, to avoid resize feedback loops caused by renderers writing inline canvas CSS sizes.
  - A presenter compositor shader places the scene render target into the frame **only inside** the game viewport rect and overlays the captured HUD everywhere else (and over the scene where UI alpha exists).
  - The final composite then runs through the ordered-dither post-process so the **same dithering/pixelation** applies to both 3D and HUD.

## 13) Content pipeline (placeholders-first)
Canonical generic stand-in art is **`Content/poi_placeholder.png`**. Each NPC kind or POI without final art keeps a **dedicated filename** under `Content/` (e.g. **`npc_swarm.png`**, **`poi_bed.png`**) as a **binary copy** of **`poi_placeholder.png`** until real assets replace them; code points **`NPC_SPRITE_SRC`** / **`POI_SPRITE_SRC`** at those paths.

The **canonical** art folder is `Content/`. The web client loads PNGs via stable `'/content/...'` URLs (no cache-busting query params). Vite serves the repo-level `Content/` at `/content/*` in dev and copies `Content/` into `dist/content/` on build.

Asset types:
- Item PNG (transparent)
- Tile texture (dungeon environment: per-`FloorType` triples in `dungeonEnvTextures.ts` — base **`Dungeon`** / **`Cave`** / **`Ruins`** plus reskin **`jungle_*`**, **`livingbio_*`**, **`bunker_*`**, **`golem_*`**, **`catacombs_*`**, **`palace_*`** → `/content/…`)
- NPC sprite PNG (transparent)
- Portrait
- Mouth PNG (transparent)
- Eyes PNG (transparent)
- HUD shell PNG (transparent): `Content/ui/hud/ui_hud_background.png`
- HUD character-equip strip PNG (transparent): `Content/ui/hud/ui_hud_charequip_background.png`

### 13.1 Asset serving & caching (web)
- Runtime-served assets (portraits, NPC sprites, UI art, cursor sprites) live under repo-level `Content/` and are referenced via **stable** `/content/...` URLs.
- The game relies on **browser HTTP caching** for these static URLs; avoid cache-busting query params for runtime art.
- Imperative image loads (e.g., measuring portrait aspect ratio or preloading sprite layers) go through a small shared **in-app image cache** (`web/src/ui/assets/imageCache.ts`) to dedupe concurrent loads and reduce repeated decodes/revalidation requests.

### 13.2 Procgen content coverage (data discipline)
- **Seeded item defs** live in `web/src/game/content/items.ts` (`DEFAULT_ITEMS`). **Not every item is meant to appear from dungeon generation**: some are **craft-only**, **Well transforms**, or **quest-chain** items. **Headwear** (`WoolCap`, `HerbCirclet`, `SporeCap`) is both **craftable** and can appear from **floor spawns** and **POI container** loot when wired in `spawnTables` / `poiLootTables`.
- **Procgen** places floor items via `web/src/procgen/spawnTables.ts` (`pickFloorItemDefFromTable`); **locks** add keys via `web/src/procgen/locks.ts`. **Chest / barrel / crate** opening uses deterministic tables in `web/src/game/content/poiLootTables.ts` (rolled in `web/src/game/state/poi.ts`). **Neutral NPC quests** reference a small want/hated item set from `web/src/procgen/population.ts`.
- **Policy**: when adding a new `ItemDefId`, either wire it into one of those pipelines or add it to **`ITEM_DEF_IDS_INTENTIONALLY_NON_PROCGEN`** in `web/src/tools/procgenContentNonSpawnAllowlist.ts` so the audit stays honest.
- **Audit CLI** (from `web/`): `npm run audit:procgen-content` — compares seeded items to the union of procgen + POI loot + quest references; exits non-zero if a seeded item is neither used nor allowlisted, or if listed ids are missing from `DEFAULT_ITEMS`. **Status effects** are not placed by procgen; they are applied in gameplay (combat, feed, hazards). Items may reference statuses via `feed.statusChances` (e.g. **Blessed** / **Sick** on mushrooms).
