## 1) High concept
**Elfenstein** is a web-based, grid-based dungeon crawler with a first-person **2.5D** view. Visuals combine **3D dungeon geometry** with **camera-facing textured billboards** (**Mesh** + **Lambert**) for characters, POIs, doors, and floor items so they share the same **point-light** read as walls/floors—**one** primary equipped **`playerLight`** **PointLight** **sums** every party member’s lit **hands/head** gear (**torch** / **lantern** / **headlamp** / **glowbug**; **headlamp** has the highest per-slot default intensity, **glowbug** scales with bugs per row) for **intensity**, clamped by **`render.equippedLightIntensityCap`** after **global × flicker**; **distance** combines as **max(d) + √(Σ_rest d²)** over those sources (each **d** from the matching **`render.*Distance`**, **glowbug** rows × **√jar bugs** per row)—**two** lit slots add like a **sum**; it sits in **world** space from **player** + **yaw** unless **any** **headlamp** is equipped (then **camera**-child), plus capped lights on **dropped** glowing items and **nearest** **POI** accents; **NPC**/**POI** debug boosts scale **Lambert** **`color`** and **emissive** so changes read under lantern/torch lighting; **Elder** uses **`npcSpriteBoost`** on procedural **RGB**. **Well** water **glow** is an **unlit** **`Sprite`** with **additive** blending (emissive read, not Lambert-lit); **sparkle** frames stay a normal transparent **sprite** on top — both multiply **`SpriteMaterial.color`** by **`render.poiSpriteBoost`** like the well’s **Lambert** base. The look prioritizes a crunchy dithered/print feel and a highly contextual, low-clutter UX.

## 2) Product goals
- **Modular + extensible**: content and stats are compartmentalized and easy to expand (items, NPCs, species, classes, floor types, etc.).
- **Multiplayer-ready later**: generation and state should be compatible with a future model where the **host player** runs the authoritative database and **1–3 clients** connect.
- **Contextual UX over menus**: few actions are available at any moment; actions branch into contextual interactions/modals.

## 4) Core player experience (pillars)
- **Touch what you see**: the hand cursor is the primary verb (click, drag, drop).
- **Party as a single organism**: shared inventory with a **fixed** slot grid (see §7.2).
- **Discovery through interaction**: crafting by experimentation; NPC language as a memorization puzzle.
- **Atmosphere via light + sound**: darkness is default; proximity audio supports exploration.

## 5) Core loop
Explore → find POIs/NPCs/items → manage inventory/craft → resolve encounters → recover at POIs → push deeper (keys/doors) → repeat.

### 5.1 Title, hub, and entering the dungeon

## 6) Controls & interaction model
### 6.1 Cursor (hand) states
- **OS cursor**: hidden globally (`cursor: none` on `document.body`); interactive HUD widgets also set **`cursor: none`** so controls (e.g. navigation buttons) never revert to the system pointer—only the **`CursorLayer`** hand sprite is visible. **Exception**: while the **Bobr** starting cutscene is active (**`ui.bobrIntroUntilMs`** and **`nowMs`** before that deadline), **`CursorLayer`** renders nothing and **`document.body.style.cursor`** is cleared so the **system** cursor is visible—the custom hand sits at **`z-index: 9999`** and would otherwise cover the portaled intro (**ADR-0374**). **`CursorLayer`** is a **sibling** of **`FixedStageViewport`** (not inside the scaled **`.stage`**): **`position: fixed`** with **`clientX`/`clientY`** must use the **viewport** containing block; a transformed ancestor would offset the hand from the real pointer. **`CursorLayer`** uses **`z-index: 9999`** above most UI; **all descendants use `pointer-events: none`** so the hand/ghost/affordance never **capture** clicks meant for DOM below. The **NPC dialog** portaled hit layer uses **`z-index: 10100`** (**above** the cursor layer) so **Close** / drag targets stay reliable; it stays **visually empty** (**`opacity: 0`**) so the hand remains visible through it.
- **Default**: `Hand_Point`
- **Combat attack (3D NPC hover)**: **`hand_attack_01.png` / `hand_attack_02.png`**, **~280 ms** flip; encounter active, acting PC’s turn, pointer over a hostile in-roster NPC, weapon resolves (`resolveWeaponItemIdForPcTurn`); dragging a **weapon** still shows **⚔ Attack**.
- **Hub tavern (open trade hit)**: **`hand_trade_01.png` / `hand_trade_02.png`**, same cadence. **`screen: hub`**, **`hubScene: tavern`**, **`innkeeperTrade`** has **`data-drop-kind="hubInnkeeperTrade"`**; **`HubViewport`** publishes the hotspot’s screen rect to **`hubTavernTradeHoverRectRef`**, and **`CursorProvider`’s `applyPointerMove`** sets **`virtualHover`** to **`hubInnkeeperTrade`** when the pointer lies inside that rect (**before** `elementsFromPoint`), so **`hoverTarget`** matches under the **opacity-0** interactive HUD. Drop on that region rejects (**`ui/sfx` `reject`**).
- **Holding/dragging**: `Hand_Hold` (held item sprite follows pointer)
- **Hover active**: `Hand_Active` when pointing at an interactable
- **Craft-ready hover**: while dragging an inventory item over another inventory item that can be crafted with it, the hand cursor flickers between `Hand_Hold` and `Hand_Active` and the affordance pill shows **⚗ Craft**; an icon badge shows `?` for unknown recipes or the result icon for discovered recipes.
- **Click micro-feedback**: on **pointer down**, the hand cursor can perform a **tiny shake** (purely cosmetic; debug-tunable in F2 under **Cursor**). This is a *micro* cue meant to add tactility without implying a successful action (success/failure feedback still comes from SFX/log/shake events).
- **Global pointer sync**: **`CursorProvider`** listens for **`pointermove`** on **`window`** (**passive**) and runs the same hit-test / drag-promotion path as the legacy React **`onPointerMove`** API, so the hand tracks the pointer over **any** DOM (including **`stageModalLayer`** modals that are **not** under **`HudLayout`**, e.g. **Settings**). Per-widget **`onPointerMove`** forwarding was removed to avoid double updates and to keep ordering with **`setVirtualHover`** predictable (**React** handlers on the HUD tree run in the same native event **before** the **`window`** bubble listener consumes the one-frame virtual hover ref).
- **Deadzone behavior**: hovering non-interactive UI areas should clear any previous “active” hover so the cursor returns to `Hand_Point`; combined with the global listener, cursor state does not get stuck when crossing panels or modals.
- **Title / hub vs dungeon**: **3D pick / `floorDrop` virtual hover** runs only when **`ui.screen === 'game'`** (not on **title** or **hub**), so the center cell does not pin **`hoverTarget`** to **`floorDrop`** and block **`Hand_Active`** on real UI.
- **Menu chrome without `data-drop-kind`**: **Title**, **Settings**, and **combat menu** buttons (**DEFEND** / **WAIT** / **FLEE**) set **`data-cursor-hand-active`** so **`CursorLayer`** can show **`Hand_Active`** over those controls (via **`elementFromPoint`** + **`closest`**), matching other interactive affordances.
- **Combat action menu (world hit shield)**: while combat chrome is shown, the bottom **`centerCluster`** (**`CombatCenterActionMenu`**) uses **`pointer-events: auto`** so the flex cluster (menu card, turn-aside, gaps) captures pointer hits above the **3D** viewport; it sets **`data-combat-action-chrome`**. **`CursorProvider` `hitTestDropTargetAtPoint`** returns **no** world **`data-drop-kind`** when the pointer stack hits that subtree first, so the custom cursor does not show **attack** / **floor** affordances through the menu. **`HudLayout`** skips the **3D** **`pickObject` → `setVirtualHover`** injection when **`elementFromPoint`** is under **`[data-combat-action-chrome]`**, matching that shield (**ADR-0390**). **`GameViewport`** also bails out of **`pointerdown`/`click`** when **`elementFromPoint`** lies under **`[data-combat-action-chrome]`** (**ADR-0389**).
- **3D viewport hover override**: during **game**, the WebGL viewport may inject a **virtual hover target** (e.g. `floorDrop`) on pointer-move to compensate for `elementFromPoint` seeing the compositor canvas rather than a DOM hit target. This override is **ephemeral (one move event)** so normal DOM-derived hover targets (inventory slots, portraits, etc.) take over immediately when the pointer leaves the viewport.

### 6.2 Interaction rules (mouse-first)
- **Click**: attempt to use/interact with target (object/NPC/POI/UI element).
- **3D viewport ray pick**: when several pickables lie on the same camera ray, **floor items take priority** over everything else so loot is not blocked by large POI billboards (e.g. a chest in front of a dropped item). **Closed/locked** door billboards block targets **behind** them on that ray. **Passable open-door** tiles (**`doorOpen`** / **`doorOpenOctopus`**) use a **pickable** billboard (for **drag-to-smash** and click-to-step when it is the intended target) but **do not** block a **PoI** farther along the same ray—so clicking a chest (etc.) through an open doorway resolves to the **PoI**, not a bogus **`player/step`** onto the PoI’s cell (**“Something is in the way.”**). Among a **passable open-door** billboard and an **NPC** on the same ray, the **closer** pick wins (so an NPC in front of the doorway is not hidden by the open-door mesh behind them, and an open door in front of a farther NPC still reads as the door for click-to-step).
- **Drag (items)**: dragging starts when you **press and move** beyond a small threshold (to avoid accidental drags on click). **Press + hold** is also supported as a fallback to begin dragging items from **inventory** and from **world floor items** in the 3D view. **Movement promotion** in **`CursorProvider`** uses the same **synchronous refs** as **`beginPointerDown`** (**`pendingPayload`**, **`dragStartPos`**, **`dragPromotedRef`**) so the first moves after **`pointerdown`** are not ignored while React state is still catching up (otherwise the UI falls back to the **~140ms** hold timer and feels laggy). **`pointerup`** / **drop** resolution uses the same **“drag promoted”** flag so **`pointerdown`’s `preventDefault`** does not strand **tap-to-pick** UX (e.g. wanted inventory → **trade offer**) waiting on a stale **`dragging.started`** read. **Trade** / **inventory** item buttons still use **`stopPropagation` on `pointerup`** where needed so parent handlers do not double-resolve drops.
- **Drop**:
  - Onto **inventory**: store item
  - Onto **empty 3D view**: **cursor-aimed drop** (raycast to the floor under the pointer), clamped by a max Manhattan range (tunable). If the aimed cell is invalid/blocked, snap to the nearest valid floor cell within range; if none exist, fall back to dropping near the player. **POI / NPC / floor-item** targets use the same **`WorldRenderer.pickTarget`** raycast at pointer-up via **`resolveGameViewportDragDropTarget`** (not DOM `data-drop-kind`), including drags that start on **inventory** or **equip** slots—those use **`pointerCapture`**, so **`HudLayout`’s** root **`pointerup`** does not run (**ADR-0302**, **ADR-0303**). **Open-door** billboard hit with a valid breaking item resolves to **`openDoor`** (same range as drop) and clears the doorway (**§8.1**).
  - Onto **another item**: attempt crafting / recipe discovery (see §7.3)
  - Onto **NPC**: use item on NPC (may be rejected/consumed/apply status/attack). **Weapon** hits and **Captured Swarm** / **Captured slime** releases that **deal damage** to an NPC who **survives**: if they were **neutral** or **friendly**, they **become hostile** (activity log) first, then **start an encounter** when the roster rules in §7.7 apply (same as a **hostile** who survives an opening hit). Already-**hostile** targets behave as before. A **lethal** opening strike does **not** start combat.
  - Onto **portrait eye**: inspect interaction
  - Onto **portrait mouth**: feed interaction
  - Onto **character equipment strip** slots (**`head`**, **`handLeft`**, **`handRight`**): equip via **`equipmentSlot`** drops (same rules as §7.4 / paperdoll; **`equipItem`** in the reducer)
  - While **`ui.tradeSession`** is open (**§5.1**): onto **trade offer** slot — stage an inventory item whose **`defId`** is in the merchant’s **`wants`** (item leaves the grid until traded or the modal closes), or **click** a **highlighted** inventory slot to stage it (no drag); **their stock** — **click** only to select what you receive (**no** drag from stock); drag the staged offer back onto **inventory** to return it
- **Timing**: drop resolution uses the pointer-up timestamp (same `performance.now()` clock as `time/tick`) so short-lived UI cues (e.g. portrait mouth “chomp” burst) can’t expire immediately due to a stale tick time.
- **Preview affordance**: contextual icon appears near relevant elements indicating what releasing will do.

### 6.3 UX feedback
Interactions should resolve with:
- short relevant animation (e.g. eyes/mouth)
- sound
- **Activity log** (**`ui.activityLog`**, rendered as **`ActivityLog`** in a **bottom-right corner stack** in **`HudLayout`** with **`CombatEncounterCorner`** during encounters—the log sits **above** encounter chrome so the two do not overlap): each line of feedback (POI outcomes, crafting, combat hits, door messages, inventory pickup/full, inspect/feed text, debug **`ui/toast`** dispatches, etc.) is **appended** (newest last); the list is **capped** in memory (**`ACTIVITY_LOG_MAX_ENTRIES`**), each line **expires** after **10 seconds** of game time (**`ACTIVITY_LOG_ENTRY_TTL_MS`**, pruned on **`time/tick`**; pruning is **paused** while the **death** modal is up so the recap stays stable), and the widget **shows at most the four newest** entries. It uses **no** filled panel background (text sits over the scene, using **`--buttonTitle*`** color and shadow for contrast). **`ActivityLog`** applies **`pointer-events: none`** to **all** descendants so when the stack is **raised above** tavern trade (**`gameCornerStackAboveTrade`**, §5.1) long lines cannot **intercept** clicks meant for **Trade** / modal chrome. Lines are **right-aligned** in that corner stack. Typography reuses **`--buttonTitleFontFamily`** / spacing / color / shadow from **`index.css`** at **`font-size: calc(var(--buttonTitleFontSize) - 5px)`**, with **regular (400)** weight (unlike **`--buttonTitleFontWeight`** on modal buttons). There is **no** separate centered transient **`ui.toast`** overlay; the **`ui/toast`** action remains as a convenience and **only** pushes a log line.
- subtle camera shake for key interactions (driven by `ui.shake` and tunable in Debug/F2). **Shake length / hold (ms)** and **shake decay / fade (ms)** control the envelope for the **3D camera**, **HUD overlay shake**, and **portrait frame shake** (shared `shakeEnvelopeFactor` in `web/src/game/shakeEnvelope.ts`). With length 0, decay uses the legacy ramp `min(1, remaining/decay)`; with length above 0, full strength holds for the scaled hold segment, then linearly fades over the scaled decay segment within each event’s `startedAtMs`→`untilMs` window.
- **Portrait frame shake** on portrait interaction resolution (inspect and feed) and when an **NPC hit** deals damage to a PC in **`npcTakeTurn`**, driven by `ui.portraitShake` on the **struck** character’s slot (existing encounter **`ui.shake`** for the 3D view is unchanged—no second global shake)

### 6.4 Grid movement (first-person)
- **Model**: discrete cells; facing is one of four compass directions (**`playerDir`** is canonical 0=N,1=E,2=S,3=W); the 3D camera **yaw** matches facing when idle. **Strafe** moves one cell **left or right relative to facing** without rotating (**A** / **D**). **W** / **S** move **forward** / **backward** along facing.
- **Turning**: left/right turns always animate the **shortest** 90° rotation. Internally, the animated camera yaw may be temporarily **unwrapped** across the \(0 \leftrightarrow 2\pi\) boundary so the interpolation never takes a 270° path; when the turn completes the camera yaw snaps back to the canonical \(dir \cdot \pi/2\) in \([0,2\pi)\).
- **Keyboard** (see `GameApp.tsx`): **W** / **S** (and **↑** / **↓**) forward/back; **A** / **D** strafe left/right; **Q** / **E** turn left/right. **F2** toggles the debug panel. **Escape**: if **settings** are open → close them; else if **NPC dialog** (or F2 NPC preview) → **close dialog**; else if **paperdoll** → close; else if **trade** → **close trade**; else **toggle settings** (opens pause menu from title, hub, or game). **During an encounter** (§7.7): **R** attempts **flee**; **F** **defends**; **Space** **skips** the acting PC’s turn (same **`CombatCenterActionMenu`** rows as those keys when initiative is a **PC**). **ATTACK** has **no** keybind—use the menu **ATTACK** button or **click** an encounter foe in the **3D** view.

- **Portrait inspect**: resolving an eye drop plays a short **portrait frame shake** (gentler than feed), likewise without 3D shake. The **inspected character** (whose eyes you dropped onto) makes a **deterministic Perception check**: **d20 + Perception** vs a **DC** derived from the item’s tags (food/material **10**, container **11**, weapon/tool/hat **12**, quest **14**, default **12**; when multiple apply, the **highest** DC applies). Results are logged to **`ui.activityLog`**: always **who** inspects **which item name** and the **numeric check** (`d20 + Perception = total vs DC`). **Fail** (`total < DC`): a short “nothing more” line. **Success** (`total ≥ DC`): one line with a **high-level read** of the item plus **character-relevant** hints (e.g. remedy fit if they have a matching ailment, or species-tied feed effects when applicable). **Great success** (`total ≥ DC + 5`): **one or two** extra lines with **full mechanical detail** (feed deltas, possible status chances, weapon stats, equip slots, POI transforms) when the item defines them.
- **Portrait reaction rendering**: the HUD base is captured to a texture for postprocess. **Mouth flicker** (and **closed-mouth** swap when authored) is a **compositor-time overlay** in the WebGL presenter (using the live HUD layout rects) so chomp reads at **rAF** cadence without waiting on serialized **`html2canvas`** work—even when **`portraitToasts`** float above the frame, because the shader only draws inside **`portraitRectPx`** and toast pixels sit outside that rect (**ADR-0484**, supersedes the toast-bake path in **ADR-0483**). **`PortraitPanel`** may still bump **`portraitCaptureHudRev`** for other capture-only portrait deltas; **`DitheredFrameRoot`** treats **`ui.portraitMouth`** as active against the same **rAF `performance.now()`** used when the cue is authored. **Portrait idle** (ambient, **`portraitIdlePulse`**, and **frame press**) is **capture-baked**: the offscreen **`PortraitPanel`** hides base+eyes and includes the idle `<img>` in the rasterized HUD, and **`portraitIdleOn`** in the presenter stays **off** so the compositor does not double-draw idle over a stale **`uiTex`** (**ADR-0311**). **`killStaleUiUnderCompositorIdle`** in **`CompositeShader`** remains for the idle uniform path but is inactive while **`portraitIdleOn`** is zero (**ADR-0304**, **ADR-0305**). HUD capture still runs in a **burst** during **`portraitIdlePulse`** and **portrait press** (**`highFpsUi`**); the capture **`hudKey`** includes **pulse**, **press**, and **`pCapIdle`** so **`html2canvas`** tracks idle toggles. The **`chars=`** segment includes each row’s **`species`** plus **`tHue=`** (**`Character.tentReplacementPortraitHueDeg`**, or **`-`**) so duplicate-species CSS, tent tint, and F2 **Regenerate** / **Replace party** change **`hudDirty`** when portrait art would differ even if rounded vitals match, and the dithered **`uiTex`** does not stay stale. **`renderOnce`** reads **`cursorLatestRef`** for drag-hover and press hit-testing so rAF bursts see the **live** cursor (**ADR-0307** cursor-ref entry). While **`ui.screen === 'title'`**, the presenter **does not** apply portrait overlays (zero rects and flags): portrait nodes still exist under the full-screen title hit layer, but **mouth / idle / inspect** compositor quads must not paint on top of the main menu (e.g. **Afonso** closed-mouth swap).
- **Portrait shake tuning**: portrait frame shake (inspect, feed, and **NPC hit damage**) has its own envelope and amplitude tuning via `RenderTuning`:
  - `portraitShakeLengthMs` / `portraitShakeDecayMs` (envelope)
  - `portraitShakeMagnitudeScale` (amplitude multiplier applied to `ui.portraitShake.magnitude`)
  - `portraitShakeHz` (oscillation speed for the portrait shake transform)
  - Portrait shake duration is at least the authored base duration (combat hits use a **minimum** window so short tuning still reads), but increases to match the tuned envelope window so “hold/decay” can produce multi-oscillation shakes.
- **Character equipment strip** (`CharacterEquipStrip` beside each portrait): vertical chrome PNG **`ui_hud_charequip_background.png`** (**198×589** px, `Content/ui/hud/`) with three hit regions (**head** → **handLeft** → **handRight**, top to bottom) aligned to the art frames. Each region shows a **2px** frame via a **`::after`** border: default **`var(--hud-inventory-slot-border)`** (inventory slot color). While the cursor has an **active drag** (`CursorProvider` **`dragging.started`**), any slot where the dragged item **fits** (**`itemFitsCharacterEquipmentSlot`** in **`equipment.ts`**: **`hat`** / **`oneHand`** / **`twoHand`** plus optional **`equipSlots`**) uses **`var(--hud-stamina-vital-fill)`** (stamina bar color); on release or cancel, borders return to the default token. Strip **width** and **equipped icon** sizing use **`CharacterEquipStrip.module.css`** **`--equip-strip-scale`** (**1.15** vs a **78px**-wide baseline) so the whole widget scales together. Each region is an **`equipmentSlot`** drop target; equipped items render as draggable icons (void-drop **stow** uses the same **`fromPortrait`** equipment-drag path as before). **Two-handed** items show in the **bottom (right-hand) frame** only; drag/unequip still uses the **`handLeft`** slot id when both hands reference the same item. **Placement** follows **`charRail`** + **`charRailPushEnd`** (see **HUD** bullet above) so **left** and **right** rails are **symmetric**. ### 7.3 Items & crafting
- Items are found on dungeon floor and via POIs/NPC drops.
- **Crafting trigger**: drag one item onto another.
- **Feed (`ItemDef.feed`)**: optional **`primaryStamina: true`** marks items the UX should call out as deliberate **stamina snacks** (combined portrait toast line **“Stamina food: …”** when **`feed.stamina`** is positive).
- **If recipe exists**:
  - show a visible timer
  - perform a skill check (**d20** + party-best skill vs recipe DC); the **d20** is **per craft attempt** (each time the craft timer starts), not fixed forever for a pair of ingredient stacks—still derived deterministically from floor seed, item ids, recipe fields, and **`startedAtMs`**
  - on success: crafted result
  - on failure: notify; chance to destroy one involved item (only among ingredients that the recipe **would** consume on success; **preserved** catalyst slots are excluded)
- **Persistent catalysts**: some recipes set **`preserveA`** and/or **`preserveB`** so that ingredient is **not** consumed on a successful craft (and is **not** eligible for the failure “something broke” destroy roll). Slot **A** is the **source** stack, **B** the **target** stack, matching drag order. Examples: **Stick** + **Flint** → **Campfire kit** (**Flint** **`preserveB`**); **Stone** + **Chisel** or **Stone** + **Club** → **Stone shard** (**Chisel** / **Club** **`preserveB`**); **Foodroot** + **Stone** → **Mortar meal** and **Sweetroot** + **Stone** → **Mushed root** (**Stone** **`preserveB`**, reusable mortar).
- **Recipe discovery (cursor telegraph)**: while hovering a valid craft combo in the inventory, the cursor shows **⚗ Craft** and a small result badge. Before the recipe is discovered it shows `?`; after a successful craft it shows the resulting item icon.

Moving equipped items into inventory or onto the floor **clears equipment** first so items are not duplicated between equipment and grid.

### 7.7 Combat (encounter mode)
- **Trigger**: attempting to step into a **hostile** NPC starts an **encounter** (the player does not move into the NPC cell). **Dealing damage** to a **hostile** while **not** in an encounter (e.g. **weapon** drag onto the NPC in the **3D** view, or **Captured Swarm** / **Captured slime** release onto a valid target) **starts an encounter** if the NPC **survives** the hit (**lethal** opening strikes do **not** start combat); roster rules below apply immediately after that damage resolves. **Non-hostile** NPCs do not block movement or start encounters. **Roster**: the encounter always includes the **trigger NPC** (when eligible). Other **hostile** NPCs join only if they pass the usual spatial rule **and** are within **Chebyshev distance ≤ `render.combatEncounterJoinChebyshevMax`** of the **player** (F2 / `debug-settings.json`; default **5**). **Spatial rule**: every other joiner must be in the **same procgen room** as the trigger (first `floor.gen.rooms` rect containing the trigger cell). If no room contains that cell (corridor / missing gen), hostiles within **Chebyshev distance ≤ 1** of the **player** are candidates instead (they still must satisfy the join cap, which is **≥ 1** by clamp, so corridor behavior is unchanged at defaults). **Solo encounter** **boss** triggers (**`bossTraitId`** with **`soloEncounter`** in **`npcBosses.ts`**) **do not** pull other same-room hostiles into the roster. **Swarms** are excluded from the roster while the party holds a **Swarm Queen** (same neutrality rule as outside combat).
- **Mode**: while an encounter is active, **movement is blocked**; the encounter proceeds in **turn order**.
- **NPC combat data**: per-**`NpcKind`** baselines live in **`web/src/game/content/npcCombat.ts`** (`NPC_COMBAT_BY_KIND`). **`npcKindHpMax`** applies **`NPC_SPAWN_HP_TABLE_MUL`** (**×3**) to table **`hpMax`** for spawn + HUD; **`npcCombatTuningFromContent`** applies **`NPC_COMBAT_DAMAGE_TABLE_MUL`** (**×3**) to table **`baseDamage`** (minimum **1**). **Encounters** resolve **`Speed`**, **armor**, **resists**, and **on-hit statuses** per **NPC instance** via **`resolveNpcCombatTuning`** (boss scaling + traits in **`npcBosses.ts`**), using that scaled **damage** as the boss base. **`hp`** / **`hpMax`** are on the runtime row (spawn and encounter **HUD**; **`hydrateFloorNpcs`** back-fills **`hpMax`** for older saves). Shipped defaults stack **ADR-0423** table tuning with these global multipliers (**ADR-0425**–**ADR-0429** history; **ADR-0440** restores **×3**), with bosses still scaled **above** trash via **`npcBosses.ts`**.
- **Turn order (initiative)**: each living **party member** and each encounter **NPC** gets a turn; initiative is driven primarily by **Speed** with a small **deterministic tie-break** so ordering is stable per encounter seed.
- **Primary PC actions** (on the **acting** party member’s initiative — see **`CombatCenterActionMenu`** + keys): **Attack**, **Defend**, **Skip**, **Flee**.
  - **Attack** (cursor-first): dragging a **weapon** onto the target NPC performs an **attack** (consumes that character’s turn when the dragged weapon originates from that character’s equipment; inventory-origin weapons act as the current turn’s character). On a **PC’s turn**, **clicking** an **encounter enemy** in the **3D** view **or** the centered menu **ATTACK** row performs an **attack** using the same weapon resolution: **equipped** hands (**left**, then **right**; two-hand held once), else the **first weapon** found in the **inventory grid** (if none, feedback only; drag a weapon explicitly if needed). The **menu** path targets the **first living** NPC in **encounter roster** order; the **3D** path targets the picked foe.
  - **Defend** (HUD or **F**): spends **stamina** ( **`effectiveDefendStaminaCost`** from **`stats.endurance`**, base **`COMBAT_DEFEND_STAMINA_COST`**) and grants **temporary armor and resist** against **NPC hits** until that PC’s **next** turn begins (the buff clears when their initiative slot comes around again).
  - **Skip** (HUD or **Space**): **no** stamina; logs that the PC **waits**; advances initiative (`combat/skip`, **`skipPcTurn`**). **`combat.lastAction`** records **`skip`**.
  - **Flee** (HUD or **R**): only when initiative is a **PC** (the acting character). **Stamina** paid to **attempt** flee scales with that PC’s **`stats.speed`** (base **`COMBAT_FLEE_STAMINA_COST`** in **`combat.ts`**); the **success roll** also uses **Speed** vs a reference enemy (fastest in the encounter). Success ends the fight; failure provokes a **free enemy hit**, then **initiative advances**. On an **NPC** turn, flee input is **rejected** (**Not your turn.**). **Too exhausted to flee** rejects without advancing.
- **Additional turn-consuming actions** (still on the acting PC’s turn, same initiative rules as attacks): dragging a **non-weapon** item onto an NPC performs a **give/use** attempt (quest, hated, reject) and **consumes the turn** as today. **Fireshield** (see §7.3): drag onto that PC’s portrait **hands** (not onto an NPC). Spends **stamina**, **consumes** the item, applies **Fire resist** for **`shieldTurns`** of that PC’s **turns**. **`combat.lastAction`** uses **`skip`** for this path too (same **turn-economy bucket** as explicit **Skip**). **Does not** equip through this path during combat.
- **Combat HUD**: **`CombatEncounterCorner`** (**`CombatIndicator.tsx`**) in the **bottom-right** stack with **`ActivityLog`** lists **roster enemies** with a **compact HP bar** each (**current / hpMax**). **`CombatCenterActionMenu`** is **centered horizontally** and anchored **low** in the **game viewport** (full-bleed layer with **`pointer-events: none`** except the menu card so **click-to-attack** on **3D** hostiles still works): header is the **acting PC’s name** (em dash when it is not a **PC** turn), rows **ATTACK** **(click)** + **STA** cost or **—**, **DEFEND (F)** + **`COMBAT_DEFEND_STAMINA_COST`**, **WAIT (Space)** (**0** STA), **FLEE (R)** + **`COMBAT_FLEE_STAMINA_COST`**; **Current:** / **Next:** sit beside the menu. The **acting PC**’s **`PortraitPanel`** frame shows a **thick red border** during their initiative (border, not CSS outline, so dithered **`html2canvas`** capture includes it). **Capture HUD** shows the same layout read-only. No separate encounter modal.
- **NPC actions**: on NPC turns, the NPC automatically attacks a party member using **soft targeting**: **sort** living PCs by **lower current HP** first, **deprioritize PCs who are Defending** (virtual HP padding), and a tiny **deterministic jitter** per PC for stable ordering. The **acting NPC** then chooses among the **up to three** softest PCs with **fixed weights 3:2:1** (softest heaviest), using a **deterministic roll** keyed by **`floor.seed`**, **`combat.encounterId`**, NPC id, and **`combat.turnIndex`** (no `Math.random()`). Pool size: **`NPC_TARGET_CANDIDATE_POOL`** in **`combat.ts`**.
- **NPC hit vs party**: **d20 + NPC Speed** vs **10 + min(target PC Speed, `NPC_VS_PC_DEFENSE_SPEED_CAP`)** (then **−1** each for **Parasitized** / **Spored** on the target) decides hit/miss; **crit** on natural 20. Damage uses that NPC kind’s **base damage** and **damage type**, then **PC armor** (Blunt/Pierce/Cut only), **resistances**, **Defend** bonuses, and **Fireshield** extra **Fire** resist only (`npcTakeTurn`).
- **PC damage resolution**: in **encounter mode**, a **d20 + Perception + Agility** check vs **10 + NPC Speed** decides hit/miss; **crit** on natural 20. **Weapon stamina** (encounter only): integer STA per swing uses **`effectiveCombatAttackStaminaCost`** — **`weapon.staminaCost`** as base, adjusted when **`damageStat`** is set (same **Str / Agi / Int** mapping as damage); **hit or miss**. Weapons **without** **`damageStat`** pay the base **only**. **Defend** / **flee attempt** STA use **`effectiveDefendStaminaCost`** / **`effectiveFleeStaminaCost`** (**endurance** / **speed** from bases **`COMBAT_*`**). **Weapon damage** starts from **`ItemDef.weapon.baseDamage`**, optional **`damageStat`** adds **`floor(stat × 0.25)`** before run **damage %** bonus, then **NPC kind** applies **armor** (Blunt/Pierce/Cut only) and **per-type resist** (capped like PC resists). **Durability**: on each **successful** hit (encounter or open world), the used weapon stack may lose **`render.itemDurabilityWeaponHitCost`** when **`itemDurabilityEnabled`** and the def has **`durabilityMax`** (see §7.2). **Open-world** weapon strikes (not in an encounter) skip the to-hit roll but still apply stat bump and mitigation; **open-world** swings do **not** use the combat attack STA helpers (no encounter stamina gate on **`npc/attack`** outside combat).
- **Activity log (combat swings):** **Encounter** PC attacks and **NPC** attacks each append **one line** that spells out the **to-hit** math in the same style as inspect checks: PC lines use **`d20+Per+Agi … = total vs 10+Spd …`**, then **miss** or **hit** (note **nat 20** when the d20 is 20) and damage; **NPC** vs **PC** lines in **debug** use **`d20+Spd …`** vs an **AC** breakdown that includes **`10+min(Spd, cap)`** (and raw **Spd** when it exceeds the cap), then status **AC** modifiers. Open-world weapon hits keep the shorter outcome line (no encounter roll).
- **Quest line in combat:** Hostile NPCs with **`quest.wants`** sometimes echo their quest in the activity log **on their turn**, **before** the swing line. Chance is **20%** per NPC turn, **deterministic** from **`floor.seed`**, **`combat.encounterId`**, NPC id, and **`combat.turnIndex`** (same keyed hash style as other combat rolls). The line is **`{name}: "{gibberish}"`**, using the same **`toGibberish`** / NPC **`language`** as the dialog speech strip (**`npcQuestGibberishLine`** in **`web/src/game/npc/npcQuestSpeech.ts`**; English substrate for generation only is **`npcQuestEnglishLine`**).
- **Weapon procs**: `weapon.statusOnHit` can apply **timed debuffs** to **NPCs** on a successful hit (deterministic % roll).
- **World & crafting during combat**: **POI** use (viewport click or dragging an item onto a POI), **floor item pickup** (pickup action or drag onto a floor item), **starting** an inventory recipe craft, and **finishing** an in-flight craft (timer resolution) are **blocked** until the encounter ends (“Not while in combat.”). **Equip**, **unequip**, non-recipe inventory rearrangement, and **portrait** inspect / feed / most equip **stay available** during combat and do **not** consume initiative—**except** **Fireshield** dropped on the acting PC’s **hands**, which **consumes** the turn (recorded as **`lastAction` `skip`**). **Portrait feed** (food, remedies, and similar **feed**-path items) during an encounter is **deliberate party triage**: it does **not** cost the acting PC’s turn, so players can stabilize vitals or statuses under pressure; balance leans on **item scarcity**, **stamina** costs on attacks, and **enemy pressure** rather than blocking feed mid-fight.
- **Item actions during combat**: **Hive** floor drops, **Swarm Basket** capture, **Captured Swarm** release, **Swarm Queen** “calm swarm” updates, **Slime phial** capture, and **Captured slime** release are **blocked** during an encounter (“Not while in combat.”); use normal **weapon** / **give** / **skip** / **flee** / **defend** flows instead.
- **End conditions**: encounter ends when all encounter NPCs are dead (**victory**, grants **run XP** — see **Run XP / leveling** below) or when all party HP ≤ 0 (**party wipe**, death screen).
- **Orphan encounter reconciliation**: if **`combat.participants.npcs`** does not match any **living** NPC on **`floor.npcs`** (rare desync), **`time/tick`** and the start of **`combat/fleeAttempt`** apply the same **victory** cleanup as a normal win (**run XP**, **Encounter won.** log). The **Flee** HUD row stays **disabled** when there is no **Attack**-valid foe on the floor.
- **Run XP / leveling** ( **`state.run`**: **`xp`**, **`level`**, **`bonuses`**, **`perkHistory`**; implementation **`web/src/game/state/runProgression.ts`**):
  - **Curve**: XP to go from level **L** to **L+1** is **`XP_LEVEL_COST_BASE` + (L−1)×`XP_LEVEL_COST_STEP`** (defaults **25** + **20**×(L−1)).
  - **Combat victory**: **`combatVictoryXp`**: for each encounter NPC id, **round**(**`COMBAT_XP_PER_HP`**×runtime **`hpMax`** + **`COMBAT_XP_PER_DMG`**×**scaled** **`baseDamage`** from **`npcCombatTuningFromContent`**) plus **`COMBAT_XP_BOSS_FLAT`** when **`variant === 'boss'`**; sum, then multiply by **`combatXpDepthMul(state)`** from **`floor.floorIndex`** and **`floor.difficulty`** (capped at **`COMBAT_XP_DEPTH_MUL_CAP`**). If a roster id has no **`floor.npcs`** row, **+10** XP fallback for that slot. Activity log: one line **`Encounter won. +N XP`** (and **`level +k`** when applicable); perks still apply silently aside from that summary.
  - **Floor descent** (including **Exit**): **`XP_FLOOR_DESCEND`**.
  - **Locked door** (key consumed): **`XP_DOOR_UNLOCK`**.
  - **Crafting** success: **`xpForCraftSuccess(recipe.dc)`** = **`max(1, round(dc / XP_CRAFT_DC_DIVISOR))`**.
  - **POI / exploration** (first success where noted): **Barrel** / **Crate** / **Chest** open **`XP_CONTAINER_OPEN`**; **Shrine** use that removes **Cursed** from someone **`XP_SHRINE_PURIFY`**; **Cracked wall** tool success **`XP_SECRET_OPEN`**; **Campfire** roast success **`XP_COOK_SUCCESS`**; each **Kuratko nest** egg pried **`XP_NEST_EGG`**. **No** XP for **Well** checkpoint-only, **Bed** rest-only, **failed** craft/cook, or an extra grant on **Exit** beyond floor descent.
  - **Level-up lines**: **`pushLevelUpActivityLogs`** after **`applyXp`** for floor descend, door unlock, POI, and craft grants (correct **`Reached level N`** when multiple levels fire at once). Combat victory does **not** emit those separate lines today.

### 7.8 Status effects
- Status effects apply to PCs and NPCs; positive/negative/neutral.
- Some decay over time; some persist until cured by an item. **`time/tick`** prunes expired entries for **both** party characters and **floor NPCs**.
- Baseline content set includes **Poisoned, Blessed, Sick**; additional ids (e.g. **Bleeding**, **Burning**, **Frightened**) appear from items, combat procs, and **room hazards**.
- **Expansion**: **NanoTagged** (encounter: **−1** to PC attack to-hit total), **Spored** and **Parasitized** (encounter: each **−1** to PC “AC” vs NPC hits, i.e. `10 + Speed` defense term). **Salt** / **Moss** / **Antitoxin vial** and some feeds clear or mitigate these (see `interactions.ts`).

## 8) Dungeon & world generation
### 8.1 Taxonomy (tags drive content)
- **Floor types**: `Dungeon`, `Cave`, `Ruins`, plus **reskin types** `Jungle`, `LivingBio`, `Bunker`, `Catacombs`, `Golem`, `Palace`. Each reskin maps to a **layout profile** that reuses one of the three base realizers (**Cave**, **Ruins**, or **Dungeon** geometry): Jungle/LivingBio → Cave; Bunker/Golem → Dungeon; Catacombs/Palace → Ruins. Bespoke generators per fantasy floor are deferred; differentiation adds **`floorTopologyTuning`** (`web/src/procgen/floorTopologyTuning.ts`): per-**`FloorType`** knobs for the shared realizers (BSP / cave worm / ruins stamps), **room shaping** (pillars / alcoves / perimeter jitter), **`injectLoops`**, **junction-derived rooms**, **door-frame** caps (Dungeon profile only), **decorative-door** caps (**`decorativeDoorsMax`**, Dungeon profile only, with **`decorativeDoorFrameChance`**), **CA smooth passes**, and **`layoutScore`** weights among validation rerolls (Ruins profile uses extra **`junctionAntiBlob*`** fields in **`scoreLayout.ts`** to penalize hall-like junction density among rerolls). **`resolveTopologyTuning`** also applies small deltas from **`floorProperties`** (e.g. **Destroyed**, **Overgrown**, **Infested**, **Cursed**). Still layered with **theme** (`floor.gen.theme`), **env texture triples** (each **`FloorType`** has its own **`/content/*.png`** filenames in `dungeonEnvTextures.ts`; reskin triples are often **binary copies** of the matching base art until bespoke environment PNGs ship), **music bias**, and **spawn tables**.
- **Run progression (segments & camps)**: Dungeon floors are grouped into **segments** of **`render.campEveryFloors`** floors (default **10**, clamped **1–99** in tuning). Each segment uses one **floor type** from the fixed rotation: Cave → Ruins → Dungeon → Jungle → Catacombs → Palace → Bunker → LivingBio → Golem (then repeats). **`floor.floorIndex`** still counts individual dungeon floors (0-based). **Procgen difficulty** (`0` / `1` / `2`) follows **segment index**: segments **0–1** → easy, **2–4** → normal, **5+** → hard. After completing the last floor of a segment (**e.g.** exiting floor **9** when **N = 10**), the game opens a **camp hub** (`ui.hubKind === 'camp'`): same village + tavern flow as the **starting hub**, but **camp tavern/bartender** art under **`/content/camp_tavern_*.png`** / **`camp_bartender_*.png`**, **`CAMP_INNKEEPER_TRADE`**, and a **themed camp village** backdrop: **`/content/camp_<skin>.png`** where **skin** is **`layoutProfile`** applied to the **`FloorType`** of the **segment the player just finished**—**`floorTypeForFloorIndex(max(0, floor.floorIndex - 1), clamped campEveryFloors)`** (same schedule as **`runFloorSchedule.ts`**; while at camp, **`floor.floorIndex`** is already the **next** dungeon floor, so **index - 1** is the last floor of the completed segment): **`Cave`** profile (**`Cave`**, **`Jungle`**, **`LivingBio`**) → **`camp_cave`**; **`Dungeon`** profile (**`Dungeon`**, **`Bunker`**, **`Golem`**) → **`camp_dungeon`**; **`Ruins`** profile (**`Ruins`**, **`Catacombs`**, **`Palace`**) → **`camp_village`**. On the **interactive** village view, pointer hover over **Tavern** or **Cave** (dungeon exit) shows **`/content/camp_<skin>_tavern_hover.png`** or **`_dungeon_hover.png`** respectively; the **starting hub** uses **`village_tavern_hover.png`** / **`village_dungeon_hover.png`** over **`village.png`**. **Capture** hub rendering omits hover overlays. The **next** floor is **pre-generated** before the camp screen; play resumes when the player leaves the camp via the cave entrance. The **starting village** after **New run** is not a camp (`hubKind` unset).
- **Floor properties**: Infested, Cursed, Destroyed, Overgrown, None (not shown)
- **Room function**: Passage, Habitat, Workshop, Communal, Storage
- **Room properties**: Burning, Flooded, Infected, SporeMist, NanoHaze, Unstable, Haunted, RoyalMiasma (on-enter effects and HUD telegraph; some skew spawns)
- **Room status**: Overgrown, Destroyed, Collapsed
- **Doors/keys**: passages can be blocked by door types requiring appropriate keys; procgen records **`keyDefId`** per locked door in **`floor.gen.doors`** (gameplay opens a door only if the party holds a matching item, e.g. **IronKey** / **BrassKey**). Grid tiles include plain **`door`** / **`lockedDoor`** plus variant **`doorOctopus`** / **`lockedDoorOctopus`** (same walk/open rules; procgen rolls **`lockedDoorOctopus`** on a subset of locks, biased upward on **Infested** floors). **Dungeon** profile: after the lock pass, unused **door-frame** throat **`floor`** cells (**`isDoorFrameCandidate`**, skipping **`occupied`**) are **shuffled**, then each is rolled against **`decorativeDoorFrameChance`** in **`floorTopologyTuning`** (default **0.75**) until **`decorativeDoorsMax`** closed doors are placed (per-**`FloorType`**, e.g. **8** on base **Dungeon**, **4** on **Bunker**, **11** on **Golem**); no **`gen.doors`** entry — interaction opens them **without** a key. After open/unlock, tiles become **`doorOpen`** or **`doorOpenOctopus`**: **walkable** like **`floor`**, with a **persistent** 3D billboard (**`door_open.png`** or, for octopus, **`door_octopus_opening_03.png`** after the opening strip). **Octopus** only: on open, **`tryOpenDoor`** appends **`ui.doorOpenFx`** (**`visual: 'octopus'`**, **900 ms**); **`WorldRenderer.syncDoorFx`** advances **`door_octopus_opening_01..03`** at **`DOOR_OCTOPUS_OPEN_FRAME_MS`** (**280 ms**) per frame (clamped to one pass), while **`syncDoorSprites`** hides the static **`doorOpenOctopus`** pickable at that cell so the mesh does not stack with the overlay. **`time/tick`** drops expired **`doorOpenFx`** rows. The player may **destroy** an open doorway by dragging an eligible item onto that billboard (within **`dropRangeCells`**): tile becomes plain **`floor`**, and any **`floor.gen.doors`** row at that **`pos`** is removed so metadata matches. Procgen never emits **`doorOpen*`**; they appear only from play. **`RenderTuning`** **`doorWoodenSprite*`** / **`doorOctopusSprite*`** (F2; §3) adjust **height**, **vertical center**, and **XZ nudge** per door **visual** for **closed** and **open** door billboards and the **`ui.doorOpenFx`** overlay.
- **Difficulty** (`0` = easy, `1` = normal default, `2` = hard): **`GameState.floor.difficulty`** and **`FloorGenInput.difficulty`**. Tunes how many **validation rerolls** run (**8** / **6** / **5**), minimum entrance→exit shortest-path length before placing locks (**8** / **4** / **4** cells for any lock; two-lock branch when path length ≥ **11** at normal and ≥ **9** at hard — easy keeps two-lock **disabled**), and scales the **`layoutScore`** lock+loop bonus when choosing among valid rerolls (easier prefers layouts with less lock+loop weight; harder the opposite). Echoed as **`floor.gen.meta.difficulty`**.

### 8.2 Starting algorithm (prototype baseline)
**Floor type** (full **`FloorType`** union; see §8.1) selects a **layout profile** and thus a **geometry realizer**, then shared post-passes and population:

1. **Layout (per `layoutProfile(floorType)` → `Dungeon` | `Cave` | `Ruins`)**, parameterized by **`resolveTopologyTuning(floorType, floorProperties)`** (defaults preserve legacy behavior for the three base **`FloorType`** values).
   - **Dungeon profile**: BSP partition (min leaf / depth / room fractions / extra connector distances from tuning), sibling **L-corridor** stitch, then a guarded **door-frame** pass (**`doorFramesMax`** from tuning, e.g. **7** on base **Dungeon**) that introduces 1-tile **throats** in straight corridors so rooms read as connected by doorways (locks prefer these frames).
   - **Cave profile**: seeded worm carve + occasional widen + chamber blobs (digger count, step budget, widen/blob rates, chamber cap from tuning); **GenRoom** list from largest open chambers or fallback bbox.
   - **Ruins profile**: macro-cell stamps (cell size, skip rate, **`doorwayChance`** for **extra** adjacency only, **`spanningTreeDoorways`**, macro cluster cap from tuning). **`runRuinsLayout`** builds a **random spanning forest** over **orthogonal adjacency** between stamped cells and **always attempts** a macro doorway carve along those edges (subject to the usual floor-mass guard), then rolls **`doorwayChance`** for each **remaining** adjacency. Stamped chambers are **clustered into macro `GenRoom`s** by **union–find**: an edge exists when a **macro doorway carve succeeds** between two stamped cells, or when two stamp rects **touch or overlap** (Chebyshev rect distance **0**). If the component count exceeds **`maxMacroClusters`**, smallest components merge into nearest neighbors by rect distance (unchanged cap). Reskin example: **`Catacombs`** uses a **tighter** macro cell than base **`Ruins`**; **`Palace`** uses a **wider** cell. **`injectLoops`** pair-picking and thick-corridor chance are per-type via **`loopInject`**; **Ruins** profile defaults **`injectLoopsMax` to 0** (reskins may raise slightly). **`spanningTreeDoorways: false`** restores legacy **independent Bernoulli** doorways on every edge (tests / special tuning).
2. Connectivity repair (deterministic), then CA **carve-only** smoothing (**`smoothPasses`** from tuning; **Ruins** profile **0** so thin inter-cell walls are not eaten in one pass; **Dungeon** / **Cave** profiles default **1**).
3. Exit selection (exit = **farthest** reachable floor cell from the entrance by BFS).
4. **Derived connector rooms**: a small number of **junction/connector** rooms are derived from corridor junction clusters (bounded) and treated as **Passage** anchors for tags/spawn bias. Max derived rooms comes from topology tuning (defaults: higher on **Dungeon** than on Cave/Ruins).
5. **Districts**: seeded Voronoi on room centers → **`district`** tags (`NorthWing`, `SouthWing`, …) for spawn bias.
6. **Room tagging**: quota-aware function assignment (e.g. try to keep at least one **Storage** among tiny rooms) plus rolls from **`floorProperties`** (Infested/Cursed/Destroyed/Overgrown). Derived connector rooms keep their fixed `Passage` function.
6. **Tag constraints** (post-quota): **Storage** prefers a **dead-end** room center (swap with a tiny **Passage** room when needed); on **Cursed** floors, **Flooded** expands to **edge-adjacent** rooms with a fixed probability so hazards form a small cluster.
7. Population pass:
   - POIs (**Well** / **Bed** / **Chest** / **Barrel** / **Crate** / optional **Shrine** / optional **CrackedWall** / optional **Kuratko nest** / **Exit**) from entrance–exit heuristics + storage-room bias for chest/barrel/crate (Exit spawns at/near `gen.exit`). **Well** may spawn **`drained`** on **`floorIndex ≥ 1`** (~**30%**, seed-stable hash on **`attemptSeed`**); **floor 0** is always **filled**. **Kuratko nest** (see §9) is attempted only on **Cave** / **Jungle** / **LivingBio** / **Ruins** when a suitable **Habitat** room center is free. Each blocking PoI candidate is accepted only if **progression stays valid**: from the party spawn cell implied by **`pickPlayerSpawnCell`** at **`entrance`** (see §6.4), BFS over **`isWalkable`** tiles treats every **placed PoI cell** and the **exit** cell as non-traversable and must still reach **at least one orthogonal neighbor of `gen.exit`** (so the Exit POI remains usable from beside the stairs).
   - **NPCs**: optional **boss** placements first (**`spawnNpcsAndItems`** + **`npcBosses.ts`**, up to **three** weighted picks, separate RNG split from **`streams.population`**; dumps use **`floor.gen.meta.genVersion` 6** with optional **`variant` / `bossTraitId`** on rows). Unit tests that assert exact NPC totals pass **`spawnBosses: false`**. Then the usual NPC count roll and **`spawnTables.ts`** bias (room function, **`roomStatus`**, district, **`floorType`**, **`floorProperties`**, path band). When a **Kuratko nest** POI survives dedupe, **`generateDungeon`** passes **`kuratkoNestPos`** into **`spawnNpcsAndItems`**, which runs **`ensureKuratkoInNestRoom`** (see §9): the nest’s **`GenRoom`** ends with **≥1** non-boss **Kuratko** (retune a non-boss NPC already in that rect, else inject on the first free **floor** cell in scan order—the inject path can add **+1** NPC if the room had no non-boss spawns).
   - Floor items from **`spawnTables.ts`** (same contextual inputs as NPC kind rolls) plus forage-patch clusters where applicable
8. **Lock/key pass**: first on the entrance→exit **shortest path** (BFS **shuffles** orthogonal neighbor order with the **locks** RNG stream so ties vary by seed). Each candidate must **separate** exit from entrance when treated as a closed lock. Long paths may get **two** gates (**A** + **IronKey**, **B** + **BrassKey**); shorter paths use **one** (**A** + **IronKey**). **`Palace`** floors pass **`targetLockCount`** **`0` / `1` / `2`** from **`meta.plannedMission`** so procgen lock count matches the abstract template (**`0`** skips locks entirely). When **`targetLockCount === 2`**, a two-lock attempt may run even if base **`difficulty`** would set **`allowTwoLock`** false. If no path slot works (common after **loop inject** widens alternate routes), **fallback** scans **all** eligible **`floor`** cells (skipping **`occupied`** / entrance / exit), preferring **door-frame** throats in **near-entrance** order, for the same **separatesExit** property; key position uses the shortest path from **`entrance`** to that cell (or **`entrance`** when the lock is the first step). Otherwise skip locks. Keys carry **`forLockId`** in **`floor.gen.floorItems`** for validation/debug. **Dungeon profile only**, after updating **`occupied`** with lock cells and keys: **decorative closed doors** — collect door-frame throat **`floor`** cells, skip **`occupied`**, **shuffle** candidates (Fisher–Yates on **`decorDoors`**), then visit in shuffled order: roll **`decorativeDoorFrameChance`** (topology tuning; default **0.75**) until **`decorativeDoorsMax`** placements or the list ends; each placement sets **closed** **`door` / `doorOctopus`** (same octopus weighting as locks, e.g. **Infested** bias). Count recorded as **`meta.layoutMetrics.decorativeDoorsApplied`**.
9. **`missionGraph`**: nodes (entrance, exit, POIs, locks, keys); **path** edges chain POIs by BFS distance from entrance, then the lock sequence, then exit; when **`meta.plannedMission`** is set (**Palace**), lock and key nodes follow the planned **linear** path order; **`shortcut`** edge (entrance→exit) is present when multiple shortest-length routes exist; **`hasAlternateEntranceExitRoute`** mirrors that flag for tools.
10. **Theme**: **`floor.gen.theme`** (`id` + floor/wall/ceiling **color** multipliers) from a dedicated RNG stream; **`WorldRenderer`** tints **Lambert** dungeon materials (textures unchanged).

**First load**: `makeInitialState` runs the same `generateDungeon` pipeline as `floor/regen` (**31×31**), with **`floor.seed`** (random on cold start), **`floor.floorIndex`** (0 on first floor), **`floor.floorType`** (default **Dungeon**), **`floor.floorProperties`** (default empty), and **`floor.difficulty`** (default **1**). The canonical bundle is **`floor.gen`** (`genVersion` **6**, or **7** when **`meta.plannedMission`** is stored — currently **Palace** only); **`playerPos`** is **`pickPlayerSpawnCell(..., gen.entrance, gen.pois)`** (§6.4), not necessarily the raw **`gen.entrance`** tile when that tile is a door or blocked by a POI.

**Meta-progression**: floors are advanced by interacting with an **Exit POI** placed at/near `gen.exit`. Descending increments **`floorIndex`**, cycles **`floorType`** through the full set (base three then reskins, in code order), and generates a **new random `floor.seed`** for the next floor.
Default floor dimensions are **31×31**; `floor/regen` keeps the current **`floor.w` / `floor.h`** and reapplies **`floorType` / `floorProperties` / `floorIndex` / `difficulty`** from state unless you add UI to change them.

### 8.3 Determinism requirement
Generation should be **seeded** and phase-stable (changes to one phase shouldn’t ripple unpredictably into others) to support future host-synced multiplayer.

Validation is part of generation: the generator runs **bounded deterministic rerolls** (attempt count depends on **`difficulty`**; see §8.1) when **ordered** lock/key constraints fail, when **safety POIs** violate reachability caps (**Well** within **3** BFS steps of **`entrance`**, **Bed** within **48**), when **PoI layout** would block **spawn → exit-adjacent** reachability (same rule as placement; applies even when there are **no** procgen locks), or when the floor has **any** procgen **locks** but the **geometric** entrance→exit shortest-path **lattice** is too narrow (**`latticeCells` must exceed `shortestLen + 1`**). That lattice metric uses the same walkability as **`isWalkable`** in **`validate.ts`**: **all door tile types** (including **`lockedDoor` / `lockedDoorOctopus`**) count as traversable **only for measuring band width**, so validation does not require an entrance→exit path through **closed** locks. **Gameplay** checks still use lock-aware BFS: with all locks **closed**, the **exit** must be **unreachable**; with locks opened in order, each **key** must be reachable; with all locks **open**, the **exit** must be reachable; **Well** / **Bed** distance caps use BFS where **`lockedDoor` / `lockedDoorOctopus`** are **not** traversable. **`layoutScore`** (function **`scoreLayout`** in **`web/src/procgen/scoreLayout.ts`**) measures **reachable walkable passage** from **`entrance`** using **`isWalkable`** (so **locked doors** count like **floor** for mass, junction, and dead-end terms), adds a **lock presence** bonus when procgen **`doors`** include an ordered lock (scaled by **`difficulty`** so rerolls prefer validated lock floors over near-tie no-lock layouts), and still adds **lock+loop** when the geometric lattice branch bonus is non-zero. Among attempts that pass validation, the run picks the one with the highest **soft `layoutScore`**; if none validate, it returns the **last** attempt (still full geometry) like before. Output always has a non-empty **`rooms`** list except catastrophic fallback (single carved room).

**Meta** records **`inputSeed`**, **`attemptSeed`**, **`attempt`**, **`layoutScore`**, **`difficulty`**, and **phase streams** (`layout`, `tags`, `population`, `locks`, `districts`, `score`, **`theme`**, **`mission`** for **`planMissionBeforeGeometry`** on **Palace** (`null` / omitted for other **`floorType`**s today), **`decorDoors`** for **Dungeon** decorative door rolls). When **`plannedMission`** is present, it is JSON-serializable ( **`templateId`**, **`nodes`**, **`edges`** ). **`layoutScore`** uses per-**`FloorType`** weights from topology tuning (reroll comparison only); changing tuning or weights changes which attempt wins for the same **`attemptSeed`** stream without bumping **`genVersion`**.

### 8.4 Reference pipeline (expanded)
See `Dungeon_generation_plan_summary.md` for an expanded phased pipeline (mission graph → lock correctness → geometry → districts → tagging solve → population → validation → debug overlays). The **geometry-first** pipeline is still what ships for layout; **`web/src/procgen/missionFirst.ts`** defines **`PlannedMission`** / **`planMissionBeforeGeometry`**. For **`Palace`** only, **`planMissionBeforeGeometry`** returns a validated linear plan (**`palace_spine`**, **`palace_seal_a`**, **`palace_seal_ab`**) before Ruins layout; **`placeLocksOnPath`** enforces **`targetLockCount`**; **`validatePlannedMissionRealized`** gates rerolls. **Track B** in that file still describes future **stamp↔node** embedding for Ruins geometry and broader **`floorType`** coverage.

**Debuggability**: the F2 Debug panel includes a **Procgen** readout (**difficulty**, **floorType**, **floorIndex**, a compact **`topologyTuning`** line from **`resolveTopologyTuning`** (includes **`doorFr`** / **`decorMax`** among other knobs), **floorProperties**, **genVersion**, **`streams.mission`**, **`plannedMission`** (**`templateId`**, **Palace**), **attempt**, **layoutScore**, **theme** id, **missionGraph** size and alternate-route hint), **Cycle type** (advances through all **`FloorType`** values for the next regen), **Cycle difficulty** (easy→normal→hard for the next regen), **Proc overlay** (cycles **off** / **districts** / **roomTags** / **mission** tints on the 3D floor from **`floor.gen`**), and **Dump `floor.gen`** to download the canonical JSON.

## 9) Points of interest (POIs)
Static interactables. POIs **do not** auto-trigger on **walking past** them, but their grid cell **blocks player occupancy**: you **cannot stand on the same tile** as a POI (see §6.4). Interact via **click** or by **dragging items onto them** (typically from beside the object in the 3D view).

- **Placement invariant**: there is **at most one POI per grid cell**. If procgen produces multiple POIs for the same cell, the game resolves it deterministically (canonical IDs win, then kind priority) so runs remain stable and interactions remain unambiguous. **Progression invariant**: no PoI may sit on a tile that would make it **impossible** to walk from the **entrance spawn** (see §6.4) to **beside the Exit** while treating all PoI tiles and the **exit stair** tile as blocked; procgen skips or relocates candidates (optional **Shrine** / **CrackedWall** / **Kuratko nest** may be omitted if no valid tile exists).
  - **3D view**: POIs render as **sprite billboards** (same texture pipeline as NPC billboards: nearest filtering, transparent PNG), except **Kuratko nest**, **Campfire**, and **CrackedWall**, which use **emoji** billboards: **🪺** / **🪹** for the nest, **🔥** for **Campfire** (**Campfire** only: slow **opacity** flicker in **`WorldRenderer`**, per-instance phase), **🛘** for **CrackedWall** — all drawn with the same **canvas** path as **floor-item** emoji icons. **Well (filled)** uses **`npc_well.png`** plus extra **non-pickable** billboards: **`npc_well_glow.png`** (slightly larger halo) and a small **sparkle** layer cycling **`npc_well_sparkle_1..3.png`** (~280 ms per frame); **`render.poiSpriteBoost`** scales **Lambert** on the base well mesh and **`SpriteMaterial.color`** on **glow** + **sparkle** so the whole stack tracks F2 like **NPC** billboards + **Elder**. **Well (drained)** uses **`npc_well_drained.png`** only (no glow/sparkle). **Chest**, **Barrel**, and **Crate** use closed/open PNG pairs (`chest_*`, `barrel_*`, `crate_*`). **Exit** uses **`stairs_down.png`** at **2×** the default POI billboard scale (other POIs use a uniform base height). **Shrine** uses **`shrine_gnome.png`** until the first interaction, then **`shrine_gnome_off.png`**. **Bed** uses **`/content/poi_bed.png`** until dedicated POI art exists. Billboards use a **bottom-centered pivot** (`Sprite.center`) so the texture’s lower edge anchors to the floor; vertical placement is **`poiFootLift`** (F2: **POI above ground**, independent of **`npcFootLift`**). Per-kind **`render.poiGroundY_*`** (F2 for **every** **`PoiKind`**) shifts where the “ground contact” sits within the PNG (for emoji POIs, within the square **canvas**). Brightness uses **`render.poiSpriteBoost`** (F2; default tuned brighter than 1.0).

Initial POIs:
- **Well**: **checkpoint save point** (clear notification); used to fill **empty** portable water vessels (**Waterbag**, **Waterskin**, **Travel flask**) when the well still has water; a successful **empty-vessel** use sets the POI to **drained** (visual swap + VFX off). **F2** offsets **glow**/**sparkle** layers vs the well with **`render.poiWellGlowNudge*`** / **`render.poiWellSparkleNudge*`** (**X** **−1..0.5**, **Y**/**Z** **±0.5** world units). Procgen may place a **dry** well on **`floorIndex ≥ 1`** (~**30%**, deterministic); **floor 0** wells always start **filled**. Dragging an **empty** vessel onto a **drained** well logs *The well is dry.* **Checkpoint save still works when drained**. Checkpoints are reloadable from the **death screen** and the **title** screen. The same snapshot is written to **`localStorage`** (**`elfenstein.runCheckpoint`**) whenever **`run.checkpoint`** updates, and removed when the run has no checkpoint (**ADR-0373**); **single slot**, browser-local, subject to normal **localStorage** size limits.
- **Chest**: opens (sprite change) and drops one item, chosen **deterministically** from a **per-`floor.floorType`** pool (`STORAGE_POI_LOOT_DEF_IDS_BY_FLOOR` in `poiLootTables.ts`; roll: `hash(floor.seed + ":chest:" + poiId) % pool.length`).
- **Barrel**: opens (sprite change) and drops one item the same way (**`:container:`** hash namespace so barrel/crate rolls differ from chest for the same poi id pattern); the **open** barrel still **blocks** its cell until a **further** use **clears** it (POI removed, no rubble sprite).
- **Crate**: same as barrel (**shared** per-floor pool and **`:container:`** namespace).
- **Bed**: **one use per floor**: restores HP and Stamina (same heal amounts as before); further clicks log **Already used.** (POI stays on the tile; `opened` tracks use)
- **Shrine**: **use** (click or item drop without a matching **`useOnPoi.Shrine`** hook) removes **Cursed** from the party and marks the shrine used (sprite swap). Dragging an item whose def includes **`useOnPoi.Shrine`** with **`consumeOffering`** / **`blessPartyMs`** runs the **offering** path instead: consumes that item, applies **Blessed** per content, logs a toast; does not replace the standard “purge **Cursed**” use (player can still use the shrine normally afterward if not yet opened).
- **Kuratko nest** (procgen, optional): drag **Chisel** or a non-consuming **weapon** (same eligibility as splintering a **passable open door**) onto the nest to **pry** one **Kuratko egg** per action onto the nest tile (**`party.items`** + **`itemsOnFloor`**, log *You pry a Kuratko egg loose.*). Wrong items log *That will not reach into the nest.* A bare **viewport click** only hints that a tool is needed. **`FloorPoi.eggsLeft`** starts at **2–4** (deterministic from **`floor.gen.meta.attemptSeed`** and a fixed id string via **FNV-1a**, without consuming the **`population`** RNG stream in **`placePois`**); when it reaches **0**, **`opened`** becomes **true** and the **3D** billboard swaps from **🪺** to **🪹**; further **click** logs *The nest is empty.* Dragging a **valid tool** onto an **empty** nest **removes** the **POI** (*You scatter the empty nest.*). Spawned only on **Cave** / **Jungle** / **LivingBio** / **Ruins** when a **Habitat** room center passes progression checks. **Population tie-in**: the **`GenRoom`** whose **`rect`** contains the nest tile is guaranteed **≥1** non-boss **`Kuratko`** NPC after **`spawnNpcsAndItems`**—retune the first eligible non-boss NPC in that rect (stable **`id`** order) to **Kuratko** (**hostile**, refreshed **HP**), or if none, inject **`g_npc_Kuratko_nest_{x}_{y}`** on the first free **floor** cell in that rect (deterministic scan); **boss** NPCs are never retuned, and if the room has no free **floor** cell the guard is skipped.
- **Exit**: descends to the next floor (meta-progression)

## 10) Audio
Three audio layers run simultaneously via the Web Audio API:

- **Background music** (`MusicPlayer` / `MusicLayer`): loads an audio file, loops it continuously, and applies a tunable master volume (`masterMusic`). Starts on first user interaction (browser autoplay policy). Currently plays `Assets/sounds/theme.mp3` (served from `public/sounds/`).
- **Spatial ambient** (`SpatialAudio` / `SpatialAudioLayer`): NPCs and POIs emit continuous synthesized tones audible from a configurable number of cells away. Attenuation uses both **lowpass** and **volume** with distance (tunable in F2).
- **Procedural SFX** (`SfxEngine` / `FeedbackLayer`): discrete one-shot sounds generated from scratch using oscillators, noise, envelopes, and filters. Triggered via `ui.sfxQueue`. Kinds: `ui`, `hit`, **`swing`** (weapon swing / to-hit miss, distinct from UI **`reject`**), `reject`, `pickup`, `munch`, `step`, `bump`.

Volume controls: `masterMusic` (music layer) and `masterSfx` (SFX + spatial) are both in `AudioTuning` and adjustable in F2.

## 11) Graphics & rendering spec
- Three.js WebGL with `MeshLambertMaterial` on dungeon geometry (diffuse + emissive; no PBR metalness/roughness).
- **Dungeon albedo**: floor / wall / ceiling voxels use **`MeshLambertMaterial.map`** on unit-ish box faces. **`Dungeon`**, **`Cave`**, and **`Ruins`** use tiled PNGs from `Content/` (stable **`/content/…`** URLs; Vite serves the repo **`Content/`** tree in dev and copies it into **`dist/content/`** on build), selected by **`getDungeonEnvTextureSrcs`** in **`dungeonEnvTextures.ts`** (**`Dungeon`** — **`dungeon_floor.png`**, **`dungon_wall.png`**, **`dungon_ceiling.png`**; **`Cave`** — **`cave_*`**; **`Ruins`** — **`ruins_*`**). **`Jungle`** uses the same **`overgrown_*`** triple as overgrown rooms (single shared texture load in **`WorldRenderer`**). **`Bunker`**, **`Golem`**, **`Catacombs`**, **`Palace`**, and **`LivingBio`** use small CPU-generated **`THREE.DataTexture`** albedos (**`dungeonEnvProceduralTextures.ts`**: each of the five types gets its **own** floor / wall / ceiling **pattern** (e.g. metal deck vs flagstone vs diamond stone vs marble checker vs organic mottle; corrugated vs ashlar vs narrow brick+arch vs flutes vs membrane blobs; vents vs radial cracks vs coffers vs rosettes vs veins), plus per-theme RGB tint; no PNG decode, minimal VRAM).
- **Overgrown rooms (selective env swap)**: procgen rooms with `roomStatus: Overgrown` render their **floor + ceiling** using **`overgrown_floor.png`** and **`overgrown_ceiling.png`** (served at `/content/overgrown_*.png`). **Walls** bordering an overgrown walkable tile use **`overgrown_wall.png`** (4-neighbor adjacency) so room boundaries read clearly without changing the core geometry pipeline.
- Anti-aliasing **off**; pixel ratio capped at **1.5×** (sharp edges, dither-friendly).
- Lighting:
  - warm **PointLight** on camera (“lantern”): **no** equipped **`playerLight`** anywhere → **`bareLightIntensity` / `bareLightDistance`** × theme **`lanternIntensityMult`** × global × flicker. With **any** equipped source → **intensity** is the **party-wide sum** of per-slot bases (**`heldTorch*`**, **`equippedLantern*`**, **`headlamp*`**, **`glowbug*`** × per-row bug count) × per-tag theme mults, × global × flicker, then **`min(..., render.equippedLightIntensityCap)`**; **distance** is **max(d) + √(Σ_rest d²)** over each contributing source’s configured reach (same per-mode **`render.*Distance`** keys; **glowbug** **d** includes **√jar bugs** per row). **POI** torches use **`torchIntensity` / `torchDistance`** only.
  - optional forward **SpotLight** “beam” aligned to camera direction (tunable; often off via intensity scale). When effective beam intensity is zero, the spot is **not visible** and does **not** cast shadows (no wasted shadow pass).
  - POI **torch** PointLights: **`torchIntensity` / `torchDistance`** only; up to **`torchPoiLightMax`** (F2), choosing the **nearest** POIs to the player by Manhattan grid distance (0 disables torches). Shipped **`debug-settings.json`** uses a short **`torchDistance`** (**2** world units) as the authored baseline.
  - torch flicker via per-torch sine intensity
  - **Shadow mapping** is enabled only while at least one lantern light has shadows on: **point** shadows (expensive cube map) and **beam** shadows are separate F2 toggles (`shadowLanternPoint`, `shadowLanternBeam`). **Shadow map size** and **filter** (basic / PCF / PCF soft) are tunable in F2; shipped **`debug-settings.json`** sets **`shadowFilter`** to **Basic** (**0**). Dungeon geometry casts/receives where applicable.
  - low emissive lift on dungeon materials; the lantern is the primary visibility driver
  - `FogExp2` is **optional** for depth falloff; **disabled by default** (can be enabled in F2 debug); the renderer reuses one fog instance instead of allocating every frame
- Post-process: ordered dithering as shader pass (EffectComposer), controls:
  - Strength
  - Colour preserve
  - Pixel size
  - Levels
  - Post-dither levels (classic lift/gain/gamma; affects HUD + 3D equally)
  - Matrix (Bayer 2×2/4×4/8×8)
  - Palette (Dungeon warm, Cold crypt, Monochrome, Sepia print, No palette snap)
  - **Warm palette mix** (F2): when palette is **Dungeon warm** (0), blends between **quantised dither only** (like no snap) and **full warm five-colour snap**; other palette indices ignore this control
- **Room-property telegraph (compositor)**: when the player is standing in a room tagged with a **room property** (`Burning` / `Flooded` / `Infected` / `SporeMist` / `NanoHaze` / `Unstable` / `Haunted` / `RoyalMiasma`), the compositor applies a **luma-preserving color tint override** (blend toward a hazard RGB at uniform strength over the full 3D viewport—**no vignette**) so the hazard read clearly **overrides** the floor theme cast while keeping local brightness structure. Hazard telegraph **intensity pulses slowly** (~**3.8 s** sine) by scaling the **graded tint only** (fixed hazard RGB ratios), not the blend weight—so the cast stays **one hue** and does not swing toward unrelated scene colors. **`Burning`** reads as **hot red**; **`ruins_umber`** floor theme lighting uses **brown umber** intent (not fire red) so the two do not collide. Room lookup matches **`roomForCell`** (containment only; no nearest-room fallback). Tuning/preview can be forced via the debug telegraph override.
- **Room-hazard floor decals (3D)**: `WorldRenderer` draws **non-interactive** sprite billboards on **`floor`** tiles inside procgen rooms whose **`tags.roomProperties`** matches the hazard. Each property has a **dedicated** decal path in **`ROOM_HAZARD_SPRITE_SRC`** (`hazardDefs.ts`): **`Burning`** / **`Flooded`** / **`Infected`** use **`hazard_fire.png`**, **`hazard_water.png`**, **`hazard_poison.png`**; **`SporeMist`**, **`NanoHaze`**, **`Unstable`**, **`Haunted`**, **`RoyalMiasma`** use **`hazard_spore_mist.png`**, **`hazard_nano_haze.png`**, **`hazard_unstable.png`**, **`hazard_haunted.png`**, **`hazard_royal_miasma.png`** (currently **binary copies** of the baseline three hazard PNGs until bespoke art ships). Placement is **sparse and deterministic** (~**40%** of eligible cells, keyed by **`floor.seed`** + room id + cell), and cells occupied by a **POI**, **NPC**, or **floor item** are skipped. Decals sit low on the floor (ground-stain height), use the same **theme tint** as other billboards, and rebuild with **`floorGeomRevision`**. **Gameplay** runs `applyRoomHazardOnEnter` **once per visit** to a given procgen room (tracked by `floor.roomHazardAppliedForRoomId`): on the **first step** into a tagged room **or** when the party is **placed** in that room (initial floor, **new run**, **regen**, **descend**, **Exit** POI). It applies statuses, chip damage, log lines, SFX, and shake per hazard kind—**not** on every tile walked inside the same room.
- **Frame presentation pipeline**:
  - The 3D world is rendered offscreen into a **render target** sized to match the on-screen **game viewport rect** (the HUD “game” panel), not necessarily the full window.
  - The HUD exists as HTML/CSS twice:
    - an **interactive HUD** (`opacity: 0` over the final canvas but above the WebGL presenter, handles pointer input)
    - a **capture HUD** (offscreen, non-interactive) that is rasterized into a canvas texture. The capture HUD does not mount **`GameViewport`** (the 3D view is not part of **`html2canvas`**); the **game** grid cell uses an **invisible same-sized shell** with the same **`gameViewportRef`** attachment pattern for compositor/game-rect parity—**except** when **`ui.screen === 'hub'`**, where **`HubViewport`** renders the **2D hub image + hotspot outlines** into both **interactive** and **capture** trees so **`html2canvas`** picks up **opaque** pixels in that cell and the compositor **masks** the 3D scene there (same mechanism as other HUD-over-scene blending). The **dithered NPC dialog** mounts inside that **game** cell (**`npcCaptureLayer`**: **stretch**ed overlay; inner **`captureRoot`** fills the cell and pins **top panel + bottom speech strip**) rather than as a sibling of the whole **`HudLayout`**. The **death** modal (**real** or **debug preview**) uses the same **game** cell overlay path (**priority over NPC** in capture when both would apply): dim + panel are **centered** in the game widget. **Title** and **paperdoll** mount as **full-bleed capture** overlays on the entire HUD grid (**`captureFullHudOverlay`** → **`fullHudCaptureLayer`**). **Interactive** **title**, **death**, **paperdoll**, and **NPC dialog** hit targets mount **inside** the **`interactiveHud`** subtree in **`HudLayout`** (**`fullHudInteractiveLayer`** for title/paperdoll; **`gameCellModalHitLayer`** for death; **`npcDialogInteractiveLayer`** for NPC dialog—same padded box as capture **`npcCaptureLayer`**).
  - **Modal interaction**: an **offscreen capture** variant feeds the **dithered** HUD texture. **Interactive** **title**, **death**, **paperdoll**, and **NPC dialog** mount **inside** **`HudLayout`** with **stage-local** **`position: absolute`** layers (same **`transform: scale`** context as inventory and **`TradeModal`**), matching **trade** (**ADR-0263**, **ADR-0395**). **`GamePopup`’s** **`modalPortalHitRoot`** is unused by those modals. **`TradeModal` (interactive)** mounts in **`HudLayout`**’s **game** cell; **capture** trade still comes from **`captureFullHudOverlay`**. **`DitheredFrameRoot`’s `stageModalLayer`** mounts **Settings** (and similar) above the invisible hit HUD; **`pointer-events: none`** by default so it does not swallow **`HudLayout`** moves. **`CursorProvider`** (after each primary-button **`pointerup`**) runs a **microtask** that **`elementFromPoint`s** for **`[data-modal-chrome-hit]`** **`button`s**: if the **release coordinates** sit on that chrome but the **native `pointerup` target** was **not** inside that button (e.g. **`setPointerCapture`** on a drag source), it **synthesizes `click()`** so **Trade** / **Close** and other marked modal buttons stay reliable (**ADR-0280**).
  - Because the capture HUD is rasterized by **`html2canvas`**, prefer conservative CSS for critical visuals (avoid relying on `object-fit` for aspect-correct sprite presentation; use intrinsic sizing patterns instead).
  - **HUD shell art**: a single transparent **`ui_hud_background.png`** (`Content/ui/hud/`, mirrored to `web/public/content/ui/hud/` for some workflows) is drawn full-bleed behind the HUD grid in `HudLayout` (CSS `::before`, `background-size: contain`, centered). **Per-character equipment** uses a separate strip asset **`ui_hud_charequip_background.png`** in the portrait rails (also under `Content/ui/hud/`); it is part of the same DOM HUD capture and is dithered with the rest of the UI. Existing widgets stay in the current CSS grid; opaque “glass” panel styling is removed so content sits on the artwork (fine-tune slot positions vs the art next).
  - Presenter sizing is derived from the **viewport CSS size** (prefer `visualViewport.width/height`, else `documentElement.clientWidth/clientHeight`) rather than measuring the presenter canvas element, to avoid resize feedback loops caused by renderers writing inline canvas CSS sizes.
  - A presenter compositor shader places the scene render target into the frame **only inside** the game viewport rect and overlays the captured HUD everywhere else (and over the scene where UI alpha exists).
  - The final composite then runs through the ordered-dither post-process so the **same dithering/pixelation** applies to both 3D and HUD.

### 13.1 Asset serving & caching (web)
- Runtime-served assets (portraits, NPC sprites, UI art, cursor sprites) live under repo-level `Content/` and are referenced via **stable** `/content/...` URLs.
- The game relies on **browser HTTP caching** for these static URLs; avoid cache-busting query params for runtime art.
- Imperative image loads (e.g., measuring portrait aspect ratio or preloading sprite layers) go through a small shared **in-app image cache** (`web/src/ui/assets/imageCache.ts`) to dedupe concurrent loads and reduce repeated decodes/revalidation requests.

### 13.2 Procgen content coverage (data discipline)
- **Seeded item defs** live in `web/src/game/content/items.ts` (`DEFAULT_ITEMS`). **Not every item is meant to appear from dungeon generation**: some are **craft-only**, **Well transforms**, or **quest-chain** items. **Headwear** (`WoolCap`, `HerbCirclet`, `SporeCap`) is both **craftable** and can appear from **floor spawns** and **POI container** loot when wired in `spawnTables` / `poiLootTables`.
- **Procgen** places floor items via `web/src/procgen/spawnTables.ts` (`pickFloorItemDefFromTable`) and `web/src/procgen/population.ts` (`spawnNpcsAndItems`). **Forage patches**: when the rolled def is **Mushrooms**, **Fungus**, **Foodroot**, or **Grubling**, procgen places **several** one-qty floor entries on **nearby** floor cells in the same room (cluster radius **Manhattan ≤2** from an anchor; **2–5** cells per pass), instead of a single **room-center** pickup. Up to **four** successful item passes per floor from this loop, capped at **16** total population floor items before **lock** keys merge in. **`ItemSpawnContext`** includes **`floorType`** (e.g. **Jungle** **Habitat** can roll **Grubling**). **SporeMist** rooms split floor loot among **Moss**, **Mushrooms**, **Fungus**, plus smaller slices for **Salt** and **Mold**, with a small extra chance for **Slime** or **Glass vial** (crafting mats). On **Infested** floors, **SporeMist** **NPC** rolls use one RNG draw to pick among **Bulba**, **SporeGrub**, **Grub**, **Wurglepup**, and **Bobr** (non-Infested **SporeMist** keeps the prior **Wurglepup**/**Bobr** split). **Infested** **Habitat** rooms can likewise roll slime/vial mats before the **Grubling** grub pass. **Locks** add keys via `web/src/procgen/locks.ts`. **Chest / barrel / crate** opening uses **`STORAGE_POI_LOOT_DEF_IDS_BY_FLOOR`** in `web/src/game/content/poiLootTables.ts` (per **`FloorType`**; rolled in `web/src/game/state/poi.ts` with **`:chest:`** vs **`:container:`** hash salt). **Neutral NPC quests** reference a small want/hated item set from `web/src/procgen/population.ts`.
- **Policy**: when adding a new `ItemDefId`, either wire it into one of those pipelines or add it to **`ITEM_DEF_IDS_INTENTIONALLY_NON_PROCGEN`** in `web/src/tools/procgenContentNonSpawnAllowlist.ts` so the audit stays honest.
- **Audit CLI** (from `web/`): `npm run audit:procgen-content` — compares seeded items to the union of procgen + POI loot + quest references; exits non-zero if a seeded item is neither used nor allowlisted, or if listed ids are missing from `DEFAULT_ITEMS`. **Status effects** are not placed by procgen; they are applied in gameplay (combat, feed, hazards). Items may reference statuses via `feed.statusChances` (e.g. **Blessed** / **Sick** on mushrooms).
